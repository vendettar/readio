## 1. 存储架构

应用采用 **混合持久化模型**：

- **IndexedDB**（通过 Dexie.js 封装）：本地元数据、运行时缓存与大型二进制文件（音频、字幕）。
- **localStorage**：仅保存非敏感轻量设置（`readio-user-settings`）。
- **IndexedDB `credentials` 表**：保存敏感 Provider API Key（如 `provider_openai_key`、`provider_groq_key`）。

- **数据库名**：`readio-lite`。
- **存储模型**：尽力而为（Best-effort）。应用不申请永久存储权限，数据在磁盘压力极端时可能被浏览器清理。
- **事务安全**：所有涉及跨表的操作（如音轨移动、级联删除）均在 `db.transaction('rw', ...)` 中执行。

## 2. 数据库 Schema

| 表名 | 主键 | 唯一索引 | 用途 |
|-------|-------------|---------|------|
| **sessions** | `id` (UUID) | — | 播放会话，存储进度、元数据快照及来源信息。 |
| **audioBlobs** | `id` (UUID) | — | 存储原始音频二进制数据。 |
| **subtitles** | `id` (UUID) | — | 存储原始字幕文本内容。 |
| **remote_transcripts**| `id` (URL派生)| `&url` | 远程播客字幕缓存，支持三层读取逻辑。 |
| **folders** | `id` (UUID) | — | 存放用户上传音轨的本地文件夹。 |
| **tracks** | `id` (UUID) | — | 用户手动上传的音轨（来源：Files 页面），链接至 `audioBlobs`。 |
| **tracks**| `id` (UUID) | `[sourceFeedUrl+sourceUrlNormalized]` | 播客单集下载内容（来源：发现/节目单页），链接至 `audioBlobs`。 |
| **local_subtitles** | `id` (UUID) | — | 本地库字幕，链接至 `subtitles`。 |
| **subscriptions** | `id` (UUID) | `&feedUrl` | 订阅的播客节目。 |
| **favorites** | `id` (UUID) | `&key` | 收藏的单集，Key 为 `feedUrl::audioUrl`。 |
| **settings** | `key` | — | 应用全局设置（如语言、地区、主题色）。 |
| **credentials** | `key` | — | Provider 凭证（与非敏感设置分离存储）。 |
| **runtime_cache** | `key` | `namespace`, `[namespace+at]` | Discovery/推荐/词典缓存，包含 TTL 元信息。 |

## 3. 持久化契约

### Repository 边界（Phase 1）
- `LibraryRepository`：订阅/收藏与 Explore 国家设置持久化。
- `PlaybackRepository`：播放会话、音频 Blob、字幕 Blob。
- `FilesRepository`：文件夹/音轨/本地字幕及 files 相关设置。
- 范围：仅提取持久化边界；Dexie Schema、表名与迁移策略保持不变。

### 写入合并 (Write Coalescing)
- `exploreStore`、`filesStore`、`historyStore` 对同一资源键的并发写入会合并为单个 in-flight 任务。
- 调用方取消是“调用方作用域”：`AbortSignal` 仅短路当前调用，不会中断共享 DB 写入。
- 本地状态更新与提示会在存在活跃（未取消）调用方时才执行，避免全部取消后仍发生卸载后更新。

### 唯一性与去重
- **实体身份**：所有业务实体必须使用 UUID 作为主键。
- **逻辑去重**：`feedUrl` 和 `key` 仅用于索引查询和去重校验，禁止用作 UI 层的唯一 Key 或关联外键。

### 会话硬化 (Session Hardening)
- **Upsert 逻辑**：更新会话时自动合并字段，除非显式提供，否则保留现有的 `progress` 和 `lastPlayedAt`。
- **时间戳更新**：`lastPlayedAt` 仅在活跃播放期间更新，防止仅浏览或恢复会话时污染历史排序。

### 历史保留 (Retention)
- **规模限制**：`playback_sessions` 自动保留最新的 1000 条记录或最近 6 个月的历史。
- **执行时机**：应用启动及手动维护时分批执行。
- **运行时缓存**：Discovery 与 feed 缓存落在 IndexedDB 的 `runtime_cache`，读取顺序保持 memory-first，并执行有界保留策略。
- **Wipe All 语义**：执行“清空全部”时，会清空 IndexedDB 全部数据（含 `credentials`）、运行时内存缓存，以及 localStorage 的设置键与遗留凭证键（若存在）。

### 凭证存储契约
- **规范键名**：`provider_<vendor>_key`（snake_case），由 `CREDENTIAL_KEY_PATTERN` 校验。
- **读取优先级（启动/设置页水合）**：
  1. IndexedDB `credentials` 表
  2. 一次性遗留 localStorage 迁移源（`readio-user-credentials`）
  3. 运行时环境默认值（`READIO_OPENAI_API_KEY`、`READIO_GROQ_API_KEY`）
  4. 空字符串
- **写入边界**：所有凭证 CRUD 必须通过 `lib/db/credentialsRepository.ts`。
- **值规范化**：凭证值在写入时自动 trim，防止 API Key 中意外包含前导/尾随空白。
- **原子写入**：多键凭证写入使用单事务（`setCredentials`，全有或全无）。
- **清空竞态防护**：Wipe 流程会提升凭证写入 epoch，阻止清空前发起但延迟完成的异步写入回填敏感值。`clearAllCredentials()` 先提升内存 epoch，随后 `clearAllData()` 再次清空所有表（含 credentials）以保证表级完整性。
- **Epoch 作用域**：写入 epoch 为 per-tab 内存态。跨 Tab 一致性依赖 Wipe 后的 `reload()`；`BroadcastChannel` 方案作为 YAGNI 延后，适用于单用户 PWA 场景。

## 4. 存储保障 (Guardrails)

### 配额监控
- **实时估算**：通过 `navigator.storage.estimate()` 监控 IndexedDB 占用。
- **拦截阈值**：磁盘使用率 > 85% 或单次写入后 > 95% 时，系统自动拦截文件导入。
- **容量上限**：音频二进制数据强制执行全局 10GB 上限。

### 数据自愈 (Integrity)
- **孤儿检查**：`runIntegrityCheck` 负责标记缺失 Blob 的音轨为 `isCorrupted`，并将失效文件夹下的音轨移回根目录。

## 5. 迁移与导出

- **无迁移层**：代码仅维护当前生效的 Schema 版本，不支持旧数据的增量回填或结构转换。
- **遗留凭证迁移**：`migrateCredentialsFromLocalStorage()` 在应用启动时执行。写入成功后删除遗留键；畸形 JSON 直接丢弃；若 IndexedDB 写入失败则保留遗留键以便下次重试。
- **Vault 导出**：导出格式为单一 JSON，仅包含元数据表，不含二进制 Blob，并排除凭证相关键。
- **OPML 支持**：支持标准 OPML 导入导出，遵循合并（Merge）语义。
