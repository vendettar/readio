## 1. 基础环境要求 (Engines)

为确保工具链的一致性，系统通过 `package.json` 强制执行以下版本要求：
- **Node.js**: `>=20.0.0`
- **pnpm**: `>=10.28.0`

## 2. 持续集成与质量保障 (CI/CD)

### CI 流水线
GitHub Actions 在每次推送或 Pull Request 时自动执行：
- **代码规范**：Biome (Lint & Format) 校验。
- **类型检查**：TypeScript 类型完整性验证。
- **自动化测试**：Vitest 单元测试与 Playwright E2E 烟雾测试。
- **质量预算**：Lighthouse CI 强制执行性能 (≥90) 与无障碍 (≥95) 指标。

### 部署支持
- **Vercel (默认)**：基于 `vercel.json` 自动处理 SPA 路由重写与 CSP 响应头。
- **Docker**：提供基于 `nginx:alpine` 的镜像定义，适用于私有化部署。

## 3. PWA 与缓存规范

应用作为高质量 PWA 运行，其配置遵循以下标准：
- **Manifest**：由 `vite-plugin-pwa` 动态生成，配置集中于 `vite.config.ts`。
- **更新策略**：采用 `prompt` 模式，检测到新版本时通过 Toast 引导用户手动刷新。
- **音频 Range 缓存**：支持 HTTP Range 请求 (206)，允许在离线状态下跳转进度。
- **缓存分区**：
  - `readio-audio-v2`：远程音频（`.mp3/.m4a/.aac/.ogg/.wav`）采用 `StaleWhileRevalidate`，开启 `rangeRequests: true`，缓存状态 `[200, 206]`，上限 `maxEntries: 50`、`maxAge: 7 天`。
  - `readio-api-v1`：Discovery/Feed 请求采用 `NetworkFirst`，上限 `maxEntries: 80`、`maxAge: 24 小时`。
  - 字体缓存保持独立（`google-fonts-cache`、`gstatic-fonts-cache`）。
- **缓存策略**：本地 Blob（`blob:` URL）显式排除在 PWA runtime 缓存之外。

### 离线验证清单（Instruction 115）
- 在 DevTools Application 面板确认存在 `readio-audio-v2` 与 `readio-api-v1` 缓存命名空间。
- 在线播放远程单集并至少进行一次拖动进度，然后切换为离线。
- 离线刷新应用后，验证该缓存单集仍可启动播放。
- 离线状态下前后拖动进度，验证可以成功跳转且无需整段重播。
- 连续缓存足量不同单集触发上限后，确认 `readio-audio-v2` 中旧条目被淘汰。

## 4. 运行时配置 (Runtime Config)

应用采用动态配置机制 (`public/env.js`)，允许在不重新编译的情况下调整运行参数。

### 核心配置项
- **CORS 级联代理**：支持 `Direct -> Custom Proxy -> Default Proxy` 的三级故障转移机制。
- **超时控制**：直连阶段 (5s) 与代理阶段 (10s) 具备独立超时。
- **存储限制**：`READIO_MAX_AUDIO_CACHE_GB` (10GB) 与 `READIO_MAX_AUDIO_SIZE_MB` (300MB)。
- **语言与地区**：定义默认界面语言 (`READIO_DEFAULT_LANGUAGE`) 及默认内容地区 (`READIO_DEFAULT_PODCAST_CONTENT_COUNTRY`)。
- **英语词典端点**：`READIO_EN_DICTIONARY_API_URL` 是当前英语专用词典后端在浏览器运行时暴露的 env key。
- **凭据回退键**：`READIO_ASR_API_KEY` 与 `READIO_OPENAI_API_KEY` 仅是浏览器运行时回退字段，绝不能放置真实上游 Provider Secret。

### Runtime Config 模块契约（Phase 1）
- 对外 API 保持不变：`runtimeConfig.ts` 仍提供 `getAppConfig()` 与 `isRuntimeConfigReady()`。
- 内部拆分：
  - `runtimeConfig.defaults.ts` 存放默认值常量。
  - `runtimeConfig.schema.ts` 存放 Zod Schema 与 env key 映射。
- 词典运行时契约：
  - 内部配置字段：`EN_DICTIONARY_API_URL`
  - 浏览器 env key：`READIO_EN_DICTIONARY_API_URL`
  - 该契约当前描述的是英语专用后端，而不是通用多语言词典服务。
- Secret 护栏（fail-closed）：
  - 浏览器运行时配置会拒绝 `READIO_ASR_API_KEY` / `READIO_OPENAI_API_KEY` 的已知上游密钥前缀。
  - 已拒绝前缀包括：`sk-`、`sk-proj-`、`gsk_`、`gsk-`。
  - 被拒绝值在运行时会被忽略（按空值处理）。

### 代理协议契约
自定义代理必须满足以下技术规范：
- **方法**：始终使用 `POST`。
- **认证**：通过 `x-proxy-token` 请求头传递凭证。
- **数据格式**：必须返回 Raw (原始内容)，禁止包裹在 JSON 对象中。
- **内部网络模块拆分**：`fetchUtils.ts` 继续作为公共入口，proxy URL 处理、熔断状态、timeout/abort 线路已拆分到 `lib/networking/` 子模块。

## 5. 内容安全策略 (CSP)

CSP 策略通过服务器响应头强制执行，核心准则包括：
- **指令**：`default-src 'self'`, `style-src 'self' 'unsafe-inline'` (支持动态主题)。
- **跨域许可**：允许 `connect-src` 与 `img-src` 访问 `https:` 协议以支持远程播客资源与自定义代理。

## 6. 部署安全模式 (Deployment Security Modes)

应用支持两种处理敏感凭据 (API Keys) 的安全模式，具体取决于部署环境：

### 模式 A: 公共 Lite 部署 (BYOK)
- **场景**：公共互联网，多租户访问 (如 Vercel, Netlify, GitHub Pages)。
- **规则**：**严格执行 BYOK (自备密钥)**。
- **机制**：使用 Instruction 122/123 流程，用户需在设置页手动输入密钥。
- **限制**：严禁在此模式下通过构建时或运行时环境变量注入 API Key。注入的 Key 对所有访问者可见。

### 模式 B: 私有 Docker 部署 (Private Instance)
- **场景**：私有网络 (LAN, VPN) 或受密码保护的单租户实例。
- **规则**：仅允许注入非 secret / 公共样式的浏览器运行时回退值。
- **机制**：可在 `env.js` 里设置 `READIO_ASR_API_KEY` 和 `READIO_OPENAI_API_KEY` 作为回退；命中已知上游密钥前缀的值会被 fail-closed 拒绝。
- **风险**：所有浏览器可见值都可被 DevTools 读取；真实上游 Provider Secret 永远不属于浏览器运行时配置。

### 结论 (Handoff)
- **公开多租户**：必须使用 BYOK。
- **私有单租户**：仅可使用非 secret / 公共样式的浏览器运行时回退值。
- **所有模式**：真实上游 Provider Secret 不得通过浏览器运行时配置暴露。
- **混合模式**：不支持。
