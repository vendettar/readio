title: 文件管理 (Files)
---

## 1. 范围与归属

- 路由：`/files` 以及 `/files/folder/$folderId`。
- 主界面：`src/routeComponents/files/FilesIndexPage.tsx`。
- 架构模式：编排页 (Orchestration Page) + 模块化分区 (Modular Sections)。
- 数据持久化：IndexedDB 表 `local_tracks`（用户上传）、`podcast_downloads`（播客缓存）、`local_folders`, `local_subtitles`, `audioBlobs`。
- **逻辑隔离**：用户手动上传的文件（Files 页面）与下载的播客单集（Downloads 页面）分别存储在不同的数据库表中，从物理层面确保数据互不干扰，彻底解决 Files 页面出现冗余下载项的问题。

## 2. 页面架构

`FilesIndexPage` 承担拖拽 (DnD) 上下文管理与业务逻辑编排职责，具体渲染由以下独立分区组件执行：

- **PageHeader**：标准化的页面顶部组建，提供面包屑导航、页面标题及创建操作触发（添加音频、新建文件夹）。
- **FoldersGridSection**：文件夹网格渲染，支持视图密度切换。
- **TracksListSection**：音轨列表渲染及空状态处理。
- **NewFolderCard**：内联式文件夹创建组件。
- **FileDragPreview**：统一的拖拽跟随预览。

核心逻辑由专门的业务 Hook 驱动：`useFilesData` (数据流)、`useFileDragDrop` (移动逻辑)、`useFileProcessing` (导入流)。

## 3. 文件处理规范 (Ingestion API)

### 元数据解析
- 采用 Web Worker 异步运行 `music-metadata` 解析二进制音频，确保 UI 线程流畅。
- 优先提取音频内置元数据（标题、时长），解析失败时回退至文件名。

### 自动处理机制
- **自动去重**：同一文件夹下出现同名文件时，系统自动追加递增后缀 `(N)`。
- **字幕匹配**：批量导入时按名称前缀自动匹配字幕与音轨，确保一对一绑定。

### 存储保障 (Guardrails)
- **容量拦截**：当磁盘使用率达到 85% 或预期写入后超过 95% 时，系统自动拦截导入操作。
- **缓存上限**：音频二进制数据强制执行 `MAX_AUDIO_CACHE_GB` (10GB) 的存储上限。

## 4. 交互规范

### 导入入口与校验
- **全局拖拽 (react-dropzone)**：支持在页面任意位置释放文件。实时校验：音频限制 500MB（mp3, m4a, wav, ogg, flac, aac, webm）；字幕限制 5MB (.srt, .vtt)。
- **按钮选择限制 (Native Picker)**：
    - **“添加音频”**：严格限制系统文件选择器仅显示音频格式（`audio/*` 及指定后缀），移除字幕可选性。
    - **“添加字幕”**：严格限制仅可选 `.srt` 与 `.vtt`，防止跨职责误导。

### 视图密度 (View Density)
- 支持 `comfortable` (舒适) 与 `compact` (紧凑) 模式。
- 模式状态持久化于数据库 settings 表，全局生效。

### 菜单与安全性
- **二步确认**：删除音轨或文件夹均在组件内部实现滑动式二步确认，避免破坏性操作误触。
- **重命名校验**：执行 `trim()` 并忽略大小写。名称冲突时通过内联错误气泡提示并保持编辑状态。

## 5. 完整性与维护

- **自动维护**：应用启动时自动执行完整性检查，修复损坏的 Blob 引用及孤儿轨道。
- **手动维护**：用户可通过设置页面手动触发全量扫描并获取报告。

## 6. 下载内容字幕版本管理 (Instruction 125b)

已下载的播客单集可能会积累多个字幕版本（例如：使用不同模型多次执行 ASR，或手动上传多个版本）。系统通过“渐进式披露”原则提供管理功能：列表卡片仅显示摘要，侧边抽屉提供详细管理。

### 数据契约 (Data Contract)
- 每个字幕版本存储在 `local_subtitles` 表中，包含版本元数据：`sourceKind` (来源类型), `provider` (服务商), `model` (模型), `language` (语言), `createdAt` (创建时间), `status` (状态)。
- 每个下载项通过 `PodcastDownload.activeSubtitleId` 跟踪唯一的“当前激活字幕”。
- 并发控制：`PodcastDownload.manualPinnedAt` 用于记录用户手动选择的时间点，防止后台异步 ASR 任务错误覆盖用户的手动选择。

### 仓库层 (Repository Layer)
- `DownloadsRepository` (`src/lib/repositories/DownloadsRepository.ts`) 是所有字幕版本操作的唯一事实来源。
- 关键操作：`getSubtitleVersionSummary` (获取摘要), `setActiveSubtitle` (设为当前), `deleteSubtitleVersion` (带自动回退的删除), `exportSubtitleVersion` (导出单版本), `exportAllSubtitleVersions` (导出全部版本)。
- 所有变更操作均在 Dexie 事务中原子化执行。删除当前激活版本时，系统会自动回退至最近的一个可用 (Ready) 版本。

### UI 模式
- **列表卡片**：复用了 Files 页面中 `TrackCard` 组件的视觉设计风格（提供更精细的尺寸、时长及布局展示），但在功能上专门针对播客下载数据做了适配与裁剪。避免了在卡片下方直接展开长列表，而是采用轻量的 `SubtitleSummaryBadge` 徽章（显示可用版本数量），用户点击后会滑出详情抽屉。这样既保持了全局 UI 的一致性与高级感，也避免了直接重用 Files 文件管理器带来的重量感。
- **详情抽屉** (`SubtitleVersionSheet`)：列出所有版本及其详细元数据，支持切换激活状态、导出单个/全部版本（多版本导出为 ZIP），以及带二次确认的删除。
- **自动降级**：单版本场景会自动简化 UI（例如：不显示“设为当前”按钮，导出操作直接提供 SRT 文件而非 ZIP 压缩包）。

## 7. Overflow「无字幕播放」动作（Instruction 129）

- **Files 音轨菜单**：
  - 最新产品决策：Files 页面**不提供** `无字幕播放` 动作。
  - `TrackOverflowMenu` 与 `TrackCard` 在 Files 相关界面不得暴露该入口。
  - Files 页 3 点菜单提供 `生成字幕` / `重新生成字幕`：
    - 当前音轨没有任何字幕版本时显示 `生成字幕`；
    - 已有一个或多个字幕版本时显示 `重新生成字幕`。
  - 重新转译的持久化遵循 provider+model 的 upsert 规则：
    - provider+model 完全相同则覆盖既有 ASR 版本；
    - provider 或 model 任一不同则追加为新版本。
  - 从卡片菜单触发的转译/重新转译使用稳定分片策略（10MB / 600 秒），显式关闭 progressive 5/10/20 递增切片，因为该入口不承担“阅读首屏尽快出字”的启动目标。
- **Downloads 卡片**：
  - `DownloadTrackCard` 新增 3 点 overflow 菜单。
  - 菜单中承载 `无字幕播放`（在线且远程地址可用时）与删除下载动作。
  - 该动作的播放路径遵循“本地优先”：若单集已下载，则复用本地 blob 音频，仅抑制字幕绑定。
  - 该“本地优先”规则与进度无关：即使当前播放进度非 0，再次点击 `无字幕播放` 也应按本地会话继续，不应再去抓取/解析流式播放 URL。
  - **国家快照契约（必填）**：每次下载写入必须持久化规范化后的 `countryAtSave`。若国家为空或非法，写入必须失败（`invalid_country`），且不得创建下载记录。
  - **单集路由契约（禁止全局回退）**：Downloads 的标题跳转只能使用记录快照（`countryAtSave` + provider IDs）构造路由，禁止在点击时回退到 `globalCountry`。
