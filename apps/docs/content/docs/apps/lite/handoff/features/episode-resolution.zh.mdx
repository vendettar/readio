---
title: 单集解析契约 (Episode Resolution)
---

## 1. 设计目标

为了解决播客生态中 iTunes ID 与 RSS GUID 不一致的问题，并确保详情页 URL 在数据源变化时依然可访问，系统采用了 **ShortId 8位哈希匹配算法**。

## 2. 匹配算法流程

当用户访问单集详情 URL 时，系统执行以下确定性匹配逻辑：

1. **Token 提取**：从 URL Slug（如 `title-slug-a1b2c3d4`）中提取末尾 8 位的 `ShortId`。
2. **第一优先级：RSS GUID 匹配**：
   - 遍历 RSS Feed 中的所有单集。
   - 对每个单集的原始 GUID 执行 8 位哈希计算。
   - 若匹配成功，则建立关联。
3. **第二优先级：标题哈希匹配**：
   - 若 GUID 未命中（部分播客可能无 GUID），则对标题执行哈希计算并匹配。
4. **回退策略：Provider Lookup**：
   - 若 RSS 全量扫描失败，调用 Provider Lookup API (limit=300) 获取元数据。
   - 通过 Provider 返回的音频 URL 与标题回查 RSS。

## 3. 约束与限制

### 唯一性保障
若 8 位哈希发生冲突（多项匹配），系统将叠加以下判断：
- 比较标题的 Slug 全称。
- 优先选择发布日期 (`pubDate`) 最近的一项。

### URL 永久化
- **不可变性**：一旦单集被生成带哈希的 URL，其 `ShortId` 必须保持不变。
- **纯净性**：详情页跳转严禁携带 `source` 或 `feedUrl` 等冗余查询参数，所有信息必须能从路由参数还原。

## 4. 异常处理

- **地区失效 (Region Unavailable)**：若 Provider 返回 200 但在当前 `$country` 下无法解析内容，系统必须显示地区不可用提示，而非通用的 404。
- **元数据降级**：若最终无法关联 RSS 实体，系统应以 Provider 的元数据进行只读渲染，确保用户仍能播放。
