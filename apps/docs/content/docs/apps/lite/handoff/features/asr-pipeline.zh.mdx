Readio 自动语音识别 (ASR) 系统负责将本地和下载的远程音频文件转换为带有时间戳的字幕（利用 Groq Whisper API）。为了突破 API 的负载限制和速率干预，同时保证极佳的用户体验，系统采用了一套高级的队列调度和切片增量渲染策略。

## 架构与生命周期

当用户启动播放且未命中本地字幕缓存（且后台配置开启）时，系统将触发 ASR 处理管道。

**核心工作流 (顺序处理):**
1. **触发:** 由 `autoIngestEpisodeTranscript` 或 `autoIngestLocalTranscript` 发起任务。
2. **音频分块:** 原始的音频 `Blob` 会通过 `splitMp3Blob` 被切割成较小的流片段（Chunk），以确保体积低于 Groq 规定的 25MB 单文件上限。
3. **队列控制:** 所有分块会被按顺序投递到一个全局的并发受控队列中（`ASRQueue`，并发数定为 1）。这能有效防止瞬间高频调用导致的 `HTTP 429 Too Many Requests` API 封禁。
4. **即时预览 (JIT Preview):** 
   - 我们**不会**干等一个 50 分钟的音频全部转译完毕。
   - 只要任何一个 chunk 成功返回了 API 的结果数据，这些字幕就会被立即写入内存中的 Zustand 状态 (`playerStore.partialAsrCues`)。
   - 这样一来，在后台继续转译后续部分时，用户在播放开头几分钟就能立刻看到字幕，达到了类似“边下边播”的平滑感知。
5. **原子化落盘:** 
   - 半成品的结果严格被控制在内存级别。
   - 只有在**所有分块**全部成功结束转译后，系统才会一次性将完整数据聚合写入 IndexedDB 数据库 (`remote_transcripts` 或 `local_transcripts`)。
   - 如果用户中途关闭了应用，数据库绝不保存半途而废的残破数据，保障了状态的强一致性；下次播放时将重新触发计算。

## 分块与时间轴防漂移机制

ASR 音频切块面临的最大技术挑战是防止**文本与时间轴漂移 (Timeline Drift)**。如果把一个音频随意切成 5 段，API 返回的时间戳是基于“该段开头算起即 0 秒”的相对时间。如果强行拼接，极为容易发生错位。

Readio 通过**“物理偏移与逻辑偏移精准解耦”** 的设计模式解决了这个问题。

### 1. 物理切分：帧边界对齐
我们并没有按照字节数粗暴地一刀切（比如硬性每 10MB 砍一刀）。这样随机的横切会斩断 MP3 的压缩音频帧，导致后续解码器在边界处产生爆音，严重影响 Whisper 模型的上下文识别精度。

相反，`splitMp3Blob` 算法会去主动扫描 MP3 的**同步头信息 (Sync Words**, 例如 `0xFF 0xFB`)，确保每一次切割的起始和落幅，都在一个**完整、合法的音频帧边界**上。

正因每一帧都代表着固定不可更改的音频样本数量 (Samples，例如 MPEG-1 Layer III 的 1152 样本/帧)，我们因此可以通过采样率**严丝合缝地在数学上**计算出每一个 chunk 的精确绝对时长。

### 2. 时间平移：逻辑偏移修正
当 Groq API 处理切片 2 时，返回的 `ASRCue` 数组的第一句话时间如果是相对时间 `00:00.500`。

为了映射回全局时间轴中，`mergeAsrCues` 管道应用了一个极其精确的时间平移公式：
```text
全局绝对时间 = API返回的相对时间 + 该Chunk物理前置偏移的绝对时间
```

**根除累计误差:** 
任意一个 chunk 的 `startTimeOffset`，在数学上等于排在它前面的所有 chunk （基于帧对齐计算出的）物理时长的总和。
由于我们推算时间偏移量的基础是**离散且不可变的样本计数**，而非粗略的宽泛计算或者容易产生抖动的系统计时器（如 JavaScript Timer），最终彻底消除了微观精度溢出的风险。长达三小时的播客被切分成哪怕 20 块，最后拼合出的文本长卷依然能保持与音轨 100% 同步。

## 状态同步与读取
处于调用层（UI侧）消费记录状态时（诸如 `TranscriptView` 或 `ReadingContent` 组件），开发者应当订阅如下 `playerStore` 状态机：

- `transcriptIngestionStatus`: 应用层可依赖其状态展示加载环 (`LOADING`)、进度条 (`PROCESSING`)，以及异兆反馈 (`FAILED`)。
- `partialAsrCues`: 用户正在播放消费的流式字幕碎片。在转译大循环结束的一刹那，它会无缝交接到正式持久化的离线 DB 数组去。

**复播并发安全提示:** 
如果是网络远程流媒体正在被 `enforceAsrDownloadBlocking` 拦截以进行全量下载。此时若被用户的其他点击操作切歌中断，原先旧的时代表求派发会被数学级别的 `Epoch Guard` (`bumpPlaybackEpoch()`) 给直接拦截作废。这重机制保护了后来者居上法则，彻底排除了后台幽灵 ASR 强行把上一个文件的字幕塞入后来者的播放域内。
