# 单集解析策略

Readio 实现了一套复杂的多级解析策略，确保即使在主要数据源失效时，单集内容也始终可访问。本文档详细说明了该系统的工作原理，特别是从热门单集（Top Episodes）导航时的行为。

## 核心问题

当用户导航到单集详情页（例如从热门单集列表点击）时，系统面临两个挑战：

1. **ID 不匹配**：Apple 的 iTunes ID（`1000748472995`）与 RSS Feed 的 GUID（`gid://art19...`）不同
2. **数据质量差异**：Apple API 提供基础元数据，但 RSS Feed 包含丰富内容（章节、HTML 备注）

## 国家来源规则（当前）

- 内容路由统一为 `/$country/podcast/*`。
- 这些内容路由的国家权威仅来自路由参数 `$country`，并在 `routes/$country.tsx` 中完成校验与规范化。
- `useResolvedLibraryCountry` 已从播客内容路由去耦，不再参与内容页国家决策。
- `countryAtSave` 仍会写入 Library 记录，并仅用于 History/Favorites/Subscriptions 的纯路径链接构造。
- 单集解析正确性完全基于路径参数（`$country/$id/$episodeId`）；查询提示已不再属于应用内内容路由契约。
- 内容路由上的旧查询提示采用“容忍并剥离”策略，不做严格拒绝。

## 解析流程

### 阶段一：播客元数据查询

**请求**: `GET /lookup?id={podcastId}&country={country}&entity=podcast`

**目的**: 获取播客的 RSS Feed URL。

**缓存键**: `['podcast', 'lookup', podcastId, country]`

**触发条件**: 始终触发，除非缓存命中（24 小时内）。

### 阶段二：RSS Feed 抓取

**请求**: 通过代理请求播客的 RSS Feed URL（经由自定义代理）

**目的**: 获取全面的单集数据，包括：
- 富文本 HTML 描述（`content:encoded`）
- 章节标记（`podcast:chapters`）
- 文字稿 URL
- 精确的音频 URL

**缓存键**: `['podcast', 'feed', normalizedFeedUrl]`

**触发条件**: 当阶段一返回 `feedUrl` 时。

**回退机制**: 如果 RSS 抓取失败，系统会在同一查询链路回退到 provider episodes；若 provider 回退也失败，则该查询抛错并进入“请求失败”状态（不会进入 region-unavailable）。

### 阶段三：服务商单集查询（回退）

**请求**: `GET /lookup?id={podcastId}&country={country}&entity=podcastEpisode&limit=300`

**目的**: 
- **恢复**：如果 RSS 失败，从 Apple 获取基础单集数据
- **ID 解析**：通过标题/URL 匹配将 iTunes ID 映射到 RSS GUID

**缓存键**: `['podcast', 'provider-episodes', podcastId, country]`

**触发条件**: 当 `!!feed && !episode`（Feed 已加载但通过 ID 未找到单集）

**处理流程**:
1. 从 Apple 获取最近的 300 集
2. 查找匹配 iTunes ID 的单集
3. 如果 RSS 成功：通过标题或音频 URL 匹配找到对应的 RSS 单集
4. 如果 RSS 失败：直接使用 Apple 元数据（降级体验）

## 导航场景

### 场景 A：从热门单集直接点击（冷缓存）

**用户操作**: 从热门单集列表点击单集 → 单集详情页

**网络请求**:
1. **播客查询**（必需）：获取 RSS Feed URL
2. **RSS Feed**（必需）：获取丰富内容（返回 GUID）
3. **服务商单集查询**（必需）：解析 iTunes ID (`12345`) 到 RSS GUID → **共 3 个请求**
   - **原因**：热门单集 API 提供的 iTunes ID 几乎从未直接匹配 RSS GUID。

**请求后的缓存状态**:
- `['podcast', 'lookup', id, country]` → 播客元数据（24 小时）
- `['podcast', 'feed', normalizedFeedUrl]` → RSS Feed 或回退列表（24 小时）
- `['podcast', 'provider-episodes', id, country]` → 单集列表（12 小时）*（如果阶段三被触发）*

### 场景 B：先访问节目页（热缓存）

**用户操作**: 访问播客节目页 → 返回热门单集 → 点击单集

**网络请求**:
1. **节目页**:
   - 播客查询（缓存未命中）→ **1 个请求**
   - RSS Feed（缓存未命中）→ **1 个请求**（或在失败时回退）
2. **单集页**:
   - 播客查询 → **缓存命中** (0 个请求)
   - RSS Feed → **缓存命中** (0 个请求)
   - 服务商单集查询 → **1 个请求** (缓存未命中)
     - **原因**: 热门单集使用 iTunes ID (`12345`), 而 RSS Feed 使用 GUID (`gid://...`)。两者不匹配。
     - 系统必须获取 `provider-episodes` 来将 iTunes ID 映射到 RSS 单集。

**结果**: 单集页通过 **1 个请求** (用于 ID 解析) 完成加载，而非 3 个。

> **备注**: 如果播客节目页之前加载 RSS 失败并回退到 Apple API，`provider-episodes` 就已经被缓存了。在这种特定情况下，单集页加载是 **0 个请求**。

**用户操作**: 在自定义代理宕机时点击单集

**网络请求**:
1. 播客查询 → 成功
2. RSS Feed → **失败**（403、超时、空响应）
   - Catch 块返回包含 `episodes: []` 的占位 Feed
3. 服务商单集查询 → **被触发**（因为 `!episode`）
   - 返回 Apple 的单集列表
   - 使用 Apple 元数据渲染单集

**用户体验**: 页面成功加载，但质量降级：
- ✅ 可播放音频
- ✅ 基础描述（纯文本或有限 HTML）
- ❌ 无章节
- ❌ 无富文本备注

## 缓存优化

### 缓存键结构一致性

Lookup/provider 查询仍按国家分区；Feed 缓存改为仅按规范化 feed URL 分区（不含国家）。

**当前模式**:
```typescript
// ✅ Lookup/provider 仍按 country 分区
['podcast', 'lookup', podcastId, country]
['podcast', 'provider-episodes', podcastId, country]

// ✅ Feed 仅按 normalizedFeedUrl
['podcast', 'feed', normalizedFeedUrl]
```

Feed 存储命名空间为 `feed`。

### 地区不可用行为

- 触发条件：
  1. 路由国家有效。
  2. `lookupPodcast` 成功。
  3. 目标 show/episode 在路由国家内容链路中无法解析。
- 行为约束：
  - 显示明确的“地区不可用”提示。
  - 只提供一个显式 CTA：在当前全局国家下打开同一目标。
  - 不自动切换国家，不进行静默回退。

### 缓存生命周期

| 缓存键 | 过期时间 | GC 时间 | 理由 |
|--------|----------|---------|------|
| `podcast/lookup` | 24 小时 | 24 小时 | 元数据变化慢，且按国家维度复用 |
| `podcast/feed` | 24 小时 | 24 小时 | 以规范化 URL 为键，在多入口间共享解析结果 |
| `podcast/provider-episodes` | 12 小时 | 24 小时 | 回退 ID 映射缓存，保持比 lookup/feed 更短的新鲜度 |

### 查询契约（Instruction 117）

- Query Key 策略统一收敛到 `src/lib/discovery/podcastQueryContract.ts`。
- 内容页与解析 Hook 统一使用同一套 key builder 与缓存生命周期：
  - `buildPodcastLookupQueryKey(podcastId, country)`
  - `buildPodcastFeedQueryKey(normalizedFeedUrl)`（消费已规范化输入）
  - `buildPodcastProviderEpisodesQueryKey(podcastId, country)`
- Feed URL 规范化放在 ingestion/写入边界（provider 映射与本地持久化写入），而不是 render/query 热路径。
- 页面与 Hook 的 queryFn 都会把 React Query `signal` 透传到 discovery 调用，以保证切换路由国家时的取消语义完整。
- 当 feed 回退成功拿到 provider episodes 时，解析器会同步写入 `provider-episodes` 缓存，避免同一解析周期内的重复 provider 拉取。
- 路由组件测试与解析 Hook 测试都覆盖了切国家后的 stale 响应防护。

### 错误分类契约（Instruction 117）

- 路由渲染分支顺序固定为：
  1. 请求/运行时错误（`lookup` / `feed` / `provider`） -> 错误态
  2. not-found 语义
  3. region-unavailable 业务态
- `regionUnavailable` 仅在上游 lookup/feed/provider 成功，但当前路由国家内容无法解析时成立。
- 任何原始网络失败都不能回退成 `regionUnavailable`。

### 5xx 上游重试机制

当**所有**请求尝试（直连、自定义代理、默认代理）都因 **5xx 错误**（服务器错误）失败，且**配置了自定义代理**时，系统会自动触发重试机制：

1. **等待 3 秒**：给上游服务（如 Apple API 瞬时故障）一个恢复的时间窗口
2. **仅重试代理**：跳过直连（已经失败），只重试 自定义代理 → 默认代理
3. **单次重试**：只尝试一次，避免过度延迟

**触发条件：**
- 所有尝试都返回了 5xx 状态码（500、502、503、504 等）
- 已配置自定义代理（`CORS_PROXY_URL` 已设置）
- 请求未被中止

**为什么只在配置自定义代理时才重试？**
- 代理服务由用户自行管理，如果代理本身失效，重试意义不大
- 自定义代理（如 Cloudflare Workers）有专属容量，值得重试

**代码参考**：`apps/lite/src/lib/fetchUtils.ts` - `fetchWithFallback()`

## 性能影响

### 无缓存优化
- **首次访问（冷）**：2-3 个请求，约 1-2 秒加载时间
- **后续访问（热，但未访问节目页）**：2-3 个请求，约 1-2 秒加载时间

### 有缓存优化（当前实现）
- **首次访问（冷）**：2-3 个请求，约 1-2 秒加载时间
- **访问节目页后**：1 个请求 (ID 解析)，约 300-500 毫秒加载时间

**改进**: 对于浏览同一播客多个单集的用户，请求减少最多 **90%**，加载时间减少最多 **80%**。
