## 概述

Readio 当前使用 `dictionaryapi.dev` 作为英语词典后端。

- 基础 endpoint: `https://api.dictionaryapi.dev/api/v2/entries/en/<word>`
- 当前范围：仅英语
- transport 模式按运行时配置决定：
  - Lite：`direct` 或自定义 proxy
  - Cloud：`direct` 或同源 `/api/proxy`

这份文档定义当前 API 合同，以及前端代码必须保持的错误语义。

## 请求合同

### 查询单词定义

```http
GET https://api.dictionaryapi.dev/api/v2/entries/en/<word>
```

示例：

```http
GET https://api.dictionaryapi.dev/api/v2/entries/en/hello
```

## 成功响应

成功时返回 `200 OK`，body 为 JSON 数组。

示例：

```json
[
  {
    "word": "hello",
    "phonetic": "həˈləʊ",
    "phonetics": [
      {
        "text": "həˈləʊ",
        "audio": "//ssl.gstatic.com/dictionary/static/sounds/20200429/hello--_gb_1.mp3"
      },
      {
        "text": "hɛˈləʊ"
      }
    ],
    "origin": "early 19th century: variant of earlier hollo ; related to holla.",
    "meanings": [
      {
        "partOfSpeech": "exclamation",
        "definitions": [
          {
            "definition": "used as a greeting or to begin a phone conversation.",
            "example": "hello there, Katie!",
            "synonyms": [],
            "antonyms": []
          }
        ]
      }
    ]
  }
]
```

当前前端会把这个响应规范化成：

- `word`
- `phonetic`
- `meanings[].partOfSpeech`
- `meanings[].definitions[]`

上游可能还会返回额外字段，但当前产品合同只依赖以上规范化字段。

## 未找到合同

当单词不存在时，`dictionaryapi.dev` 返回 `404 Not Found`，并带有结构化 JSON。

示例：

```json
{
  "title": "No Definitions Found",
  "message": "Sorry pal, we couldn't find definitions for the word you were looking for.",
  "resolution": "You can try the search again at later time or head to the web instead."
}
```

只有这一类合同，才应该映射成产品层错误：

- `Word not found`

实现规则：

- 不能把泛化的 4xx 都映射成 `Word not found`
- 只有当响应是 `404` 且 body 符合 `dictionaryapi.dev` 的 not-found payload 合同时，才允许映射成 `Word not found`

## 网络错误与代理错误的分离

网络和代理失败必须和“单词不存在”严格区分。

以下情况都**不能**映射成 `Word not found`：

- 浏览器直连网络失败
- 上游直连 `5xx`
- 自定义 proxy 鉴权/配置失败
- 同源 `/api/proxy` 请求失败
- 与上游 not-found payload 无关的 proxy `4xx` / `5xx`

实现规则：

- 尽量保留具体的 proxy/network/service 错误信息
- 只有真实的上游 not-found 合同才使用 `Word not found`

## Transport 合同

词典 transport 是预定义配置，不是运行时 fallback 策略。

- `direct`
  - 浏览器直接请求词典 endpoint
- proxy transport
  - Lite 使用配置的自定义 proxy URL
  - Cloud 使用同源 `/api/proxy`

这个合同是显式的，因为词典 provider 的 CORS 行为应当是预先已知的。

## 未来新增 Provider 的规则

如果以后接入新的词典后端，不能默认继承 `dictionaryapi.dev` 的错误合同。

每新增一个词典后端，必须补齐：

1. provider 专属 API 合同文档
2. 请求格式
3. 成功响应格式
4. not-found 语义
5. transport 模式
6. proxy/network/service 错误与业务 not-found 的区分

不能默认假设：

- 新 provider 也用 `404` 表示单词不存在
- 新 provider 也返回相同的 JSON not-found payload
- 新 provider 也应该直接复用 `Word not found` 语义
