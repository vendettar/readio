---
title: Backend SQLite Governance
---

## 1. Purpose

This document records the long-lived SQLite rules for Cloud backend features.

These rules are not tied to a single feature implementation. They are the shared persistence contract for backend-owned state such as ASR usage governance and feed snapshot storage.

## 2. Migration Contract

All backend SQLite schema creation and evolution must follow one path:

- versioned SQL migrations in `apps/cloud-api/migrations`
- `goose` as the migration runner
- startup-time migration application before request serving
- no ad hoc `CREATE TABLE IF NOT EXISTS` or feature-local schema bootstrap

Production code, local development, and tests must depend on the same migration path. A test-only schema path is not acceptable.

## 3. Bootstrap Contract

An empty database is valid if the schema exists and zero rows are present.

That means:

- startup may create/apply schema, but does not need to seed feature rows
- feature code must handle empty-table bootstrap states cleanly
- absence of rows is not itself a corruption signal

## 4. Source Of Truth Discipline

Every backend SQLite feature must define which tables are authoritative and which state is derived.

Rules:

- source-of-truth tables own correctness
- helper tables, projections, counters, or cached summaries must not silently become authoritative
- if helper state is rebuilt, rebuild logic must derive from authoritative rows
- docs must say when a feature is a bounded cache/snapshot rather than a canonical archive

## 5. ASR Persistence Rules

Built-in ASR persistence is backend-owned operational state.

The durable contract includes:

- request identity / idempotency tracking
- reservation lifecycle states such as `reserved`, `consumed`, `released`, and `failed`
- deterministic usage accounting for the deployment-wide quota window
- cleanup of abandoned or stale reservations by transitioning them to a terminal released state instead of leaving them indefinitely in-flight

Built-in ASR SQLite storage is not a transcript archive. It must not claim ownership of raw audio or transcript text persistence unless a later feature explicitly adds that contract.

## 6. Feed Persistence Rules

Feed persistence is a backend cache/snapshot layer, not a canonical long-term archive.

The durable contract includes:

- feed-level metadata plus bounded episode-window storage
- parsed snapshot storage, not raw XML persistence
- paging over the stored snapshot window
- explicit freshness and refresh behavior owned by backend policy

Important boundary:

- deep pagination only guarantees access to the retained snapshot window
- docs must not imply full historical feed coverage unless the stored contract truly provides it

If response shapes do not expose retention boundaries clearly, surrounding docs must still state that the route pages over the stored snapshot, not over the infinite theoretical feed history.

## 7. Schema Design Expectations

Backend SQLite schema should encode important invariants where practical.

Examples:

- primary keys and unique indexes for stable identities
- foreign keys where ownership is explicit
- `CHECK` constraints for booleans or bounded enums when that reduces invalid state
- explicit timestamps for freshness and lifecycle transitions

This does not mean every product rule belongs in SQL, but invalid low-level states should not be left entirely to application convention when the schema can guard them cheaply.

## 8. Documentation Rule

If a new backend SQLite feature lands, its owner must update Cloud docs with:

- what the database is authoritative for
- what it is not authoritative for
- how schema changes ship
- whether the stored data is archival, bounded-cache, or operational-ledger state
