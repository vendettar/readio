---
title: 后端 SQLite 治理
---

## 1. 目的

这篇文档记录 Cloud 后端长期有效的 SQLite 规则。

这些规则不属于某一个单独功能，而是后端持久化的共享 contract，适用于 built-in ASR usage governance、feed snapshot storage 等 backend-owned state。

## 2. Migration Contract

所有后端 SQLite schema 的创建和演进都必须走同一条路径：

- versioned SQL migrations，放在 `apps/cloud-api/migrations`
- 使用 `goose` 执行 migration
- 后端启动时先应用 migration，再开始处理请求
- 不允许 ad hoc `CREATE TABLE IF NOT EXISTS` 或 feature-local schema bootstrap

Production、local development 和 tests 必须依赖同一条 migration 路径。不能存在一条只给测试使用的独立 schema 路径。

## 3. Bootstrap Contract

只要 schema 已存在，即使数据库里还没有任何业务行，空数据库也是有效状态。

这意味着：

- 启动时可以创建或应用 schema，但不需要预先 seed feature rows
- feature code 必须正确处理 empty-table bootstrap state
- 没有数据行本身不能被当成损坏信号

## 4. Source Of Truth 纪律

每个后端 SQLite 功能都必须明确哪些表是 authoritative，哪些状态只是 derived。

规则：

- source-of-truth 表负责 correctness
- helper tables、projections、counters、cached summaries 不能悄悄变成 authoritative
- 如果 helper state 需要重建，重建逻辑必须从 authoritative rows 推导
- 文档必须说明该功能到底是 bounded cache/snapshot，还是 canonical archive

## 5. ASR 持久化规则

built-in ASR 的持久化属于 backend-owned operational state。

它的 durable contract 包括：

- request identity / idempotency tracking
- `reserved`、`consumed`、`released`、`failed` 等 reservation lifecycle state
- 面向部署级 quota window 的确定性 usage accounting
- 对 abandoned 或 stale reservation 做清理时，要把它们转成 terminal released state，而不是永久保留为 in-flight

built-in ASR 的 SQLite 存储不是 transcript archive。除非后续功能明确扩展，否则它不能宣称持久化 raw audio 或 transcript text。

## 6. Feed 持久化规则

feed 持久化是后端 cache/snapshot 层，不是 canonical long-term archive。

它的 durable contract 包括：

- feed-level metadata 加上有界 episode-window 存储
- 持久化 parsed snapshot，而不是 raw XML
- 在已存储 snapshot window 上做 paging
- freshness 与 refresh 行为由后端策略明确拥有

重要边界：

- deep pagination 只保证访问已保留的 snapshot window
- 除非真实存储 contract 支持完整历史，否则文档不能暗示 full historical feed coverage

如果 response shape 没有把 retention boundary 暴露得很清楚，外围文档也必须明确说明：这个 route 分页的是 stored snapshot，不是理论上无限长的 feed 历史。

## 7. Schema 设计预期

后端 SQLite schema 应在合适的层级编码重要约束。

例如：

- 用 primary keys 和 unique indexes 固定稳定 identity
- 在 ownership 明确时使用 foreign keys
- 对 boolean 或 bounded enum 使用 `CHECK` constraints，减少无效状态
- 用明确的 timestamps 记录 freshness 和 lifecycle transition

这不意味着每条产品规则都要塞进 SQL，但如果 schema 能低成本阻止无效底层状态，就不应该完全依赖应用层约定。

## 8. 文档规则

如果一个新的 backend SQLite feature 落地，它的 owner 需要同步更新 Cloud 文档，说明：

- 数据库对什么是 authoritative
- 对什么不是 authoritative
- schema 通过什么方式发布
- 持久化的数据是 archival、bounded-cache，还是 operational-ledger state
