# Cloud Handoff

This document is the entry point for Cloud handoff.
It records the implemented `apps/cloud-ui` + `apps/cloud-api` contracts and should stay aligned with the actual code.

## 0. Documentation Map

Use the Cloud handoff tree as the SSOT for app-level frontend/backend contracts:

- `handoff/architecture.mdx`
- `handoff/database.mdx`
- `handoff/environment.mdx`
- `handoff/standards.mdx`
- `handoff/features/*`
- `frontend/*`

`handoff/` should stay focused on ownership and contracts. Long-lived frontend implementation notes live under `cloud/frontend/`.

## 1. Current Purpose

- Build `apps/cloud-ui` as a full Lite-equivalent frontend clone.
- Deploy production `apps/cloud-ui` through Cloudflare Pages on `www.readio.top` / `readio.top`.
- Deploy production `apps/cloud-api` on the VPS behind `api.readio.top`; colocated deployments may still serve `apps/cloud-ui/dist` from `apps/cloud-api`.
- Keep discovery flows backend-owned through `apps/cloud-api`, including search, top, Editor's Picks batch lookup, podcast lookup, and PI episode-list lookup. Production reaches these routes through `VITE_API_BASE_URL`.
- Prefer direct browser media access when available, and use `apps/cloud-api` as the built-in fallback proxy only for supported media failure classes.
- Keep browser-local downloads, local transcripts, and other browser-owned media artifacts browser-local; clearing site data still removes them.
- Allow backend-owned shared podcast transcript assets for eligible podcast episodes to persist separately under deployment-owned storage.
- Keep Cloud Settings free of the `CORS Proxy` block.
- Route Cloud ASR provider transcription through a dedicated relay in `apps/cloud-api`, while keeping provider credentials user-owned and browser-supplied per request.
- Keep Cloud ASR provider verification/readiness browser-direct.
- Keep Workbox/PWA disabled.
- Provide the `/api/proxy` RSS proxy contract.
- Use SQLite for the Cloud backend and keep the lifecycle explicit.

## 2. Static Serving Strategy

- `apps/cloud-api` serves the built Cloud UI output directly.
- Non-API routes fall back to `index.html` only when a direct file does not exist.
- Existing files, including extensionless assets, are served directly when present.
- `/api/*` routes remain owned by API handlers and must not fall back to the SPA shell.
- The backend resolves `dist/` from an explicit `READIO_CLOUD_UI_DIST_DIR` override first, then falls back to colocated `dist/` or repository-local `apps/cloud-ui/dist`.

## 3. Proxy Contract

- `POST /api/proxy` remains the JS client proxy surface.
- The POST request body uses the JSON proxy contract: `{ "url": "...", "method": "GET" | "HEAD", "headers"?: { ... } }`.
- `GET /api/proxy?url=...` is supported only for the built-in same-origin native-audio fallback, with a narrow request-header allowlist (`Range`, `If-Range`, and related cache-safe media headers).
- The Cloud media proxy is protected by origin allowlists, request validation, SSRF checks, and rate limits; it does not use browser-public proxy auth tokens.
- The proxy validates the requested target before dialing and rejects private, loopback, link-local, or metadata-style destinations.
- The request path is fail-closed: it uses a pinned validated target set and does not follow redirects into disallowed destinations.
- The proxy uses a custom user agent, applies a bounded fetch timeout, and keeps a basic in-memory rate limit.
- `/api/proxy` remains a public same-origin media egress relay, not a general outbound fetch API. Origin checks are browser abuse controls, not a secret boundary; every request must keep SSRF validation, rate limiting, and structured `slog` fields for `client_ip`, `origin`, `upstream_host`, status, and error class so Grafana can surface misuse.

## 4. SQLite Bootstrap Status

- SQLite is the Cloud persistence target.
- The bootstrap path opens the database, applies the platform pragma baseline, runs goose migrations, and closes cleanly on shutdown.
- Shared podcast transcript asset metadata now live in backend SQLite, while the canonical transcript cue payload is stored as backend-owned `json.gz` files under `PODCASR_TRANSCRIPTS_DIR`.
- Browser-local transcript artifacts remain browser-owned; this backend storage layer is separate and only applies to eligible shared podcast transcript assets.

## 5. Boundaries

- Do describe Cloud as a Lite-equivalent frontend clone deployed to Cloudflare Pages in production, with the Go backend deployed separately on the VPS.
- Do describe discovery requests as backend-owned flows reached through `VITE_API_BASE_URL` in production, including search, top, Editor's Picks batch lookup, podcast lookup, and PI episode-list lookup.
- Do describe Cloud Settings as no longer including the `CORS Proxy` block.
- Do describe Cloud media as browser-direct first, with backend fallback only for approved failure classes.
- Do describe Cloud ASR provider transcription as relay-owned through a dedicated backend route.
- Do describe provider credentials in Cloud ASR as user-owned and browser-supplied per request.
- Do describe Cloud ASR provider verification/readiness as still browser-direct.
- Do describe backend media fallback as pass-through and non-caching.
- Do describe browser-local media artifacts as remaining browser-local.
- Do note that Cloud keeps Workbox/PWA disabled.
- Do not claim schema, sync, auth, or multi-user SaaS behavior.
- Do not claim all audio always goes through backend.
- Do not claim `/api/proxy` handles provider transcription.
- Do not claim Cloud stores user ASR provider credentials server-side.
- Do not claim generic server-side media caching or a server-backed browser library.
- Do describe backend-owned shared podcast transcript assets as a narrow persistent backend artifact contract for eligible podcast episodes.
- Keep future Cloud evolution in docs that are explicitly labeled as future direction.

## 6. Media Fallback Audit

- The 006a media audit is recorded in `media-fallback-audit.md`.
- It classifies native audio, JS fetch prefetch, `HEAD`, `GET`, `Range GET`, transcript fetch, local `blob:` playback, and tracking URL unwrap behavior.
- The current runtime contract keeps direct browser transport first and backend fallback narrow, pass-through, and non-caching for approved failure classes.
- Keep this entry high-level; implementation details belong in the audit sub-doc.

## 7. ASR Relay Audit

- The 008a ASR transport audit is recorded in `asr-relay-audit.md`.
- It separates media fetch fallback from provider transcription submission and provider verification/readiness checks.
- The current Cloud contract now routes provider transcription through a dedicated same-origin relay, while provider verification/readiness stays browser-direct.
- Browser-local blobs and downloaded blobs remain browser-local inputs; the relay does not move them into backend persistence.
- Keep this entry high-level; implementation details belong in the audit sub-doc.

## 8. Runtime Config Ownership Audit

- The runtime config ownership audit is recorded in `runtime-config-ownership-audit.md`.
- It classifies config into three ownership buckets: server-owned runtime env, browser-public runtime config fetched from `GET /api/v1/config` or emitted by colocated backend `/env.js`, and browser-local/user-local config.
- Production Cloud runtime config is served by `apps/cloud-api` using an explicit allowlist, not dumped from process env.
- `apps/cloud-ui/public/env.js` is dev/default-template oriented only.
- Server-owned config stays in VPS env or systemd env files (e.g., `/etc/readio/readio-cloud.env`).
- Browser-public fields (e.g., `READIO_ASR_RELAY_PUBLIC_TOKEN`) are not secrets; they are lightweight abuse-control tokens.
- Keep this entry high-level; implementation details belong in the audit sub-doc.
