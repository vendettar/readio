title: 播客发现 (Explore)
---

## 1. 范围与原则

- 路由：`/explore`、`/podcast/$country/$id`、`/podcast/$country/$id/$episodeKey`
- 标识契约：节目路由使用数值型 `podcastItunesId`；单集路由使用基于稳定单集身份生成的 compact key。UUID 形态的身份继续沿用旧的 22 字符 `base64url` 形式，非 UUID 身份则使用新的通用 compact 形式。
- 路由国家契约：对于会从 browser-local 记录深链回内容路由的入口（Favorites、History、Downloads 以及其他 library 来源 surface），持久化的 `countryAtSave` 属于 route identity 的一部分。生成路由时必须复用保存下来的 `countryAtSave`，不能静默替换成当前全局 content region。
- Hybrid 边界：
  - Search / Top Shows / Top Episodes 的首跳：Apple
  - Editor's Picks 的首跳：PodcastIndex `podcasts-batch-byguid`
  - 后续详情/富化：PodcastIndex
- PodcastIndex 的可用性信号（例如 `dead`）只属于后端边界。Cloud 在输出 Readio DTO 前过滤 dead feed，前端 schema 不暴露 `dead`。

## 2. 首跳所有权

- 播客搜索：Apple Search
- 单集搜索：Apple Search
- Top Shows：Apple chart feed
- Top Episodes：Apple chart feed
- Editor's Picks：PodcastIndex batch-byguid

## 3. 二跳所有权

- 节目元数据：PodcastIndex `podcasts/byitunesid`
- 单集匹配：优先按稳定单集身份命中已缓存 RSS
- RSS 兜底：通过同源 Cloud feed route 按 `feedUrl` 获取，并作为后续单集解析的事实来源

## 4. 缓存模型

- 前端：TanStack Query 内存缓存
- 后端：
  - top / lookup 路由：带优雅降级的 relay 内存缓存
  - feed 路由：独立的小型进程内 parsed-feed 缓存，短 TTL、fresh-only，不提供 stale fallback
  - feed cache key：仅使用归一化后的 `feedUrl`
  - 省略 `limit` 时，后端使用默认响应窗口（`20`）
  - 显式 `limit` 只负责把响应裁成 `min(limit, len(episodes))`；它不会进入 cache key，也不会改变缓存里的完整 parsed feed
- 前端 feed window 契约：
  - `PodcastShowPage` 不显式传 `limit`，因此使用后端默认窗口
  - `PodcastEpisodesPage` 当前也不显式传 `limit`，因为这个页面仍然承载 “see all” 语义，且还没有分页 / load-more
  - episode route 的 RSS fallback 也不显式传 `limit`，以保证较旧的 deep link 仍然可以从完整 feed 中解析出来
  - 当未来引入显式大窗口时，TanStack Query 的 feed key 必须区分请求窗口，避免后端默认窗口和更大窗口在前端 cache 中发生碰撞
- Cloud 不再使用长期持久化的前端 discovery 缓存
