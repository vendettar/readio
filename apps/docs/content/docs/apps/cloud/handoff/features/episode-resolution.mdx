---
title: Episode Resolution Contract
---

## 1. Durable Identity

- Show route SSOT: numeric `podcastItunesId`
- Episode route SSOT: compact key derived from stable episode identity. UUID identities keep the legacy 22-character key; non-UUID identities use the generic compact form.
- Library-origin deep links must also preserve route country. If a Favorites / History / Downloads / local persisted record carries `countryAtSave`, resolution must use that saved route country rather than the current global content region.

## 2. Resolution Order

1. Decode compact key to stable episode identity
2. Query PodcastIndex `podcasts/byitunesid`
3. Ask the Cloud SQLite retained snapshot route for direct detail by `podcastItunesId + episodeGuid`
4. Episode detail performs a direct SQLite lookup and does not own podcast snapshot freshness
5. If the target `guid` is absent from the retained snapshot, end in `episodeNotFound`

## 3. Rule

Cloud must keep episode detail resolution owned by the SQLite retained snapshot route for page routes and must not reintroduce RSS fallback or Apple supplemental resolution as a normal path.

## 4. Entry-Point Strategy

Episode resolution is not the same for every discovery entry point.

### TopEpisode

`TopEpisode` is a budgeted resolver flow, not a canonical direct-identity route.

Rules:

- `TopEpisode` first-hop payload must not be treated as canonical episode identity
- the app may use `podcastItunesId` plus title to run a snapshot-backed resolver
- the resolver budget is explicit:
  - stop when scanned episodes become older than `now - 30 days`
  - stop after scanning 60 retained snapshot rows total
- once that budget is exhausted, fall back to the canonical podcast show page
- do not escalate from this budgeted resolver into any RSS/full-feed fetch

This means `TopEpisode` is an enhancement path:

- if high-confidence resolution succeeds inside the budget, navigate to the episode detail page
- otherwise, navigate to the show page

### SearchEpisode

`SearchEpisode` is a canonical episode-route entry when it already carries a valid stable episode identity.

Rules:

- if `podcastItunesId + episodeGuid + valid route country` are present, the app may build the canonical episode route directly
- this direct route is allowed because the destination episode page still performs canonical hydration
- `SearchEpisode` direct routing must not add legacy query-hint route params such as feed-owned hints, `audioUrl`, `title`, or `source`

### Cold Episode URL

A cold-opened canonical episode URL follows the same canonical episode-detail hydration path as a direct-routed `SearchEpisode`.

That means:

- the route itself is already canonical
- the destination page must hydrate via PI lookup plus the SQLite retained snapshot contract
- this path is not subject to the `TopEpisode` 30-day / 60-episode resolver budget

## 5. Failure Policy

### TopEpisode failure

If the `TopEpisode` budgeted resolver cannot produce a unique high-confidence episode match inside the resolver budget, the app must fall back to the canonical podcast show page.

### SearchEpisode and cold canonical episode URL failure

For `SearchEpisode` direct-route entries and cold canonical episode URLs, the destination episode page follows this policy:

1. resolve canonical podcast metadata through `podcasts/byitunesid`
2. request direct episode detail from the Cloud SQLite retained snapshot route using `podcastItunesId + episodeGuid`
3. do not trigger PodcastIndex refresh from episode detail
4. if the target episode is absent from that retained snapshot, render `episodeNotFound`

This failure policy is intentionally different from `TopEpisode`.

- `TopEpisode` is budget-limited before navigation
- `SearchEpisode` and cold canonical episode URLs are canonical routes first, then hydrate, then render `episodeNotFound` if the retained snapshot does not contain the target GUID after any required refresh

## 6. PI Episode List Cache Contract

Episode resolution, show page bootstrap, and the episodes page now converge on the backend SQLite paginated snapshot for each `podcastItunesId`.

Rules:

- the backend stores a bounded local snapshot in `podcast_shows`, `podcast_episodes`, and `podcast_cache_state`
- freshness is snapshot-level and comes from backend SQLite state, not from a browser-owned full-copy cache
- show page and episodes page use paginated episode-list queries over the retained snapshot window
- canonical episode detail uses direct lookup by `podcastItunesId + episodeGuid`
- the frontend episode-detail query key includes route country: `['podcast', 'episode-detail', podcastId, country, episodeGuid]`
- duplicate `episodeGuid` entries are deduplicated deterministically in the stored snapshot
- `storedTotal` is derived from `COUNT(*)` over `podcast_episodes`
- incremental refresh derives `since` from the latest retained `podcast_episodes` row selected with `ORDER BY published_at_unix DESC LIMIT 1`
- `isTruncated` means the retained snapshot window may not contain every historical episode

Invalidation rules:

- stale or failed snapshot refresh state is represented by backend cache-status metadata
- when backend refresh writes a new snapshot, the route response reflects the new retained window and metadata
- the app must not silently overwrite one podcast's retained snapshot with another podcast's payload
- invalid external URLs from the PI episode list must not survive into UI-openable actions

Implication:

- `TopEpisode` remains a budgeted pre-navigation resolver
- `SearchEpisode` and cold canonical episode URLs remain canonical-route-first hydration flows
- but all canonical page routes must converge on the same SQLite-backed retained snapshot contract once page hydration begins
