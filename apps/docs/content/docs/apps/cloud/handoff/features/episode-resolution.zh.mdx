---
title: 单集解析契约
---

## 1. 持久身份

- 节目路由 SSOT：数值型 `podcastItunesId`
- 单集路由 SSOT：基于稳定单集身份生成的 compact key。UUID 身份继续沿用旧的 22 字符 key，非 UUID 身份则使用通用 compact 形式。
- 来自 Library 的深链还必须保留路由国家。如果 Favorites / History / Downloads / 本地持久化记录里带有 `countryAtSave`，解析时必须使用这个已保存的路由国家，而不是当前全局 content region。

## 2. 解析顺序

1. 将 compact key 解码为稳定单集身份
2. 请求 PodcastIndex `podcasts/byitunesid`
3. 向 Cloud SQLite retained snapshot 路由请求 `podcastItunesId + episodeGuid` 的直接详情
4. 单集详情执行 SQLite 直接查找，不负责节目 snapshot 新鲜度
5. 如果目标 `guid` 不在 retained snapshot 中，则结束于 `episodeNotFound`

## 3. 规则

Cloud 必须保持单集详情解析由 SQLite retained snapshot 路由拥有，不得把 RSS fallback 或 Apple supplemental episode resolution 重新带回内容页主路径。

## 4. 入口策略

不同 discovery 入口的单集解析策略并不相同。

### TopEpisode

`TopEpisode` 属于预算型 resolver 流程，不是 canonical 直接身份路由。

规则：

- `TopEpisode` 的 first-hop payload 不能被当作 canonical 单集身份
- 应用可以使用 `podcastItunesId + title` 运行一个基于 retained snapshot 的 resolver
- 这个 resolver 的预算是显式的：
  - 当扫描到的单集早于 `now - 30 days` 时停止
  - 总扫描量达到 60 条 retained snapshot row 时停止
- 一旦预算耗尽，必须回退到 canonical 节目详情页
- 不允许从这个预算型 resolver 再升级成任何 RSS / full feed 拉取

这意味着 `TopEpisode` 只是增强路径：

- 如果在预算内高置信度解析成功，就进入单集详情页
- 否则进入节目详情页

### SearchEpisode

当 `SearchEpisode` 已经携带有效稳定单集身份时，它属于 canonical 单集路由入口。

规则：

- 当 `podcastItunesId + episodeGuid + valid route country` 均存在时，允许直接构造 canonical 单集路由
- 允许 direct route 的原因是：目标单集详情页仍然会继续执行 canonical hydration
- `SearchEpisode` direct routing 不得追加旧的 feed-owned hint、`audioUrl`、`title`、`source` 等 query-hint 路由参数

### 冷打开单集 URL

冷打开的 canonical 单集 URL 与 direct-route 的 `SearchEpisode` 走同一条 canonical 单集详情 hydration 链路。

也就是说：

- 路由本身已经是 canonical
- 目标页面必须继续通过 PI lookup 加 SQLite retained snapshot contract 完成 hydration
- 这条路径不受 `TopEpisode` 的 `30 days / 60 episodes` resolver 预算约束

## 5. 失败策略

### TopEpisode 失败

如果 `TopEpisode` 的预算型 resolver 无法在预算内产出唯一且高置信度的单集匹配，应用必须回退到 canonical 节目详情页。

### SearchEpisode 与冷 canonical 单集 URL 失败

对于 `SearchEpisode` 的 direct-route 入口以及冷 canonical 单集 URL，目标单集页遵循下面的失败策略：

1. 先通过 `podcasts/byitunesid` 解析 canonical podcast metadata
2. 使用 `podcastItunesId + episodeGuid` 向 Cloud SQLite retained snapshot 路由请求直接单集详情
3. 单集详情不触发 PodcastIndex refresh
4. 如果 retained snapshot 里没有目标单集，则渲染 `episodeNotFound`

这条失败策略与 `TopEpisode` 是故意不同的：

- `TopEpisode` 是“导航前、预算型”的 resolver
- `SearchEpisode` 与冷 canonical 单集 URL 是“先进入 canonical route，再 hydration；如果必要刷新后的 retained snapshot 里没有目标 GUID，就渲染 `episodeNotFound`”

## 6. PI 单集列表 Cache 契约

单集解析、节目详情页首屏以及全部单集页，现在都收敛到每个 `podcastItunesId` 对应的后端 SQLite 分页 snapshot。

规则：

- 后端把有界本地 snapshot 存在 `podcast_shows`、`podcast_episodes` 和 `podcast_cache_state`
- freshness 来自后端 SQLite snapshot state，不再由浏览器 full-copy cache 决定
- 节目详情页和全部单集页使用保留窗口上的分页单集列表 query
- canonical 单集详情使用 `podcastItunesId + episodeGuid` 直接查找
- 前端 episode-detail query key 必须包含路由国家：`['podcast', 'episode-detail', podcastId, country, episodeGuid]`
- 重复的 `episodeGuid` 会在存储 snapshot 中按确定性规则去重
- `storedTotal` 从 `podcast_episodes` 的 `COUNT(*)` 派生
- 增量刷新通过 `podcast_episodes` 的 `ORDER BY published_at_unix DESC LIMIT 1` 选择最新保留行作为 `since`
- `isTruncated` 表示保留的 snapshot window 可能不包含全部历史单集

失效规则：

- stale 或失败的 snapshot 刷新状态由后端 cache-status 元数据表达
- 当后端刷新写入新 snapshot 后，路由响应反映新的保留窗口和元数据
- 应用不得把一个播客的 retained snapshot 静默覆盖到另一个播客的 `podcastItunesId` 下
- 来自 PI 单集列表的无效外链不得流入 UI 的 external-open action

这意味着：

- `TopEpisode` 仍然是导航前的预算型 resolver
- `SearchEpisode` 与冷 canonical 单集 URL 仍然是 canonical-route-first 的 hydration 流程
- 但一旦进入页面 hydration，这几条 canonical route 都必须收敛到同一份 SQLite-backed retained snapshot contract
