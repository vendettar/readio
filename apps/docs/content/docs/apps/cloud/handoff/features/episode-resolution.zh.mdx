---
title: 单集解析契约
---

## 1. 持久身份

- 节目路由 SSOT：数值型 `podcastItunesId`
- 单集路由 SSOT：基于稳定单集身份生成的 compact key。UUID 身份继续沿用旧的 22 字符 key，非 UUID 身份则使用通用 compact 形式。
- 来自 Library 的深链还必须保留路由国家。如果 Favorites / History / Downloads / 本地持久化记录里带有 `countryAtSave`，解析时必须使用这个已保存的路由国家，而不是当前全局 content region。

## 2. 解析顺序

1. 将 compact key 解码为稳定单集身份
2. 请求 PodcastIndex `podcasts/byitunesid`
3. 先检查该 `podcastItunesId` 对应的前端 PI 单集列表 key family
4. 如果存在与权威 `lastUpdateTime` / `episodeCount` 完全匹配的条目，则直接复用
5. 如果没有 exact-authority 命中，则可以从同播客 key family 里选出最新的一条作为临时的 non-authoritative `initialData`，同时继续发起 authoritative refetch
6. 再按稳定单集身份匹配；如果刷新后的成功 PI 单集列表里没有目标 GUID，则结束于 `episodeNotFound`

## 3. 规则

Cloud 必须保持单集详情解析由 PI 单集列表拥有，不得把 RSS fallback 或 Apple supplemental episode resolution 重新带回内容页主路径。

## 4. 入口策略

不同 discovery 入口的单集解析策略并不相同。

### TopEpisode

`TopEpisode` 属于预算型 resolver 流程，不是 canonical 直接身份路由。

规则：

- `TopEpisode` 的 first-hop payload 不能被当作 canonical 单集身份
- 应用可以使用 `podcastItunesId + title` 运行一个基于 PI 单集列表的 resolver
- 这个 resolver 的预算是显式的：
  - 当扫描到的单集早于 `now - 30 days` 时停止
  - 总扫描量达到 60 条 PI 单集列表 row 时停止
- 一旦预算耗尽，必须回退到 canonical 节目详情页
- 不允许从这个预算型 resolver 再升级成任何 RSS / full feed 拉取

这意味着 `TopEpisode` 只是增强路径：

- 如果在预算内高置信度解析成功，就进入单集详情页
- 否则进入节目详情页

### SearchEpisode

当 `SearchEpisode` 已经携带有效稳定单集身份时，它属于 canonical 单集路由入口。

规则：

- 当 `podcastItunesId + episodeGuid + valid route country` 均存在时，允许直接构造 canonical 单集路由
- 允许 direct route 的原因是：目标单集详情页仍然会继续执行 canonical hydration
- `SearchEpisode` direct routing 不得追加旧的 feed-owned hint、`audioUrl`、`title`、`source` 等 query-hint 路由参数

### 冷打开单集 URL

冷打开的 canonical 单集 URL 与 direct-route 的 `SearchEpisode` 走同一条 canonical 单集详情 hydration 链路。

也就是说：

- 路由本身已经是 canonical
- 目标页面必须继续通过 PI lookup 加共享 PI 单集列表 cache contract 完成 hydration
- 这条路径不受 `TopEpisode` 的 `30 days / 60 episodes` resolver 预算约束

## 5. 失败策略

### TopEpisode 失败

如果 `TopEpisode` 的预算型 resolver 无法在预算内产出唯一且高置信度的单集匹配，应用必须回退到 canonical 节目详情页。

### SearchEpisode 与冷 canonical 单集 URL 失败

对于 `SearchEpisode` 的 direct-route 入口以及冷 canonical 单集 URL，目标单集页遵循下面的失败策略：

1. 先通过 `podcasts/byitunesid` 解析 canonical podcast metadata
2. 先检查共享的前端 PI 单集列表 key family
3. 如果目标单集不在 exact-authority 的 PI 单集列表条目中，则请求 `episodes/byitunesid?max=1000`，重新按稳定身份匹配，并替换缓存中的 PI 单集列表
4. 如果成功返回的 PI 单集列表里仍然没有目标单集，则渲染 `episodeNotFound`

这条失败策略与 `TopEpisode` 是故意不同的：

- `TopEpisode` 是“导航前、预算型”的 resolver
- `SearchEpisode` 与冷 canonical 单集 URL 是“先进入 canonical route，再 hydration，如果批准的 1000 条 PI 单集列表里没有目标 GUID，就渲染 `episodeNotFound`”

## 6. PI 单集列表 Cache 契约

单集解析、节目详情页首屏以及全部单集页，共享同一组前端 PI 单集列表 key family；它们都归属于同一个 `podcastItunesId`。

规则：

- key-family prefix 固定为 `['podcast', 'episodes', podcastItunesId, 'pi-list', 'max-1000']`
- freshness 是 PI-episode-list-level，不是 page-level
- 节目详情页、全部单集页、单集详情页必须复用同一套 PI 单集列表语义
- cache 里只存一份来自 `episodes/byitunesid?max=1000` 的规范化 JSON PI 单集列表
- 重复的 `episodeGuid` 会按确定性规则去重，只保留 PI 单集列表中的第一条
- canonical podcast 的权威 markers（`lastUpdateTime`、`episodeCount`）可以让看起来仍 fresh 的条目失效
- stale 数据只允许作为 bootstrap `initialData` 使用，不能阻止后续的 authoritative refetch
- 当 exact authority key 不存在时，可以扫描同播客 key family 并选择最新的一条做临时 bootstrap
- authority drift 下的 bootstrap 会被显式标记为 non-authoritative，不能满足 exact episode resolution
- authoritative refresh 成功后，旧的同播客 key 变体必须被删除；稳态下每个播客只保留一份完整 PI 单集列表

失效规则：

- 当权威 podcast markers 漂移时，旧的 PI 单集列表条目不能再被视为 fresh coverage
- 当刷新后的 PI 单集列表返回了不同的头部顺序或不同的成员集合时，旧列表必须整体原子替换
- 应用不得把一个播客的 PI 单集列表静默覆盖到另一个播客的 `podcastItunesId` 下
- 来自 PI 单集列表的无效外链不得流入 UI 的 external-open action

这意味着：

- `TopEpisode` 仍然是导航前的预算型 resolver
- `SearchEpisode` 与冷 canonical 单集 URL 仍然是 canonical-route-first 的 hydration 流程
- 但一旦进入页面 hydration，这几条 canonical route 都必须收敛到同一份 PI-owned 单集列表 cache contract
