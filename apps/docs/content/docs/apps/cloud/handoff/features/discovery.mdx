## 1. Scope and Principles

- Routes: `/explore`, `/podcast/$country/$id`, `/podcast/$country/$id/$episodeKey`.
- Identifier contract: show routes use numeric `podcastItunesId`; episode routes use a compact key derived from stable episode identity. Canonical UUID identities retain the legacy 22-character `base64url` form, while non-UUID identities use the newer generic compact form.
- Route country contract: for browser-local records that deep-link back into content routes (Favorites, History, Downloads, other library-origin surfaces), saved `countryAtSave` is part of route identity. Route generation must reuse the persisted `countryAtSave` and must not silently substitute the current global content region.
- Hybrid boundary:
  - Search / Top Shows / Top Episodes first-hop: Apple
  - Editor's Picks first-hop: PodcastIndex `podcasts-batch-byguid`
  - downstream detail/enrichment: PodcastIndex
- PodcastIndex availability signals such as `dead` are backend-only. Cloud filters dead feeds before emitting Readio DTOs; frontend schemas must not expose `dead`.

## 2. First-Hop Ownership

- Podcast search: Apple Search
- Episode search: Apple Search
- Top Shows: Apple chart feed
- Top Episodes: Apple chart feed
- Editor's Picks: PodcastIndex batch-byguid

## 3. Second-Hop Ownership

- Show metadata: PodcastIndex `podcasts/byitunesid`
- Exact episode fallback order:
  1. match cached RSS episodes by stable episode identity when feed data is already present
  2. resolve canonical podcast metadata through PodcastIndex `podcasts/byitunesid`
  3. fetch RSS by `feedUrl` through the same-origin Cloud feed route as the fallback source of truth

## 4. Cache Model

- Frontend: TanStack Query memory cache
- Backend:
  - top / lookup routes: in-memory relay cache with graceful degradation
  - feed route: separate small in-memory parsed-feed cache with short TTL, fresh-only semantics, and no stale fallback
  - feed cache key: normalized `feedUrl` only
  - omitted `limit` uses the backend default response window (`20`)
  - optional explicit `limit` only slices the response to `min(limit, len(episodes))`; it does not change the cache key or the cached full parsed feed
- Frontend feed-window contract:
  - `PodcastShowPage` uses the feed route without an explicit `limit`, so it receives the backend default window
  - `PodcastEpisodesPage` also omits `limit` today because the page still has "see all" semantics and no pagination / load-more path
  - episode-route RSS fallback also omits `limit` so older deep links are still resolvable from the full feed
  - TanStack Query feed keys must distinguish the requested window whenever explicit larger windows are introduced, so backend default and larger windows do not collide in frontend cache
- No long-lived frontend discovery persistence in Cloud
