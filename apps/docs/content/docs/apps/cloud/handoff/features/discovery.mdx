## 1. Scope and Principles

- Routes: `/explore`, `/podcast/$country/$id`, `/podcast/$country/$id/$episodeKey`.
- Identifier contract: show routes use numeric `podcastItunesId`; episode routes use a compact key derived from stable episode identity. Canonical UUID identities retain the legacy 22-character `base64url` form, while non-UUID identities use the newer generic compact form.
- Route country contract: for browser-local records that deep-link back into content routes (Favorites, History, Downloads, other library-origin surfaces), saved `countryAtSave` is part of route identity. Route generation must reuse the persisted `countryAtSave` and must not silently substitute the current global content region.
- Hybrid boundary:
  - Search / Top Shows / Top Episodes first-hop: Apple
  - Editor's Picks first-hop: PodcastIndex `podcasts-batch-byguid`
  - downstream detail/enrichment: PodcastIndex
- PodcastIndex availability signals such as `dead` are backend-only. Cloud filters dead feeds before emitting Readio DTOs; frontend schemas must not expose `dead`.

## 2. First-Hop Ownership

- Podcast search: Apple Search
- Episode search: Apple Search
- Top Shows: Apple chart feed
- Top Episodes: Apple chart feed
- Editor's Picks: PodcastIndex batch-byguid

## 3. Second-Hop Ownership

- Show metadata: PodcastIndex `podcasts/byitunesid`
- Page-rendering PI episode list relay: Cloud relay `GET /api/v1/discovery/podcasts/:itunesId/episodes`
- Upstream page-rendering source of truth: PodcastIndex `episodes/byitunesid?max=1000`
- PI page-rendering episode relay emits `guid` and requires a valid `artworkUrl`; rows without valid episode/feed artwork are dropped at the backend boundary
- Upstream PI may still contain a feed URL, but repo-owned Cloud runtime contracts do not expose or depend on it.
- Page rendering must not use RSS feed XML for:
  - show page episode list
  - episodes page episode list
  - episode detail hydration

## 4. Cache Model

- Frontend: TanStack Query memory cache
- Backend:
  - top / lookup routes: in-memory relay cache with graceful degradation
  - PI episode list relay route: lookup cache with bounded admission; oversized `max=1000` episode lists are returned but not cached
  - no active Cloud feed route or parsed-feed cache remains; `/api/v1/discovery/feed` is retired
  - PI episode list relay cache key: canonical `podcastItunesId` plus authoritative freshness markers
  - page-rendering ownership is a single frontend PI episode-list key family:
    - query-key prefix: `['podcast', 'episodes', podcastItunesId, 'pi-list', 'max-1000']`
    - authority tokens append `lastUpdateTime` / `episodeCount` when available
  - warm bootstrap and detail lookup may scan that same-podcast key family, pick the freshest candidate, and reuse it only as temporary `initialData`
  - if authority markers drift, the stale candidate may still bootstrap the page, but it is non-authoritative and must not be trusted for exact episode resolution
  - after a successful refresh, superseded same-podcast authority variants are deleted so steady state is one surviving full-copy PI episode-list entry per podcast
  - duplicate `guid` entries are collapsed deterministically by keeping the first PI episode-list occurrence
- Legacy feed-field note:
  - upstream feeds may still publish feed transport metadata, but repo-owned Cloud runtime contracts do not expose, persist, or key on it
  - legacy feed transport metadata must not be documented as an active discovery fetch key, route authority, or backend cache key
- No long-lived frontend discovery persistence in Cloud
