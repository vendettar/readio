# Cloud 交接

本文档是 Cloud 脚手架阶段的当前入口。
它只记录已实现的后端契约，并且必须与 `apps/cloud` 的实际代码保持一致。

## 1. 当前目的

- 直接提供 `apps/lite/dist` 构建后的前端产物。
- 暴露当前脚手架阶段的 `/api/proxy` RSS 代理契约。
- 启动 SQLite 作为 Cloud 后端脚手架的持久化基础，并保持生命周期清晰。

## 2. 静态服务策略

- `apps/cloud` 直接提供 Lite 的构建输出。
- 非 API 路由仅在不存在同名文件时回退到 `index.html`。
- 已存在的文件，包括无扩展名资源，都会被直接提供。
- `/api/*` 路由由 API 处理器独占，不能回退到 SPA shell。

## 3. RSS 代理契约

- `GET /api/proxy?url=...` 是当前代理入口。
- 代理在拨号前会校验目标地址，并拒绝私网、回环、链路本地或 metadata 风格目标。
- 请求路径是 fail-closed 的：它使用已验证的固定目标集，并且不会通过重定向跳入不允许的目标。
- 代理使用自定义 user agent，限制上游请求超时，并保留基础的内存限流。

## 4. SQLite 启动状态

- SQLite 是 Cloud 当前的持久化启动目标。
- 启动路径会创建父目录、打开数据库、执行初始化 PRAGMA，并在关闭时清理干净。
- 这一阶段不定义 schema、迁移、同步或认证层。

## 5. 边界

- 不要把单独的 Cloud Web 前端描述为已经实现。
- 不要在这一阶段声称已经具备 schema、同步、认证或多用户 SaaS 行为。
- 未来 Cloud 演进应放在明确标记为 future direction 的文档里。
