title: Database & Persistence
---

## 1. Persistence Model

Cloud frontend still uses browser-local persistence through IndexedDB and localStorage.

- IndexedDB: metadata, sessions, downloads, subtitles, browser-local library state
- localStorage: lightweight non-sensitive settings only
- credentials table: browser-local provider credentials when applicable

## 2. Important Constraint

Cloud backend does not currently replace the browser-local library model.

- Favorites, subscriptions, history, downloaded tracks, and local transcript artifacts remain local to the browser.
- Clearing site data still removes them.

## 3. Runtime Cache Status

- Discovery/search/feed frontend persistence is no longer a long-lived IndexedDB cache in Cloud.
- Cloud discovery now relies on:
  - frontend TanStack Query memory cache
  - backend in-memory cache for Apple / PodcastIndex relay paths
- `runtime_cache` still exists for remaining browser-local runtime caches that still have owners.

## 4. Key Tables Still Relevant to Cloud UI

- `sessions`
- `audioBlobs`
- `subtitles`
- `remote_transcripts`
- `folders`
- `tracks`
- `local_subtitles`
- `subscriptions`
- `favorites`
- `settings`
- `credentials`
- `runtime_cache`

## 5. Operational Rule

Cloud frontend docs should describe browser-local persistence honestly:

- no account sync
- no server-backed recovery
- no long-term archival guarantee
