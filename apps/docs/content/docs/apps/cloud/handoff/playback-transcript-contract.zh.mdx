# 播放与 Transcript 契约

本文档记录 Cloud 当前的播放 / transcript 合同，覆盖 transcript-first 播放与 Reading Area 的 artwork/CTA 状态机。

它是以下行为的参考文档：

- 节目本身已带 transcript source 时的播放语义
- 当前没有显示 transcript 时，Reading Area 如何显示 artwork + CTA
- transcript-first、纯听模式与手动 ASR 生成之间的边界

## 1. 核心产品规则

Cloud 播放当前遵循 **transcript-first** 原则。

如果当前播放 payload 已携带可用 transcript source：

- 应优先走 transcript 加载路径，而不是 automatic ASR
- 不得为了 ASR 自动下载整段音频
- 不得把 transcript source 失败重新解释为 ASR 设置问题

只有当前确实没有 transcript source 时，才允许进入 automatic ASR readiness / download / generation 路径。

## 2. 什么算 Transcript Source

当前 Cloud 契约中，以下都算 `transcript available`：

1. built-in remote transcript
   - 例如 `episodeMetadata.transcriptUrl`
2. 已缓存 / 已 ingestion 的 transcript
   - 例如基于当前 episode/audio identity 命中的 remote transcript cache
3. 本地 / 已下载 transcript
   - 与当前 `localTrackId` 绑定

本合同明确区分：

- `transcript source exists`
- `transcript currently loaded into the Reading Area`

后者并不是前者成立的前提。

## 3. Transcript-First 播放

当一个带 transcript 的远程节目开始播放时：

- transcript-bearing path 必须立刻成为主路径
- full-player 的第一个状态必须是 transcript-loading，而不是 `No transcript yet`
- automatic ASR 必须被跳过
- 为 ASR 而做的 full-audio download 必须被跳过
- 即使 transcript 后续加载失败，播放也可继续独立运行

transcript-bearing failure 仍然是 transcript failure，不应被解释成 ASR readiness failure。

## 4. Reading Area CTA 合同

Reading Area 当前可能渲染：

- transcript 内容
- loading / progress 状态
- error 状态
- artwork + CTA 状态

其中 artwork + CTA 状态必须由 transcript-first 决策边界统一控制。

### 如果 transcript source 存在

Reading Area 必须优先显示：

- `Transcript available`
- `Show Transcript`

不得错误显示：

- `Generate Transcript`
- `Set up transcript generation`

### 如果 transcript source 不存在

此时才允许按 ASR readiness 分流：

- ASR ready -> `Generate Transcript`
- ASR not ready -> `Set up transcript generation`

## 5. 纯听模式

`stream_without_transcript` 是一种展示模式，不是 transcript availability 信号。

它表示：

- 用户当前只听，不自动显示 transcript 内容

它**不表示**：

- transcript source 不存在
- transcript 不可用
- UI 应该降级成 `Generate Transcript`

因此，只要 transcript source 仍存在，即使处于 pure listening mode，Reading Area 也应继续显示：

- `Transcript available`
- `Show Transcript`

## 6. Built-In 与 Local Transcript 的动作边界

`Show Transcript` 必须按 source type 分流。

### Built-in / remote transcript

允许：

- 走既有 remote transcript ingest/load 路径

### Local / downloaded transcript

允许：

- 走本地 transcript 的 load / select / display 路径

禁止：

- 因为缺少 `transcriptUrl` 而回退到 automatic ASR
- 把 local transcript CTA 当成 remote-only transcript fetch

## 7. Local Download / Session Restore 合同

如果 session restore 最终命中一个 local/downloaded track，并且该 track 在存储层已有 transcript：

- 它必须被视为 `transcript available`
- 不得等到 cues 已经渲染后，CTA 才承认 transcript 存在
- 不得先闪出 `Generate Transcript` 或 `Set up transcript generation`

这条合同保证 local restore 与 transcript-first 语义保持一致。

补充一条 Downloads 页面规则：

- `play without transcript` 必须保留 transcript source identity
- 如果一个 downloaded podcast track 本身带有 built-in `transcriptUrl`，那么从 Downloads 入口进入时，仍应显示 `Transcript available` / `Show Transcript`
- `play without transcript` 只是不自动展示 transcript，不等于抹掉 transcript availability

## 8. 下载时持久化 Built-In Transcript

如果一个 podcast episode 在下载时已经暴露 built-in transcript source：

- 下载流程应先按正常路径持久化音频 track
- 同一条流程还应抓取并解析 built-in transcript 为 cues
- 该 built-in transcript 应作为该 downloaded track 的一条本地 subtitle version 持久化
- 因此，Downloads 卡片在下载完成后应能直接显示一条 subtitle row，而不需要先播放或手动导入

失败边界：

- built-in transcript 持久化是附加步骤，不是主下载成功条件
- 即使 built-in transcript fetch/parse 失败，音频下载成功仍应保持有效

## 9. 失败语义

如果 transcript source 存在，但当前 fetch/load 尝试失败：

- 播放仍应独立继续
- UI 可以进入 retry/show-transcript 状态
- UI 不得自动转入 ASR generation

也就是说：

- `source exists but load failed`
  不等于
- `no transcript source exists`

## 10. Transcript 导出兼容性说明

Cloud transcript 导出的是纯文本字幕内容，不是可执行内容。

当前在 macOS 上已观察到的行为：

- 浏览器驱动的 `Blob` 下载，与“先在浏览器打开 transcript，再用 Command+S 保存”并不是同一种保存路径
- Chrome 的 `Blob` 下载可能带有 `com.apple.quarantine` 元数据，因此即使是 transcript 文件，也可能触发 Gatekeeper 风格的验证提示
- 这类提示更像平台/下载路径差异，不代表 SRT 内容本身损坏或格式非法

当前导出兼容性规则：

- SRT 导出不应继续使用通用 `text/plain`
- 当前 playback transcript 导出路径对 SRT 使用 `application/x-subrip;charset=utf-8`
- VTT 导出应使用 `text/vtt;charset=utf-8`
- 当前 transcript 导出在 player surface 上已经改为按目标格式导出，而不是继续绑定原始来源文件名

操作说明：

- 如果用户可以直接在浏览器打开 transcript 链接，并通过 Command+S 正常保存，但自动导出的 transcript 文件会触发 macOS 验证提示，更可能的差异点是下载元数据 / quarantine 处理，而不是 transcript parse 失败
- 这类问题应优先视为下载路径兼容性问题，而不是 transcript 质量问题

## 11. Export 菜单契约

Cloud 当前按 surface 区分 export 入口：

- MiniPlayer 使用 `Export transcript...`
- Downloads 字幕行 hover 动作使用 `Export`
- Downloads 卡片 overflow 仍是另一条独立的音频/export surface

菜单内容：

- MiniPlayer 的二级菜单暴露 transcript 格式：
  - `srt`
  - `vtt`
- Downloads 字幕行的 export 菜单暴露相同的 transcript 格式：
  - `srt`
  - `vtt`
- 这两处应消费同一份 transcript export-format contract，避免未来新增格式时出现 surface 漂移
- Downloads 卡片 overflow 上的 audio export 与 transcript 格式选择仍是分离能力

交互规则：

- 当嵌套 overflow step 处于打开状态时，点击外部区域应直接关闭整个 overflow 菜单
- 这次外部点击不得顺带触发落点下方的页面控件

## 12. Downloads 字幕来源 Badge

Downloads 字幕行应一致地暴露来源身份：

- ASR 生成版本在 metadata 存在时继续显示 provider / model badge
- 节目自带的 built-in transcript 版本应显示 `Built-in` badge
- 用户导入的 transcript 版本应显示 `Imported` badge

展示规则：

- `Built-in` 与 `Imported` 属于来源 badge
- provider / model 仍然是 ASR metadata badge，不应被混作来源 badge
- Downloads 字幕行与 subtitle-version sheet 应使用同一套来源术语

## 13. Reviewer Checklist

Reviewer 应重点检查：

1. 带 transcript 的播放不会自动升级成 ASR
2. `stream_without_transcript` 不会抹掉 transcript availability
3. local/downloaded transcript source 能在 cues 尚未渲染时被正确识别
4. `Show Transcript` 会按 source type 正确分流
5. built-in transcript failure 不会错误显示 ASR setup CTA
6. 下载时 built-in transcript 的持久化不会阻塞或否决主音频下载成功
7. MiniPlayer 与 Downloads transcript export 菜单暴露相同的格式集合

## 13. 边界

本文档会描述：

- transcript-first 播放语义
- artwork/CTA 状态语义
- pure listening mode 语义
- local/downloaded transcript availability 语义
- 下载时 built-in transcript 持久化语义
- transcript export surface / format 语义

本文档不描述：

- 后端 `/api/proxy` transport contract
- ASR provider transport 细节
- transcript storage schema 内部实现
- full player 视觉重设计
