## 1. 持久化分层

Cloud 现在有两层不能混淆的持久化：

- 前端浏览器本地持久化
- `apps/cloud-api` 后端 SQLite 持久化

## 2. 前端浏览器本地持久化

Cloud 前端仍然通过 IndexedDB 和 localStorage 保存浏览器拥有的状态。

- IndexedDB：元数据、sessions、downloads、subtitles、browser-local library state
- localStorage：仅非敏感轻量设置
- `credentials` 表：按需要保存浏览器本地 provider credential

关键约束：

- Cloud 后端当前仍然没有替代浏览器本地图书馆模型
- Favorites、Subscriptions、History、下载音频、本地 transcript 产物仍然只存在于浏览器
- 清除站点数据仍会删除它们

## 3. 后端 SQLite 持久化

Cloud 后端现在也拥有服务端 SQLite 持久化，用于后端治理的状态。

当前和规划中的后端 SQLite 职责包括：

- built-in ASR 的 quota / reservation / idempotency 状态
- Cloud discovery 页面渲染使用的 PodcastIndex 派生节目与单集 snapshot：
  - `podcast_shows`：节目/show 领域元数据
  - `podcast_episodes`：有界保留的单集行，按稳定 newest-first 顺序排列
  - `podcast_cache_state`：刷新、退避、截断、驱逐与访问元数据
- podcast transcript asset 持久化暂搁到 migration `00004`；在它落地前，运行路径不得假设 `podcast_transcript_assets` 已存在，transcript asset storage gate 默认保持 disabled
- 其他需要跨进程重启保留的后端运营状态

Discovery snapshot 约束：

- `podcast_shows` 不应保存刷新时间、失败计数、近似字节数、驱逐优先级等 cache-state 字段
- `podcast_episodes` 是保留单集数量和最新保留发布时间的来源；路由总数使用 `COUNT(*)`，增量刷新通过 `ORDER BY published_at_unix DESC LIMIT 1` 选择最新保留行
- `podcast_cache_state` 只拥有缓存元数据；它不是用户 library state，也不是完整历史归档契约
- PodcastIndex cache SQL 通过 sqlc 从 `apps/cloud-api/internal/db/schema.sql` 和 `apps/cloud-api/internal/db/queries/pi_episode_cache.sql` 生成；在 `apps/cloud-api` 下运行 `go generate ./internal/db` 可重新生成。`apps/cloud-api/internal/podcastindex/cache_store.go` 保留事务编排，并把生成 row 映射回领域结构。

已退役方向：

- 旧的 discovery feed snapshot / feed-episode retention 规划已随着 `/api/v1/discovery/feed` 一并退役
- 文档不得再把 backend SQLite 描述成 discovery feed parsing 或 feed-route cache state 的当前 owner

这层后端 SQLite 不是浏览器本地用户库数据的通用替代品。

## 4. Runtime Cache 现状

Cloud 中 discovery / search 前端已不再依赖长期持久化的 IndexedDB 缓存。

- 前端 discovery 主要依赖 TanStack Query 内存缓存
- 当前活跃的后端 discovery 路由只在适用时使用后端内存缓存；PodcastIndex 节目详情 / 单集列表读取使用上面的 SQLite snapshot 表
- 已退役的 feed 路由不会留下活跃的 SQLite feed-cache 契约
- `runtime_cache` 仍存在，但只服务于当前仍有 owner 的浏览器运行时缓存

## 5. 后端 SQLite 治理摘要

所有 Cloud 后端 SQLite 功能都必须遵循同一套规则：

- schema 变更必须走 versioned SQL migrations
- 使用 `goose` 作为 migration 机制
- 后端启动时必须先应用 pending migrations，再开始对外服务
- 不允许 route-local 或 feature-local 的 ad hoc DDL
- production 与 test 必须依赖同一条 schema 路径
- source of truth 表与 helper/derived state 的边界必须明确记录

更细的后端规则见 [后端 SQLite 治理](/apps/cloud/handoff/backend-sqlite-governance)。

## 7. 文档边界

Cloud 文档必须如实描述持久化 ownership：

- 浏览器本地 library state 没有账号同步
- 仅存在浏览器的数据没有服务端恢复能力
- 除非具体功能明确声明，否则不能把 backend SQLite 描述成用户归档存储
- repo-owned 的 Cloud 运行时契约不会按任何旧的 feed transport 字段持久化或建立 library state key
