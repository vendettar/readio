# Grafana Cloud 集成

Readio Cloud 使用应用侧 Push 路径处理后端指标和日志，并使用 Grafana Faro 处理前端 RUM。

## 1. 后端指标 (OTLP Push)

后端服务通过 OTLP HTTP 协议直接向 Grafana Cloud 推送指标。

### 获取凭证步骤：
1. 登录 [Grafana Cloud](https://grafana.com/products/cloud/) 控制台。
2. 在左侧导航栏点击 **Connections** -> **Add new connection**。
3. 搜索并选择 **OpenTelemetry**。
4. 在配置页面中找到以下信息：
   - **Endpoint**: 类似 `https://otlp-gateway-prod-xxx.grafana.net/otlp`。
   - **Username / Instance ID**: 一个数字 ID（注意：**必须**使用 OpenTelemetry 磁贴详情中的 Username，通常与 Prometheus ID 不同）。
   - **Password / API Token**: 点击 "Generate now" 生成一个 Token（推荐使用 Cloud Access Policy 类型的 Token，并具备 `metrics:write` 权限）。

### 需要配置到 GitHub 的变量：
- Variable: `READIO_GRAFANA_OTLP_ENDPOINT`
- Variable: `READIO_GRAFANA_OTLP_INSTANCE_ID`
- Secret: `READIO_GRAFANA_OTLP_TOKEN`


---

## 2. 前端 RUM (Grafana Faro)

前端监控直接从用户浏览器向 Grafana Faro Collector 上报数据。

### 获取凭证步骤：
1. 在左侧导航栏点击 **Frontend** (或搜索 Faro)。
2. 点击 **Create new app**。
3. App Name 填 `readio-cloud`。
4. 创建后，在 "Instrumentation" 标签页找到 **URL** (Collector URL)。

### 变量配置说明：

**架构注记**：得益于 Readio 的动态配置系统，以下变量**只需**配置在 GitHub Actions/VPS 环境变量中，由后端通过 `/api/v1/config` 接口下发。Cloudflare Pages 侧**不需要**重复配置这些变量（只需保留 `VITE_API_BASE_URL` 即可）。

- Variable: `VITE_GRAFANA_FARO_URL`: 类似 `https://faro-collector-prod-xxx.grafana.net/collect/xxx`
- Variable: `VITE_GRAFANA_FARO_APP_NAME`: `readio-cloud`
- Variable: `VITE_GRAFANA_FARO_ENV`: `production` 或 `preproduction`
- Variable: `VITE_GRAFANA_FARO_SAMPLE_RATE`: 比如 `1.0` 或 `0.1`

## 3. 后端追踪 (Tempo / OTLP Push)

后端分布式追踪通过 OTLP HTTP 协议直接向 Grafana Cloud Tempo 推送 span。追踪默认禁用，与 metrics 共用同一 OTLP endpoint 和认证解析器。

### 启用步骤：
1. 从 Grafana Cloud **OpenTelemetry** tile 获取 OTLP endpoint 和 headers（与 metrics 使用的同一个 tile）。
2. 确保 access policy token 同时具备 `metrics:write` 和 `traces:write` 权限。
3. 设置 `READIO_GRAFANA_OTLP_HEADERS`（推荐），或配置 `READIO_GRAFANA_OTLP_INSTANCE_ID` + `READIO_GRAFANA_OTLP_TOKEN` fallback。
4. 设置 `READIO_TRACING_ENABLED=true`，可选配置 `READIO_TRACING_SAMPLE_RATIO`（默认 `0.1`）。

**重要**：预发初始验证时使用 `READIO_TRACING_SAMPLE_RATIO=1.0`，部署生产前降低至 ≤ 0.1。

### 需要配置到 GitHub 的变量：
- Variable: `READIO_GRAFANA_OTLP_ENDPOINT` *(必需；无论使用推荐的 headers 路径还是 legacy instance/token fallback 都要配置)*
- Secret: `READIO_GRAFANA_OTLP_HEADERS` *(推荐——从 OpenTelemetry tile 直接复制)*
- Variable: `READIO_TRACING_ENABLED` — `true` 启用，省略或 `false` 禁用
- Variable: `READIO_TRACING_SAMPLE_RATIO` — `0.0`–`1.0` 之间的小数

如果使用 instance/token fallback 代替 `READIO_GRAFANA_OTLP_HEADERS`：
- Variable: `READIO_GRAFANA_OTLP_ENDPOINT`
- Variable: `READIO_GRAFANA_OTLP_INSTANCE_ID`
- Secret: `READIO_GRAFANA_OTLP_TOKEN`

### Span 命名与安全说明：
- 入站 HTTP span 名称使用路由模板（如 `GET /api/v1/discovery/{feedURL}`），不包含原始路径、GUID、播客 ID 或 query string。
- Span 属性不包含完整 URL、query string、Referer、原始 ID、搜索文本或用户输入。
- PodcastIndex 和 Apple 出站请求可产生本地子 span，但不向第三方主机注入 `traceparent`。
- 代理/媒体上游不被追踪，也不接收 trace headers。
- 前端 Grafana Faro（`VITE_GRAFANA_FARO_*`）是独立 RUM pipeline，与后端 Tempo 追踪互不关联。

### Loki 到 Tempo 导航：
发送到 Loki 的请求日志在 payload 中包含 `trace_id` 和 `span_id` 字段（非 stream label）。如需在 Grafana Explore 中点击跳转：
1. 打开 Grafana → **Connections** → **Data Sources** → 选择 Loki 数据源。
2. 在 **Derived fields** 下新增字段：
   - **Name**: `trace_id`
   - **Regex**: `"trace_id":"(\w+)"`
   - **Query**: `${__value.raw}`
   - **Internal link**: 开启，指向 Tempo 数据源。
3. 保存后，点击 Loki 日志行中的 `trace_id` 值验证跳转。

## 4. 后端日志 (Loki Push)

后端直接将脱敏后的应用日志推送到 Grafana Cloud Logs/Loki。当三个必需 Loki 值全部配置后，该流水线即启用。

### 需要配置到 GitHub 的变量：
- Variable: `READIO_GRAFANA_LOKI_URL`：Loki push endpoint，通常以 `/loki/api/v1/push` 结尾。
- Variable: `READIO_GRAFANA_LOKI_USER`：Grafana Cloud Logs user / instance ID。
- Secret: `READIO_GRAFANA_LOKI_TOKEN`：Grafana Cloud Logs token。

可选的仅后端变量是 `READIO_LOKI_LOG_LEVEL`、`READIO_LOKI_BATCH_SIZE`、`READIO_LOKI_FLUSH_INTERVAL_SECONDS` 和 `READIO_LOKI_QUEUE_SIZE`。

Loki labels 只能是 `service`、`env` 和 `level`。Request ID、route、upstream host、用户输入、URL、headers、cookies、tokens、本地路径和原始错误不得作为 labels。

## 5. Grafana 日志查询

生产日志查看由 Grafana Cloud Logs/Loki 承担。Cloudflare Pages 上的 `/ops`
路由只是 Grafana 入口，不再由浏览器读取 `/admin/*` 内存日志端点。后端
`/admin/logs` 仍保留给同源部署、SSH tunnel，或未来受保护的 ops 反代。

Grafana Explore 查询：

```logql
{service="readio-cloud", env="preproduction"}
```

```logql
{service="readio-cloud", env="preproduction", level="error"}
```

```logql
{service="readio-cloud", env="production"}
```

```logql
{service="readio-cloud", env="production", level="error"}
```

日志面板查询：

```logql
{service="readio-cloud", env="$env"} | json
```

持续错误告警查询：

```logql
sum(count_over_time({service="readio-cloud", env="production", level="error"}[5m])) > 0
```

告警阈值应在掌握生产基线流量和预期错误量后再调优。

## 6. 线上验收清单

真实 Grafana Cloud 验证需要部署凭证和 Grafana 访问权限。如果实现阶段无法访问这些资源，运维方应在生产最终验收前完成：

**日志（Loki）：**
1. 使用 Loki 环境变量部署到 preproduction。
2. 通过正常 Cloud API 流量生成一条 info 日志和一条 warning/error 日志。
3. 在 Grafana Explore 查询 `{service="readio-cloud", env="preproduction"}`。
4. 查询 `{service="readio-cloud", env="preproduction", level="error"}`。
5. 确认日志在配置的 flush interval 内到达。
6. 确认脱敏日志行不包含 secrets、完整 Referer query、本地文件路径、transcript text 或 authorization headers。
7. 确认启用日志投递后，00B 指标仍正常到达。

**追踪（Tempo）：**
8. 使用 `READIO_TRACING_ENABLED=true` 和 `READIO_TRACING_SAMPLE_RATIO=1.0` 部署到 preproduction。
9. 访问 `/api/v1/config`、一个 discovery 端点和一个 ASR relay 或 verify 端点。
10. 在 Grafana Explore 选择 Tempo 数据源，搜索 `service.name="readio-cloud"` — 确认 traces 出现。
11. 确认入站 span 名称是路由模板（如 `GET /api/v1/discovery/{feedURL}`），而非原始路径。
12. 确认 span 属性不包含完整 URL、query string、Referer、GUID 或用户文本。
13. 打开包含 `trace_id` 的 Loki 日志行 — 确认 Tempo 链接出现（若缺失，配置 derived field，参见第 3 节）。
14. 触发一个 discovery 路由 — 确认出站子 span 出现且无敏感属性。
15. 确认 PodcastIndex 出站请求不携带 `traceparent`。
16. 确认代理/媒体上游不出现在 Tempo waterfall 中。
17. 确认 API 响应包含 `request_id`，不暴露 `trace_id`。
18. 部署生产前将 `READIO_TRACING_SAMPLE_RATIO` 降至 ≤ `0.1`。

## 7. 验证数据流入
- **指标**: 在 Grafana Explore 中搜索 `readio_cloud_http_request_duration_seconds`。
- **追踪**: 在 Grafana Explore 中选择 Tempo 数据源，搜索 `service.name="readio-cloud"`。
- **日志**: 在 Grafana Explore 中使用上面的 Loki 查询。
- **RUM**: 在 Grafana Frontend 页面查看用户会话数据。
