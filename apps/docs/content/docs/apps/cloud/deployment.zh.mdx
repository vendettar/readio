# Cloud 部署指南

本文档与以下工作流保持一致：

- `.github/workflows/release-cloud-build.yml`
- `.github/workflows/deploy-cloud-preprod.yml`
- `.github/workflows/deploy-cloud-prod.yml`

仅记录部署契约。
它不负责 VPS、nginx、TLS、防火墙或 `systemd` 的搭建。

## 0a. 生产环境运行时配置契约

在生产环境 Cloud 中：

- `apps/cloud-api` 动态提供 `/env.js`，**优先于** SPA/静态资源回退。
- Cloud **不使用** `apps/cloud-ui/public/env.js` 或 `dist/env.js` 作为运行时配置基准。
- 后端仅通过显式浏览器安全 allowlist 发出运行时配置。
- 服务端所有的运行时值（监听端口、路径、滥用控制设置）保留在 VPS 环境或 systemd 环境文件（`/etc/readio/readio-cloud.env`）中。
- PodcastIndex 的服务端凭证也保留在 VPS/systemd 运行时环境文件（`/etc/readio/readio-cloud.env`）中，不放到 GitHub Actions variables，也不发到 `/env.js`。

当前契约下不存在单独的静态 Cloud `public/env.js` 模板。
前端读取 `window.__READIO_ENV__`，而 Cloud 的运行时配置所有权来自后端动态生成的 `/env.js`。

浏览器公开的运行时配置字段不是密钥。如果字段被发出到 `/env.js`，假设每个浏览器用户都能读取它。

完整配置审计见 `handoff/runtime-config-ownership-audit.md`。

## 1. 工作流构建内容

- 使用 `pnpm -C apps/cloud-ui build` 在服务器外构建 Cloud UI。
- 使用 `apps/cloud-api` 中的 `CGO_ENABLED=0 GOOS=linux GOARCH=amd64 go build -o readio-cloud .` 构建 Linux 二进制。
- 只暂存发布产物：`readio-cloud` 二进制和 `apps/cloud-ui/dist`。
- 这里明确要求交叉编译，因为 `apps/cloud-api/package.json` 的本地脚手架脚本是 `go build -o /tmp/readio-cloud .`，不能满足 Linux CD 契约。
- Go 二进制读取的是 `PORT`；工作流中的 `CLOUD_PORT` 只是烟测端口，必须与服务的 `PORT` 值一致。

## 2. 必需的 GitHub 配置

使用两个 GitHub Environment：

- `cloud-preproduction`：自动 pre-production 部署
- `cloud-production`：手动 production promote

按 Environment 划分的配置职责：

- `cloud-preproduction` 需要用于部署到 VPS 的 SSH secrets 与 host identity material，以及下面列出的 pre-production 部署拓扑变量
- `cloud-production` 使用同一套 SSH secrets，并额外依赖下面列出的 production-facing variables

Secrets：

- `CLOUD_SSH_HOST`
- `CLOUD_SSH_PORT`
- `CLOUD_SSH_USER`
- `CLOUD_SSH_PRIVATE_KEY`
- `CLOUD_SSH_KNOWN_HOSTS`

Variables：

按 GitHub Environment 分别定义、但字段名相同的部署拓扑变量：

- `CLOUD_REMOTE_ROOT`
- `CLOUD_SERVICE_NAME`
- `CLOUD_PORT`

额外的 production 变量：

- `CLOUD_PUBLIC_BASE_URL`，用于 production 的浏览器侧 smoke check

推荐取值：

- `cloud-preproduction`
  - `CLOUD_REMOTE_ROOT=/opt/readio-pre`
  - `CLOUD_SERVICE_NAME=readio-cloud-pre`
  - `CLOUD_PORT=8079`
- `cloud-production`
  - `CLOUD_REMOTE_ROOT=/opt/readio`
  - `CLOUD_SERVICE_NAME=readio-cloud`
  - `CLOUD_PORT=8080`
  - `CLOUD_PUBLIC_BASE_URL=https://readio.top`

SSH 安全规则：

- 通过 `CLOUD_SSH_KNOWN_HOSTS` 提供主机身份材料。
- 保持 `StrictHostKeyChecking=yes`。
- 不要把 `StrictHostKeyChecking=no` 作为可接受的文档化路径。

## 2a. 远端前置条件 (Stage C: 手动运维)

在工作流能够成功执行之前，目标 VPS 必须已经具备基础环境。除了使用下方的自动化脚本外，你还需要手动完成以下操作：

1. **系统基础工具**：
   - `sudo apt update && sudo apt install rsync curl sqlite3 nginx certbot python3-certbot-nginx`
2. **用户与权限**：
   - 创建与 `CLOUD_SSH_USER` 一致的 Linux 用户，并配置好 SSH 公钥登录。
3. **外部基础设施 (Stage C)**：
   - **Nginx 反向代理**：参考 `scripts/nginx-readio-cloud.conf` 配置反向代理，将域名转发至 `8080` (Prod) 或 `8079` (Pre)。
   - **SSL 证书**：运行 `sudo certbot --nginx` 为域名申请证书。
   - **防火墙**：建议使用 `ufw` 仅开放 `80`, `443` 和 `SSH` 端口。

## 3. 远程目录布局

工作流在 VPS 上预期如下布局：

- `/opt/readio/shared/data/`：持久化 SQLite 数据
- `/opt/readio/shared/data/podcast/transcripts/`：后端持有、可跨用户复用的播客字幕资产
- `/opt/readio/releases/<git-sha>/`：发布产物
- `/opt/readio/current`：运行时指针

运行时契约：

- 服务应从 `current` 指向的发布目录执行二进制，或在 `systemd` 中使用 `WorkingDirectory=/opt/readio/current`
- Cloud 应用必须能从当前激活发布目录解析同目录下的 `dist/`
- SQLite 数据库路径必须保持在 `/opt/readio/shared/data/` 下，或其他位于发布目录之外的共享持久路径中
- `READIO_CLOUD_DB_PATH` 应直接指向 shared persistent data root 下的数据库文件，例如 `/opt/readio/shared/data/readio.db`
- 共享的 built-in 播客字幕资产必须保持在 `/opt/readio/shared/data/podcast/transcripts/` 下，或其他位于发布目录之外的共享持久路径中
- 服务必须为监听端口导出 `PORT`；不要在这个契约里使用 `READIO_PORT`

每个发布目录包含：

- `readio-cloud`
- `dist/`

工作流会先创建远程目录再传输，且不会清空部署根目录，因此 SQLite 的 `data/` 会被保留。

数据库启动契约：

- 必须显式把 `READIO_CLOUD_DB_PATH` 设置为绝对路径；`apps/cloud-api` 不会回退到 release-local 的默认数据库路径
- deployer 必须预先创建 `READIO_CLOUD_DB_PATH` 的父目录
- `apps/cloud-api` 会在启动时先打开 SQLite、验证连通性、建立 `journal_mode=WAL`，再通过 SQLite DSN 应用其余 runtime pragmas，随后自动执行 goose migrations，然后才开始提供请求
- 如果必需的 migration 失败，启动应 fail closed，服务保持不可用，直到 migration 问题被修复

## 3a. 新主机 VPS 启动

如果你在准备新的 VPS，先创建发布目录、安装 `systemd` 单元，并授予部署用户无需交互即可重启和检查 Cloud 服务的权限。

请仅使用当前运行时契约：

- `READIO_CLOUD_UI_DIST_DIR`
- `READIO_CLOUD_DB_PATH`

对于可变的运行时配置，优先使用 env file，而不是把所有值都硬编码进 `systemd` 单元：

- 像 `PORT`、`READIO_CLOUD_UI_DIST_DIR`、`READIO_CLOUD_DB_PATH` 这种稳定路径留在 unit 里。
- secrets 和可调运维参数放到 `/etc/readio/readio-cloud.env`。
- 这样可以保持 `systemd` unit 稳定，同时允许你只修改运行时配置而不反复重写服务定义。

PodcastIndex Phase 1 也使用同一套 env-file 契约：

- `PODCAST_INDEX_API_KEY`
- `PODCAST_INDEX_API_SECRET`
- `PODCAST_INDEX_USER_AGENT`

这些值属于 `apps/cloud-api` 的后端运行时配置，应写入 `/etc/readio/readio-cloud.env`。

不要放到：

- GitHub Actions variables
- 浏览器运行时配置（`/env.js`）
- 前端 `.env` 文件
- `apps/cloud-ui`

1. **运行初始化脚本**：建议使用项目根目录下的脚本进行一键配置：
   ```bash
   # 生产环境
   ./scripts/cloud-setup-vps.sh production
   ```
2. **脚本功能**：
   - 创建目录布局 (`/opt/readio`, `/etc/readio` 等)。
   - 生成 `systemd` 服务单元 (`readio-cloud.service`)。
   - 生成基础环境变量文件 (`/etc/readio/readio-cloud.env`)。
   - 设置 `sudoers` 权限，允许 CD 工作流重启服务。
3. **关键变量配置**：
   - 脚本执行后，请确保在 GitHub Actions Secrets 中配置好所有敏感 Token。
   - 首次部署时，CI/CD 会自动同步这些 Secrets 到服务器。


重要细节：

- 服务使用 `WorkingDirectory=/opt/readio/current`。
- 二进制路径是 `/opt/readio/current/readio-cloud`。
- Cloud UI 静态资源从 `/opt/readio/current/dist` 提供。
- SQLite 文件应保留在发布目录之外，例如 `/opt/readio/shared/data/readio.db`。
- 共享的 built-in 播客字幕资产也应保留在发布目录之外，例如 `/opt/readio/shared/data/podcast/transcripts/`。
- `READIO_CLOUD_DB_PATH` 的父目录由 deployer 预先创建；服务可以假设该目录已经存在。
- `/etc/readio/readio-cloud.env` 是推荐的 Cloud 可变运行时配置位置。
- PodcastIndex supplementary hydration 的凭证和 user agent 也应写在 `/etc/readio/readio-cloud.env`。
- 如果缺少 `PODCAST_INDEX_API_KEY` 或 `PODCAST_INDEX_API_SECRET`，`/api/v1/discovery/podcasts/:itunesId/episodes` 会返回 `503 DISCOVERY_PROVIDER_NOT_CONFIGURED`。
- Cloud ASR relay 的防滥用边界依赖 `READIO_ASR_ALLOWED_ORIGINS`、`READIO_ASR_RELAY_PUBLIC_TOKEN`、`READIO_ASR_RATE_LIMIT_BURST`、`READIO_ASR_RATE_LIMIT_WINDOW_MS`。
- Built-in Cloudflare ASR 能力由 `READIO_CF_ACCOUNT_ID`、`READIO_CF_API_TOKEN` 和 `READIO_ASR_BUILTIN_ALLOWED` 共同决定。
- Built-in ASR 的 quota policy 不会在首次启动时自动启用。部署完成后，需要进入 `/ops` -> `ASR Usage` 显式写入运行时 policy row。
- Cloud `/api/proxy` 的应用级速率限制由 `READIO_PROXY_RATE_LIMIT_BURST` 和 `READIO_PROXY_RATE_LIMIT_WINDOW_MS` 控制。
- 现在已经没有活跃的 `READIO_DISCOVERY_FEED_*` 运行时环境变量。退役 `/api/v1/discovery/feed` 时，应从 Cloud env file 中移除遗留值；当前 discovery surface 只依赖其余 Apple / PodcastIndex relay 配置。
- `READIO_ASR_RELAY_PUBLIC_TOKEN` 只需要在后端运行时环境中配置一次。生产 Cloud 会由 `apps/cloud-api` 动态生成 `/env.js`，把同一个浏览器可见值下发给前端，因此不再需要额外维护一份前端运行时 token。它只是轻量级防滥用值，不是强 secret 边界。
- Cloud ASR relay 是 browser-facing 的 same-origin endpoint。请求必须带合法 `Origin`；缺失或不被允许 of `Origin` 会被直接拒绝。
- `READIO_ASR_ALLOWED_ORIGINS` 是运维显式编写的 allowlist。请写完整 origin，例如 `https://www.readio.top` 或 `https://example.com:8443`；不要依赖 allowlist 内部对默认端口做隐式归一化。
- Built-in ASR readiness 通过 `GET /api/v1/asr/builtin/status` 暴露。它刻意只返回 availability 与窄 reason code；精确 quota 仍然只在 `/ops` 中对运维可见。
- 指标监控采用 OTLP Push 模式，不再使用 Prometheus Pull，因此不需要配置 `READIO_METRICS_TOKEN`。
- 工作流已经使用 `sudo -n`，因此部署用户必须能够无密码执行 `systemctl restart` 和 `systemctl is-active`。
- 工作流会在重启服务前把 `/opt/readio/current` 指向新发布目录，所以新主机第一次重启前必须已经存在对应发布目录。

如果 VPS 的 sudoers 规则只允许 `systemctl restart readio-cloud`，却不允许 `systemctl is-active readio-cloud`，那么 GitHub Actions 里的非交互烟测步骤仍会失败。

## 3b. Pre-production 启动差异

不要为 pre-production 再复制一整份 production bootstrap script。应复用相同结构，只替换下面这些环境相关值：

- `REMOTE_ROOT=/opt/readio-pre`
- `RELEASES_DIR=/opt/readio-pre/releases`
- `SHARED_DATA_DIR=/opt/readio-pre/shared/data`
- `PODCASR_TRANSCRIPTS_DIR=/opt/readio-pre/shared/data/podcast/transcripts`
- `CURRENT_DIR=/opt/readio-pre/current`
- `DB_PATH=/opt/readio-pre/shared/data/readio.db`
- `SERVICE_NAME=readio-cloud-pre`
- `EnvironmentFile=/etc/readio/readio-cloud-pre.env`
- `Environment=PORT=8079`

pre-production 还应保持这些边界：

- 使用独立的 `systemd` unit（`readio-cloud-pre.service`）
- 使用独立的 env file（`/etc/readio/readio-cloud-pre.env`）
- 使用 `/opt/readio-pre/shared/data/` 下独立的 SQLite 数据和 transcript 资产
- 用 Cloudflare Access 保护 `pre.readio.top`，不要依赖应用内 token gate
- service 按需启动即可；即使停止 pre-production service，数据也应持续保留

应用 pre-production 这套变体时，应执行 `systemd` reload 并重启 `readio-cloud-pre.service`，不是 `readio-cloud.service`。

1. **运行初始化脚本**：
   ```bash
   # 预发布环境
   ./scripts/cloud-setup-vps.sh preproduction
   ```
2. **独立性保证**：
   - 使用独立的 `systemd` unit (`readio-cloud-pre.service`)。
   - 使用独立的 env file (`/etc/readio/readio-cloud-pre.env`)。
   - 使用 `/opt/readio-pre/shared/data/` 下独立的 SQLite 数据。
   - 监听端口为 `8079`。


如果一次发布后服务在启动阶段立刻失败，先看 service logs。Cloud 启动链路现在包含自动 goose migrations，因此 migration error 本来就应该阻止服务继续启动，而不是带着不匹配的 schema 提供请求。

建议在 VPS 上做的预启动验证：

```bash
sudo -n "$(command -v systemctl)" restart readio-cloud
sudo -n "$(command -v systemctl)" is-active --quiet readio-cloud
curl -fsS http://127.0.0.1:8080/healthz
```

## 4. 发布流程

部署采用按版本号分段的发布方式，并且仅在指针层面保持原子性：

要求的发布流程是：

1. `Release Cloud Build` 在 `main` 上运行，并按 commit SHA 上传一份 release artifact。
2. `Deploy Cloud Pre-Production` 在 `main` 上成功 build 后自动执行。
3. 针对这个完全相同的 SHA 做 pre-production rehearsal。
4. 只有在 pre-production 通过后，才允许手动触发 `Deploy Cloud Production`。
5. production 必须部署已经通过 pre-production 的同一个 SHA / artifact；不要在 promote 时重新 build，也不要静默切换到更新的 commit。

正式生产发布前，先在 pre-production 环境跑同一个 release，并确保它使用相同的启动路径、SQLite/goose wiring 和 shared-data 布局形态。生产环境不应该成为第一次执行一组新 migrations 的环境。

pre-production 环境规则：

- 只要机器资源足够，pre-production 可以与 production 部署在同一台 VPS 上
- 如果与 production 共用同一台主机，pre-production 仍必须使用独立的端口、release root、shared data root、SQLite 文件、transcript 目录、`systemd` unit 和 env file
- 不要让 pre-production 指向 production 的 SQLite 数据或 production 的 transcript 资产
- 浏览器入口应由 Cloudflare Access 这类基础设施访问控制层保护，不要依赖仅在前端首页做 token gate
- pre-production 可以按需运行，而不是 24/7 常驻；但它的持久化数据应在多次 rehearsal 之间长期保留
- rehearsal 完成后，手动停止 pre-production service 是可接受的运维模型

promote 规则：

- production 部署是手动触发，不是 `main` 上的自动部署
- 手动 production workflow 必须接收一个明确的 commit SHA
- 这个 SHA 必须已经同时具备：
  - 成功的 `Release Cloud Build`
  - 成功的 `Deploy Cloud Pre-Production`
- 如果这个 SHA 没有通过 pre-production，production promote 必须停止

这不是原地覆盖式部署。工作流不会先删除持久化数据，也不会直接往 live 路径里复制构建产物。

## 5. 重启与烟测

deploy workflows 会检查：

- `systemctl is-active --quiet <service>`
- 当前发布目录中存在 `readio-cloud`
- 当前发布目录中存在 `dist/`
- `curl -fsS http://127.0.0.1:${CLOUD_PORT}/healthz`

smoke-check 目标的差异：

- pre-production deploy 检查绑定在 `127.0.0.1:8079` 的本机服务，因为它的公网浏览器入口预期会被 Cloudflare Access 保护
- production deploy 通过 `CLOUD_PUBLIC_BASE_URL` 检查对外浏览器入口

如果需要手动验证运行时状态，也应使用同样的检查。

## 6. 回滚

回滚是手动、按发布目录进行的：

1. 保留上一个成功发布的目录。
2. 用 `ln -sfn` 把 `/opt/readio/current` 指回上一个发布目录。
3. 再次重启服务。
4. 重新执行相同的 smoke 检查。

如果上一个发布目录不存在，就需要从其他已知正常的发布目录恢复。工作流不会自动重新生成回滚产物。

## 7. 可信代理与客户端 IP 契约

Go 后端通过可信代理网关推导有效的客户端 IP。这确保速率限制和滥用控制针对的是真正的浏览器客户端，而非代理链。

### 信任链：Cloudflare → nginx → Go

请求经过三层：

1. **Cloudflare** — 边缘过滤、机器人防护和 DDoS 缓解。
2. **nginx** — VPS 上的反向代理；终止 TLS，将请求转发到本地 Go。
3. **Go (`apps/cloud-api`)** — 对 `/api/proxy` 和 `/api/v1/asr/transcriptions` 执行面向路由的应用级速率限制。

Go 只信任**紧邻的代理跳**（loopback/nginx），不信任 Cloudflare 公共 CIDR。可信代理集合通过 `READIO_TRUSTED_PROXY_CIDRS` 配置。

### nginx real-ip 配置

nginx 必须先从 **Cloudflare** 恢复原始客户端 IP，再把恢复后的客户端 IP 传给 Go。

这一步不要在 nginx 的 `set_real_ip_from` 中使用 loopback 地址。在生产链路 `Cloudflare -> nginx -> Go` 中，nginx 的直接网络对端是 Cloudflare，因此 nginx 在采纳 `CF-Connecting-IP` 时必须信任 **Cloudflare 当前公布的 IP 段**。

请在合适的 `http`、`server` 或包含的 nginx 配置作用域中添加 real-ip 指令，并保持 Cloudflare IP 段为最新：

```nginx
# 下面只是示例；请替换成 Cloudflare 当前公布的 IP 段。
set_real_ip_from 173.245.48.0/20;
set_real_ip_from 103.21.244.0/22;
set_real_ip_from 2400:cb00::/32;

real_ip_header CF-Connecting-IP;
real_ip_recursive on;
```

然后，在代理到 Go 的 `location` 块中配置：

```nginx
# 完成上面的 real-ip 步骤后，$remote_addr 已经是恢复后的客户端 IP。
proxy_set_header X-Real-IP          $remote_addr;
proxy_set_header X-Forwarded-For    $proxy_add_x_forwarded_for;
proxy_set_header X-Forwarded-Proto  $scheme;
proxy_set_header Host               $host;
proxy_pass http://127.0.0.1:8080;
```

`X-Forwarded-Proto` 是后端 same-origin 契约的一部分。它必须与外部请求协议保持一致（即使用 `$scheme`），不要硬编码也不要省略；`apps/cloud-api` 只会在紧邻对端可信时采信它，并用它区分 same-origin fallback 中的 `http` / `https`。

### `READIO_TRUSTED_PROXY_CIDRS` 运行时配置

- **类型**：逗号分隔的 CIDR 列表
- **默认值**：空（无可信代理，所有转发头被忽略）
- **示例**：`127.0.0.1/32,::1/128`
- **范围**：仅后端 —— 不得出现在浏览器 `/env.js` 中

当紧邻的对等方 IP 匹配可信代理 CIDR 时，Go 会采纳 `X-Real-IP`（优先）和 `X-Forwarded-For`（回退）。当对等方不受信任时，转发头被忽略，速率限制直接应用于对等方 IP。

在 `/etc/readio/readio-cloud.env` 中设置：

```bash
READIO_TRUSTED_PROXY_CIDRS=127.0.0.1/32,::1/128
```

### 分层滥用控制

| 层级 | 作用 |
| --- | --- |
| Cloudflare | 边缘过滤、机器人检测、DDoS 缓解 |
| nginx | 反向代理把关、TLS 终止、请求缓冲 |
| Go (`apps/cloud-api`) | 基于有效客户端 IP 的面向路由应用级速率限制 |

各层独立运行。Go 的速率限制器即使请求通过代理链到达，也针对真正的浏览器客户端。

## 8. 运维边界

- 该工作流假设 VPS 已经存在。
- 它不负责安装或管理 `systemd`。
- 它不负责配置 nginx、TLS 或防火墙。
- 它不要求为已批准的媒体 fallback 路径配置单独的 CORS proxy。
- 它不会强制所有音频都走后端；在浏览器可用时仍优先使用浏览器直连媒体访问。
- 对已批准的媒体相关失败类，Go 后端只作为透传 fallback 代理，不缓存也不持久化媒体字节。
- 浏览器本地的下载、转录和设置仍然留在浏览器本地。
- 该工作流不会重新启用 Workbox/PWA；Cloud 当前保持 PWA 关闭。
- 它不部署 SQLite schema、同步、认证或多节点能力。

## 9. ASR 代理 (Cloudflare Workers) 配置

为了规避 Provider (如 Groq) 基于 IP 的速率限制并利用分布式出口 IP，Cloud 版本支持使用可选的 Cloudflare Workers 网关作为 ASR 转写代理。

该 Worker 仅作为**后端出口跳**（`apps/cloud-api` → Worker → Groq），不是浏览器直连的代理，也不是通用出站隧道。当前仅启用 **Groq 转写提交**。

### 9.1 部署 Worker
1. 在 Cloudflare 面板创建新的 Worker。
2. 将 `apps/cloud-api/resources/workers/asr-proxy.js` 中的源码粘贴至编辑器。
3. 在 Worker 的 **Settings -> Variables** 中添加环境变量：
   - `READIO_ASR_WORKER_SHARED_SECRET`: 设置一个长随机高熵字符串（Secret）。
4. 部署 Worker 并记录其公网 URL（例如 `https://asr-proxy.example.workers.dev`）。

### 9.2 后端接入配置
在 VPS 的运行时环境文件 `/etc/readio/readio-cloud.env` 中添加以下字段，使 `apps/cloud-api` 能够正确对接 ASR 代理：

```bash
# ASR Proxy Worker 的基础 URL
READIO_ASR_WORKER_BASE_URL=https://asr-proxy.example.workers.dev
# 必须与 Worker 侧配置的 READIO_ASR_WORKER_SHARED_SECRET 严格一致
READIO_ASR_WORKER_SHARED_SECRET=your-high-entropy-secret-string
```

两个值都必须配置 Worker transport 才会激活。任一缺失或为空时，后端默认走 direct-to-Groq。

回滚方式：清空或删除这两个值并重启服务即可。无论传输模式如何，浏览器 contract 不受影响。

### 9.3 安全性说明
- **鉴权**：后端请求 Worker 时会自动携带 `X-Readio-Cloud-Secret` Header，Worker 校验失败将拒绝转发。
- **收敛**：Worker 源码级限制了仅允许转发至 `api.groq.com`，其他 provider 显式拒绝 (404)，不是通用代理。
- **密钥卫生**：Worker 共享密钥和 provider API key 均不出现在日志或响应体中。
- **性能**：Worker 使用流式透传模式处理音频 Body，支持大体积音频文件。

## 10. 管理后台可观测性

| 变量 | 默认值 | 说明 |
|---|---|---|
| `READIO_ADMIN_TOKEN` | (空 = 禁用) | `/admin/*` 端点的 Bearer token |
| `READIO_ADMIN_LOG_BUFFER` | 2000 | 环形缓冲区容量 (最小 100，最大 10000) |

### 10.1 访问入口

- 浏览器 UI：`/ops`
- 后端 JSON API：
  - `/admin/logs`
  - `/admin/health`
  - `/admin/metrics/summary`
- `/ops` 只是运维查看界面，不替代 `/admin/*` 的后端 token 校验

### 10.2 启用条件与鉴权

- 当 `READIO_ADMIN_TOKEN` 未设置或为空时：
  - `/admin/*` 返回 `404`
  - `/ops` 页面即使能打开，也无法拿到任何 admin 数据
- 当 `READIO_ADMIN_TOKEN` 已设置时：
  - `/admin/*` 需要 `Authorization: Bearer <token>`
  - 缺失或错误的 bearer token 返回 `401`

### 10.3 Token 使用规则

- `READIO_ADMIN_TOKEN` 只存在于服务端
- 不会通过 `/env.js` 输出
- 浏览器 token 只在 `/ops` 页面手动输入
- `/ops` 页面只把 token 存在 `sessionStorage`
- 不要把 token 放进 query 参数、路由参数或硬编码前端配置

### 10.4 推荐的外围收口

- nginx 层建议：
  - 保证 `/ops` 与 `/admin/*` 不可缓存
  - 作为 defense in depth，可在反向代理层额外加 `Cache-Control: no-store`
  - 强烈建议对 `/ops` 与 `/admin/*` 加 IP allowlist
  - **必须**：在站点配置（`/etc/nginx/sites-enabled/default`）的 `location /api/v1/asr/` 下设置 `client_max_body_size 20m;` 以允许 ASR 音频上传（单次请求，需匹配或超过后端限制；Readio 前端限制分块 10MB，后端允许 20MB）。用 `sudo nginx -t` 测试，用 `sudo systemctl reload nginx` 重载
- Cloudflare 层建议：
  - 对 `/ops*` 与 `/admin/*` 做更严格的路径级控制
  - 视情况增加 WAF / rate limit / 国家或 IP 限制
- 这些外围限制不能替代 `READIO_ADMIN_TOKEN`，只是额外加固

### 10.5 运维操作

- 用高熵随机值生成 token，例如：

```bash
openssl rand -base64 48
```

- 将 `READIO_ADMIN_TOKEN` 放到服务端 env file，通常是：
  - `/etc/readio/readio-cloud.env`
- 新增或修改 token 后，重启服务：

```bash
sudo systemctl restart readio-cloud
```

- 如果 token 泄露：
  1. 生成新 token
  2. 更新 `/etc/readio/readio-cloud.env` 中的 `READIO_ADMIN_TOKEN`
  3. 重启服务
  4. 视为所有旧的 `/ops` 会话全部失效

### 10.6 Env 示例

```bash
READIO_ADMIN_TOKEN=replace-with-a-long-random-token
READIO_ADMIN_LOG_BUFFER=2000
```

## 11. 观测性 (OpenTelemetry / OTLP)

`apps/cloud-api` 现在通过 OTLP HTTP 协议直接向 Grafana Cloud 推送指标。这种方式比传统的 Prometheus Pull 模式更节省资源，非常适合 1C1G 等小内存服务器。

运行时契约：
- 指标推送是可选的。如果缺失 `GRAFANA_OTLP_*` 环境变量，监控逻辑会自动降级为无操作（No-op），不会影响服务运行。
- 默认推送间隔为 30 秒，可通过 `READIO_OTLP_PUSH_INTERVAL_SECONDS` 调整。
- **无需在本机安装 Grafana Alloy 或 Prometheus**。

应用指标名称：
| Metric | 类型 | Labels |
| --- | --- | --- |
| `readio_cloud_http_request_duration_seconds` | histogram | `route`, `status_class`, `error_class` |
| `readio_cloud_upstream_request_duration_seconds` | histogram | `provider`, `route`, `status_class`, `error_class`, `cache_status` |
| `readio_cloud_upstream_errors_total` | counter | `provider`, `route`, `error_class` |
| `readio_cloud_asr_relay_requests_total` | counter | `provider`, `mode`, `status_class`, `error_class` |

## 12. 日志 (Loki)

当前版本中，`readio-cloud` 继续将结构化日志输出到 stdout。为了将日志对接到 Grafana Cloud Loki，推荐使用 systemd-journald 配合 Grafana Cloud 的自动抓取功能。

目前推荐的轻量级方案：
1. 保证服务运行在 systemd 中，日志进入 journal。
2. 开发者可以通过 `journalctl -u readio-cloud -f` 实时查看日志。
3. 如果需要集中化存储，可按需配置单机版 `promtail` 或 `alloy` 仅负责日志转发。

## 13. Grafana Faro 浏览器 RUM

`apps/cloud-ui` 可以通过 `/env.js` 下发的浏览器公开运行时配置初始化 Grafana Faro。Faro 默认禁用。

浏览器公开配置：
| 变量 | 默认值 | 说明 |
| --- | --- | --- |
| `VITE_GRAFANA_FARO_URL` | 空 | 公开的 Faro collector URL。为空时禁用 Faro。 |
| `VITE_GRAFANA_FARO_APP_NAME` | 空 | 可选的公开 app name 覆盖值。 |
| `VITE_GRAFANA_FARO_ENV` | 空 | 可选的环境名，例如 `production`。 |
| `VITE_GRAFANA_FARO_SAMPLE_RATE` | `0` | 采样率，范围 `0` 到 `1`。 |

隐私与安全：
- **绝对禁止**将 `GRAFANA_OTLP_TOKEN` 或任何管理 Token 暴露给浏览器。
- Faro 数据传输直接从用户浏览器发往 Grafana Cloud，不经过后端中转。

## 14. 运维总结与回滚

- **指标回滚**：删除或清空 `GRAFANA_OTLP_TOKEN` 并重启服务即可。
- **RUM 回滚**：清空 `VITE_GRAFANA_FARO_URL` 即可通过 `/env.js` 在所有客户端禁用监控。
- **本地诊断**：始终保留 `/ops` 和 `/admin/*` 作为 direct-URL 的轻量诊断手段，它们不依赖 Grafana Cloud 的可用性。

## 15. 运维总结与边界

- 该工作流假设 VPS 已经存在并完成基础系统初始化。
- 运维人员负责管理 Cloudflare Workers 的生命周期及其 Secrets 同步。
- 它不负责配置 nginx TLS 证书自动续期。
- 浏览器本地数据（IndexedDB）的持久化安全性由浏览器同源策略保证，不属于后端部署边界。

## 11. 分离部署 (Cloudflare Pages + VPS)

推荐将前端 SPA 部署在 Cloudflare Pages 以获得更好的全球加速效果，后端 API 继续保留在 VPS 上。

### 11.1 前端配置 (Cloudflare Pages)

1.  **构建命令**：`pnpm -C apps/cloud-ui build`
2.  **输出目录**：`apps/cloud-ui/dist`
3.  **环境变量**：在 CF Pages 控制面板中设置：
    *   生产环境：`VITE_API_BASE_URL = https://api.readio.top`
    *   预览环境：`VITE_API_BASE_URL = https://api-pre.readio.top`
4.  **路由兼容**：在 `apps/cloud-ui/public/` 下确保有 `_redirects` 文件，内容为：
    ```text
    /* /index.html 200
    ```

### 11.2 后端配置 (VPS)

1.  **域名设置**：为后端分配独立域名，如 `api.readio.top`。
2.  **CORS 允许列表**：在 GitHub Variables 中设置 `READIO_PROXY_ALLOWED_ORIGINS`，支持通配符：
    *   `READIO_PROXY_ALLOWED_ORIGINS = https://readio.top,https://*.readio.top`
3.  **ASR 允许列表**：同理设置 `READIO_ASR_ALLOWED_ORIGINS`。

### 11.3 运行逻辑

*   前端启动时会检查 `VITE_API_BASE_URL`。
*   如果存在且当前环境没有配置信息，它会自动请求 `GET ${API_BASE_URL}/api/v1/config` 获取运行时参数（如 ASR Token、Grafana 配置等）。
*   后端已针对 `*.readio.top` 开启跨域支持。
