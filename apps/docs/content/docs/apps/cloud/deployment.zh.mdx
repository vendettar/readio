# Cloud 部署指南

本指南描述 Readio Cloud 的标准 Docker 化部署流程。

## 1. 运行时配置 (Runtime Config)

系统采用后端驱动的异步配置机制：

- **抓取流程**：前端启动时请求 `GET /api/v1/config` 获取环境参数。
- **配置隔离**：敏感密钥留在后端 `.env`；仅非敏感参数（如 ASR Token、监控 URL）发往浏览器。

## 2. GitHub 配置清单

你需要在 GitHub 的 **Settings -> Environments** 中创建 `cloud-production`、`cloud-preproduction` 和 `cloud-ui-production` 三个环境。

### 2.1 基础密钥 (Secrets)
这些变量通常对于每个环境是必须且敏感的：

| 名称 | 说明 |
| :--- | :--- |
| `CLOUD_SSH_HOST` | 服务器公网 IP |
| `CLOUD_SSH_USER` | 登录用户名 (如 root) |
| `CLOUD_SSH_PRIVATE_KEY` | SSH 私钥 (需在 VPS 的 `authorized_keys` 中) |
| `CLOUD_SSH_PORT` | SSH 端口 (默认 22) |
| `READIO_ADMIN_TOKEN` | 管理后台 Bearer Token |
| `PODCAST_INDEX_API_KEY` | PodcastIndex API Key |
| `PODCAST_INDEX_API_SECRET` | PodcastIndex API Secret |
| `READIO_GRAFANA_OTLP_TOKEN` | Grafana OTLP 上报 Token |
| `READIO_ASR_RELAY_PUBLIC_TOKEN` | 浏览器 / 运行时读取的 ASR relay public token |
| `READIO_ASR_WORKER_SHARED_SECRET` | `cloud-api` 与 Cloudflare ASR Worker 之间的共享密钥 |
| `CLOUDFLARE_API_TOKEN` | Cloudflare Pages 部署 workflow 使用的 API Token |

### 2.2 环境变量 (Variables)
这些变量可以在 GitHub 环境设置的 "Variables" 选项卡中配置：

| 名称 | 说明 |
| :--- | :--- |
| `READIO_ASR_ALLOWED_ORIGINS` | ASR 允许的跨域来源 |
| `READIO_PROXY_ALLOWED_ORIGINS` | 代理允许的跨域来源 |
| `READIO_GRAFANA_OTLP_ENDPOINT` | Grafana OTLP 上报地址 |
| `READIO_TRUSTED_PROXY_CIDRS` | 信任的代理 IP (用于获取真实 IP) |
| `READIO_ASR_WORKER_BASE_URL` | Cloudflare ASR Worker 的基础 URL |
| `CLOUDFLARE_ACCOUNT_ID` | Cloudflare Pages 项目的 account ID |
| `CLOUDFLARE_PAGES_PROJECT_NAME` | `apps/cloud-ui` 对应的 Cloudflare Pages 项目名 |
| `CLOUD_UI_VITE_API_BASE_URL` | 注入到 `cloud-ui` 生产构建中的公开 API Base URL |

## 3. VPS 新机初始化 (Bootstrap Checklist)

对于一台全新的 Ubuntu/Debian VPS，请按照以下步骤快速初始化：

### 3.1 基础环境安装
执行以下命令安装 Docker 引擎：
```bash
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER
```

### 3.2 权限配置
如果你使用非 root 用户进行部署，请确保该用户拥有免密 `sudo` 权限：
```bash
# 执行 sudo visudo
# 添加一行: 用户名 ALL=(ALL) NOPASSWD:ALL
```

### 3.3 SSH 密钥准备
将你的公钥内容（通常是 `id_ed25519.pub`）添加到 VPS 的 `~/.ssh/authorized_keys` 文件中。

### 3.4 镜像拉取前提
VPS 必须能够拉取 `ghcr.io/${owner}/readio-cloud-api` 镜像。做法有两种：

- 将 GHCR 包设为 public
- 或者预先在服务器上执行 `docker login ghcr.io`

## 4. 持续集成与发布 (CI/CD)

- **前端 / Cloudflare Pages**：`deploy-cloud-ui-pages.yml` 会在 push 到 `main` 或手动触发时构建 `apps/cloud-ui`，并把 `apps/cloud-ui/dist` 发布到 Cloudflare Pages。
- **前端路由**：仓库已提交 `apps/cloud-ui/public/_redirects`，内容为 `/* /index.html 200`，因此节目页、单集详情页这类深链在 Pages 上不会 404。
- **测试 API 环境**：`deploy-cloud-preprod.yml` 在 push 到 `main` 时自动部署，也可以手动执行。
- **生产 API 环境**：`deploy-cloud-prod.yml` 需要在 Actions 页面手动点击 `Run workflow`，并且只有选中的 ref 是 `main` 时才会真正部署。
- **镜像清理**：每次部署后，VPS 会保留当前运行中容器正在使用的镜像，以及最近 5 个后端镜像 tag；更早的本地镜像会自动删除。
- **GHCR 清理**：后端镜像只使用部署对应的 `sha8` 作为 tag。每次部署后，GHCR 会保留最近 10 个后端镜像版本；更早的版本会自动删除。
- **目录隔离**：
  - preproduction 使用 `/etc/readio/pre`，compose project 名为 `readio-pre`
  - production 使用 `/etc/readio/prod`，compose project 名为 `readio-prod`
  - 两套环境可以在同一台 VPS 上并存
  - `cloud-ui` 的生产发布与 VPS 解耦，不走 Docker Compose
- **自动化流**：CI/CD 会自动在 VPS 上创建对应目录并完成配置同步。

## 5. 运维常用命令

- **查看日志**：`docker logs -f readio-api-prod`。
- **重启测试环境**：`cd /etc/readio/pre && docker compose -p readio-pre restart`
- **重启生产环境**：`cd /etc/readio/prod && docker compose -p readio-prod restart`
- **测试环境数据库**：`/etc/readio/pre/data/readio.db`
- **生产环境数据库**：`/etc/readio/prod/data/readio.db`
