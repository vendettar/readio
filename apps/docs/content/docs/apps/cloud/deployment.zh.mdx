# Cloud 部署指南

本文档与 `.github/workflows/cd-cloud.yml` 保持一致，仅记录部署契约。
它不负责 VPS、nginx、TLS、防火墙或 `systemd` 的搭建。

## 0a. 生产环境运行时配置契约

在生产环境 Cloud 中：

- `apps/cloud-api` 动态提供 `/env.js`，**优先于** SPA/静态资源回退。
- 生产环境 Cloud **不使用** `apps/cloud-ui/public/env.js` 或 `dist/env.js` 作为运行时配置基准。
- 后端仅通过显式浏览器安全 allowlist 发出运行时配置。
- 服务端所有的运行时值（监听端口、路径、滥用控制设置）保留在 VPS 环境或 systemd 环境文件（`/etc/readio/readio-cloud.env`）中。

在前端独立开发时：

- `apps/cloud-ui/public/env.js` 仍是开发用默认模板。
- `apps/cloud-ui/public/env.local.js` 仍是本地覆盖路径。
- 前端在两种环境下都读取 `window.__READIO_ENV__`。

浏览器公开的运行时配置字段不是密钥。如果字段被发出到 `/env.js`，假设每个浏览器用户都能读取它。

完整配置审计见 `handoff/runtime-config-ownership-audit.md`。

## 1. 工作流构建内容

- 使用 `pnpm -C apps/cloud-ui build` 在服务器外构建 Cloud UI。
- 使用 `apps/cloud-api` 中的 `CGO_ENABLED=0 GOOS=linux GOARCH=amd64 go build -o readio-cloud .` 构建 Linux 二进制。
- 只暂存发布产物：`readio-cloud` 二进制和 `apps/cloud-ui/dist`。
- 这里明确要求交叉编译，因为 `apps/cloud-api/package.json` 的本地脚手架脚本是 `go build -o /tmp/readio-cloud .`，不能满足 Linux CD 契约。
- Go 二进制读取的是 `PORT`；工作流中的 `CLOUD_PORT` 只是烟测端口，必须与服务的 `PORT` 值一致。

## 2. 必需的 GitHub 配置

使用受保护的 `cloud-production` GitHub Environment 存放部署 secrets，并可用于人工审批。

Secrets：

- `CLOUD_SSH_HOST`
- `CLOUD_SSH_PORT`
- `CLOUD_SSH_USER`
- `CLOUD_SSH_PRIVATE_KEY`
- `CLOUD_SSH_KNOWN_HOSTS`

Variables：

- `CLOUD_SERVICE_NAME`，默认 `readio-cloud`
- `CLOUD_PORT`，默认 `8080`

SSH 安全规则：

- 通过 `CLOUD_SSH_KNOWN_HOSTS` 提供主机身份材料。
- 保持 `StrictHostKeyChecking=yes`。
- 不要把 `StrictHostKeyChecking=no` 作为可接受的文档化路径。

## 2a. 远端前置条件

在工作流能够成功执行之前，目标 VPS 必须已经具备：

- 与 `CLOUD_SSH_USER` 一致的 Linux 用户
- 该用户可通过上传的私钥进行 SSH 登录
- VPS 上已安装 `rsync`
- VPS 上已安装 `curl`，用于健康检查
- 存在一个名称与 `CLOUD_SERVICE_NAME` 一致的 `systemd` 服务
- SSH 用户具备重启并检查该服务状态的权限
  - 要么服务直接由该 SSH 用户管理
  - 要么该 SSH 用户可非交互式执行所需的 `systemctl` 命令

本部署指南假设这些前置条件已经存在。如果其中任意一项缺失，工作流即使构建成功，也仍可能在发布或烟测阶段失败。

## 3. 远程目录布局

工作流在 VPS 上预期如下布局：

- `/opt/readio/shared/data/`：持久化 SQLite 数据
- `/opt/readio/releases/<git-sha>/`：发布产物
- `/opt/readio/current`：运行时指针

运行时契约：

- 服务应从 `current` 指向的发布目录执行二进制，或在 `systemd` 中使用 `WorkingDirectory=/opt/readio/current`
- Cloud 应用必须能从当前激活发布目录解析同目录下的 `dist/`
- SQLite 数据库路径必须保持在 `/opt/readio/shared/data/` 下，或其他位于发布目录之外的共享持久路径中
- 服务必须为监听端口导出 `PORT`；不要在这个契约里使用 `READIO_PORT`

每个发布目录包含：

- `readio-cloud`
- `dist/`

工作流会先创建远程目录再传输，且不会清空部署根目录，因此 SQLite 的 `data/` 会被保留。

## 3a. 新主机 VPS 启动

如果你在准备新的 VPS，先创建发布目录、安装 `systemd` 单元，并授予部署用户无需交互即可重启和检查 Cloud 服务的权限。

请仅使用当前运行时契约：

- `READIO_CLOUD_UI_DIST_DIR`
- `READIO_CLOUD_DB_PATH`

对于可变的运行时配置，优先使用 env file，而不是把所有值都硬编码进 `systemd` 单元：

- 像 `PORT`、`READIO_CLOUD_UI_DIST_DIR`、`READIO_CLOUD_DB_PATH` 这种稳定路径留在 unit 里。
- secrets 和可调运维参数放到 `/etc/readio/readio-cloud.env`。
- 这样可以保持 `systemd` unit 稳定，同时允许你只修改运行时配置而不反复重写服务定义。

示例启动脚本：

```bash
#!/usr/bin/env bash
set -euo pipefail

DEPLOY_USER="$(id -un)"
DEPLOY_GROUP="$(id -gn)"
SYSTEMCTL_BIN="$(command -v systemctl)"
SERVICE_NAME="readio-cloud"
REMOTE_ROOT="/opt/readio"
RELEASES_DIR="$REMOTE_ROOT/releases"
SHARED_DATA_DIR="$REMOTE_ROOT/shared/data"
CURRENT_DIR="$REMOTE_ROOT/current"
DB_PATH="$SHARED_DATA_DIR/readio.db"

sudo mkdir -p "$RELEASES_DIR"
sudo mkdir -p "$SHARED_DATA_DIR"
sudo mkdir -p /etc/readio
sudo chown -R "$DEPLOY_USER:$DEPLOY_GROUP" "$REMOTE_ROOT"

sudo tee /etc/readio/readio-cloud.env >/dev/null <<EOF
READIO_ASR_ALLOWED_ORIGINS=https://www.readio.top
READIO_ASR_RELAY_TOKEN=replace-with-a-long-random-token
READIO_ASR_RATE_LIMIT_BURST=60
READIO_ASR_RATE_LIMIT_WINDOW_MS=60000
EOF
sudo chmod 600 /etc/readio/readio-cloud.env

sudo tee "/etc/systemd/system/${SERVICE_NAME}.service" >/dev/null <<EOF
[Unit]
Description=Readio Cloud Backend
After=network.target

[Service]
Type=simple
User=$DEPLOY_USER
WorkingDirectory=$CURRENT_DIR
ExecStart=$CURRENT_DIR/readio-cloud
Restart=always
EnvironmentFile=/etc/readio/readio-cloud.env

Environment=PORT=8080
Environment=READIO_CLOUD_UI_DIST_DIR=$CURRENT_DIR/dist
Environment=READIO_CLOUD_DB_PATH=$DB_PATH

[Install]
WantedBy=multi-user.target
EOF

sudo tee "/etc/sudoers.d/readio-cloud-cd" >/dev/null <<EOF
$DEPLOY_USER ALL=(root) NOPASSWD: $SYSTEMCTL_BIN restart $SERVICE_NAME, $SYSTEMCTL_BIN is-active $SERVICE_NAME
EOF
sudo chmod 440 /etc/sudoers.d/readio-cloud-cd

sudo "$SYSTEMCTL_BIN" daemon-reload
sudo "$SYSTEMCTL_BIN" enable "$SERVICE_NAME"
```

重要细节：

- 服务使用 `WorkingDirectory=/opt/readio/current`。
- 二进制路径是 `/opt/readio/current/readio-cloud`。
- Cloud UI 静态资源从 `/opt/readio/current/dist` 提供。
- SQLite 文件应保留在发布目录之外，例如 `/opt/readio/shared/data/readio.db`。
- `/etc/readio/readio-cloud.env` 是推荐的 Cloud 可变运行时配置位置。
- Cloud ASR relay 的防滥用边界依赖 `READIO_ASR_ALLOWED_ORIGINS`、`READIO_ASR_RELAY_TOKEN`、`READIO_ASR_RATE_LIMIT_BURST`、`READIO_ASR_RATE_LIMIT_WINDOW_MS`。
- 浏览器需要发送 `X-Readio-Relay-Token`，所以同一个 relay token 需要同时存在于后端运行环境和 Cloud 前端运行时环境中。
- 工作流已经使用 `sudo -n`，因此部署用户必须能够无密码执行 `systemctl restart` 和 `systemctl is-active`。
- 工作流会在重启服务前把 `/opt/readio/current` 指向新发布目录，所以新主机第一次重启前必须已经存在对应发布目录。

如果 VPS 的 sudoers 规则只允许 `systemctl restart readio-cloud`，却不允许 `systemctl is-active readio-cloud`，那么 GitHub Actions 里的非交互烟测步骤仍会失败。

如果修改了 unit 或 env file，记得执行：

```bash
sudo systemctl daemon-reload
sudo systemctl restart readio-cloud
```

建议在 VPS 上做的预启动验证：

```bash
sudo -n "$(command -v systemctl)" restart readio-cloud
sudo -n "$(command -v systemctl)" is-active --quiet readio-cloud
curl -fsS http://127.0.0.1:8080/healthz
```

## 4. 发布流程

部署采用按版本号分段的发布方式，并且仅在指针层面保持原子性：

1. 在 GitHub runner 上构建 Cloud UI 和 Cloud API。
2. 将产物先暂存到本地 `release/`。
3. 为当前提交 SHA 创建远程发布目录。
4. 只把 `release/` 传到 `/opt/readio/releases/<git-sha>/`。
5. 使用 `ln -sfn` 把 `/opt/readio/current` 切到新发布目录。
6. 重启 `CLOUD_SERVICE_NAME` 指定的服务，默认值为 `readio-cloud`。
7. 校验服务处于 active 状态。
8. 校验当前发布目录中的二进制和静态资源目录存在。
9. 使用 `CLOUD_PORT` 指定的端口做健康探测，默认值为 `8080`，路径为 `/healthz`。

这不是原地覆盖式部署。工作流不会先删除持久化数据，也不会直接往 live 路径里复制构建产物。

## 5. 重启与烟测

工作流使用 `CLOUD_SERVICE_NAME` 指定的服务名，并检查：

- `systemctl is-active --quiet <service>`
- 当前发布目录中存在 `readio-cloud`
- 当前发布目录中存在 `dist/`
- `curl -fsS http://127.0.0.1:${CLOUD_PORT}/healthz`

如果需要手动验证运行时状态，也应使用同样的检查。

## 6. 回滚

回滚是手动、按发布目录进行的：

1. 保留上一个成功发布的目录。
2. 用 `ln -sfn` 把 `/opt/readio/current` 指回上一个发布目录。
3. 再次重启服务。
4. 重新执行相同的 smoke 检查。

如果上一个发布目录不存在，就需要从其他已知正常的发布目录恢复。工作流不会自动重新生成回滚产物。

## 7. 运维边界

- 该工作流假设 VPS 已经存在。
- 它不负责安装或管理 `systemd`。
- 它不负责配置 nginx、TLS 或防火墙。
- 它不要求为已批准的媒体 fallback 路径配置单独的 CORS proxy。
- 它不会强制所有音频都走后端；在浏览器可用时仍优先使用浏览器直连媒体访问。
- 对已批准的媒体相关失败类，Go 后端只作为透传 fallback 代理，不缓存也不持久化媒体字节。
- 浏览器本地的下载、转录和设置仍然留在浏览器本地。
- 该工作流不会重新启用 Workbox/PWA；Cloud 当前保持 PWA 关闭。
- 它不部署 SQLite schema、同步、认证或多节点能力。
