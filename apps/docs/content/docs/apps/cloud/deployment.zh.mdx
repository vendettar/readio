# Cloud 部署指南

本指南描述 Readio Cloud 基于 Docker Context 的标准部署流程。

## 1. 运行时配置 (Runtime Config)

系统采用后端驱动的异步配置机制：

- **抓取流程**：前端启动时请求 `GET /api/v1/config` 获取环境参数。
- **配置隔离**：服务端密钥保存在 GitHub Actions secrets 和后端容器环境中；只有浏览器公开运行时值会通过 `/env.js` 暴露。

## 2. GitHub 配置清单

你需要在 GitHub 的 **Settings -> Environments** 中创建 `cloud-production` 和 `cloud-preproduction` 两个环境。

### 2.1 基础密钥 (Secrets)
这些变量通常对于每个环境是必须且敏感的：

| 名称 | 说明 |
| :--- | :--- |
| `CLOUD_SSH_HOST` | 服务器公网 IP |
| `CLOUD_SSH_USER` | 登录用户名 (如 root) |
| `CLOUD_SSH_PRIVATE_KEY` | SSH 私钥 (需在 VPS 的 `authorized_keys` 中) |
| `CLOUD_SSH_PORT` | SSH 端口 (默认 22) |
| `CLOUD_SSH_KNOWN_HOSTS` | Docker Context 严格主机校验使用的固定 SSH host key 记录 |
| `READIO_ADMIN_TOKEN` | 后端 `/admin/*` 端点的 Bearer Token。生产 Cloudflare Pages `/ops` 路由不使用这个 token。 |
| `PODCAST_INDEX_API_KEY` | PodcastIndex API Key |
| `PODCAST_INDEX_API_SECRET` | PodcastIndex API Secret |
| `READIO_GRAFANA_OTLP_TOKEN` | Grafana OTLP 上报 Token（instance/token fallback 认证；优先级低于 `READIO_GRAFANA_OTLP_HEADERS`） |
| `READIO_GRAFANA_OTLP_HEADERS` | 可选。首选的 header-based OTLP 认证，直接从 Grafana Cloud OpenTelemetry tile 复制（例如 `Authorization=Basic <base64>`）。设置后会对 metrics 和 traces 同时覆盖 `READIO_GRAFANA_OTLP_INSTANCE_ID` + `READIO_GRAFANA_OTLP_TOKEN`。禁止出现在 `/env.js`、浏览器白名单、日志或文档示例中（真实值）。 |
| `READIO_GRAFANA_LOKI_TOKEN` | 可选 Grafana Cloud Logs Token。只有同时配置它、`READIO_GRAFANA_LOKI_URL` 和 `READIO_GRAFANA_LOKI_USER` 时，Loki 日志投递才会启用。 |
| `READIO_ASR_RELAY_PUBLIC_TOKEN` | 浏览器 / 运行时读取的 ASR relay public token |
| `READIO_ASR_WORKER_SHARED_SECRET` | `cloud-api` 与 Cloudflare ASR Worker 之间的共享密钥 |

### 2.2 环境变量 (Variables)
这些变量可以在 GitHub 环境设置的 "Variables" 选项卡中配置：

| 名称 | 说明 |
| :--- | :--- |
| `READIO_ASR_ALLOWED_ORIGINS` | ASR 允许的跨域来源 |
| `READIO_DISCOVERY_ALLOWED_ORIGINS` | Discovery 允许的跨域来源 |
| `READIO_PROXY_ALLOWED_ORIGINS` | 代理允许的跨域来源 |
| `READIO_GRAFANA_OTLP_ENDPOINT` | Grafana OTLP 上报地址 |
| `READIO_GRAFANA_OTLP_INSTANCE_ID` | 与上报 token 配套的 Grafana OTLP instance ID |
| `READIO_DEPLOY_ENV` | 后端部署环境，用于 host metrics 和 Loki labels；workflow 设置为 `preproduction` 或 `production` |
| `READIO_GRAFANA_LOKI_URL` | 可选 Grafana Cloud Loki push endpoint，通常以 `/loki/api/v1/push` 结尾 |
| `READIO_GRAFANA_LOKI_USER` | 可选 Grafana Cloud Logs user / instance ID |
| `READIO_LOKI_LOG_LEVEL` | 可选后端 Loki 最低日志级别：`debug`、`info`、`warn` 或 `error`；默认 `info` |
| `READIO_LOKI_BATCH_SIZE` | 可选后端 Loki batch size |
| `READIO_LOKI_FLUSH_INTERVAL_SECONDS` | 可选后端 Loki flush interval |
| `READIO_LOKI_QUEUE_SIZE` | 可选后端有界 Loki queue size |
| `READIO_TRUSTED_PROXY_CIDRS` | 信任的代理 IP (用于获取真实 IP) |
| `READIO_ASR_WORKER_BASE_URL` | Cloudflare ASR Worker 的基础 URL |
| `READIO_EN_DICTIONARY_API_URL` | 通过 `/env.js` 暴露给浏览器的英文词典 API 地址 |
| `READIO_EN_DICTIONARY_API_TRANSPORT` | 通过 `/env.js` 暴露给浏览器的英文词典传输契约 |
| `READIO_TRACING_ENABLED` | 可选。设为 `true` 时通过 OTLP 向 Grafana Cloud Tempo 推送分布式追踪数据。默认禁用。需配置 OTLP endpoint 和认证（`READIO_GRAFANA_OTLP_HEADERS` 或 instance/token）。 |
| `READIO_TRACING_SAMPLE_RATIO` | 可选。采样率，取值 `0.0`–`1.0`；默认 `0.1`（10%）。预发初始验证时使用 `1.0`，部署生产前降低采样率。 |

## 3. Cloudflare Pages 前端配置

前端 UI (`cloud-ui`) 已交由 Cloudflare Pages 原生托管并自动构建。因此，所有的前端环境变量都必须直接在 Cloudflare Pages 控制台对应项目（`readio-pre` 和 `readio-prod`）的 **Settings -> Environment variables** 中配置。

| 名称 | 说明 |
| :--- | :--- |
| `VITE_API_BASE_URL` | 注入到 `cloud-ui` 生产构建中的公开 API 地址。预发环境设为 `https://api-pre.readio.top`，生产环境设为 `https://api.readio.top`。 |
| `VITE_GRAFANA_FARO_URL` | 浏览器公开的 Grafana Faro 收集器 URL。 |
| `VITE_GRAFANA_FARO_APP_NAME` | 浏览器公开的 Faro 应用名。 |
| `VITE_GRAFANA_FARO_ENV` | 浏览器公开的 Faro 环境名（如 `preproduction` 或 `production`）。 |
| `VITE_GRAFANA_FARO_SAMPLE_RATE` | 浏览器公开的 Faro 采样率。 |

## 4. VPS 新机初始化 (Bootstrap Checklist)

对于一台全新的 Ubuntu VPS，请使用 `infra/ansible/vps_optimize.yml` 完成初始化。

初始化后必须满足：

- Docker Engine 和 Docker Compose V2 已安装。
- Nginx 已安装并按 `scripts/nginx-readio-cloud.conf` 配置 HTTPS 反向代理。
- 部署用户、SSH authorized keys、有限 sudo 权限由 Ansible 变量管理，不在 workflow 内硬编码。
- `/etc/readio/pre/data` 和 `/etc/readio/prod/data` 已存在，并且所有权兼容部署用户和 Docker runtime。
- 生产端口 `8080` 与测试端口 `8079` 未被其他服务占用。

### 3.1 镜像拉取前提
VPS 必须能够拉取 `ghcr.io/${owner}/readio-cloud-api-pre` 和 `ghcr.io/${owner}/readio-cloud-api-prod` 镜像。做法有两种：

- 将 GHCR 包设为 public
- 或者预先在服务器上执行 `docker login ghcr.io`

## 5. 持续集成与发布 (CI/CD)

项目采用严格的“分支对应环境”部署模式。前端部署由分支更新自动触发；后端部署只在对应分支的 backend/deploy 相关路径变更时自动触发。

### 预发环境 (`main` 分支)
`main` 分支作为集成主干，自动部署至**预发环境**（Pre-production）。
- **触发**：每当有代码提交或 PR 合并进 `main` 时触发。
- **前端 (`readio-pre`)**：Cloudflare Pages 自动监听 `main` 分支进行构建。
- **后端 API**：GitHub Action `deploy-cloud-preprod.yml` 自动运行，部署至 VPS 的 `preproduction` 容器。
- **目的**：用于开发者自测、全链路观测验证（Tracing/Logging）。

### 生产环境 (`release` 分支与 Tag)
生产环境采用严格的 **“Tag 触发”** 模式，确保每一份上线代码都有版本标记。
- **触发**：仅在推送以 `v` 开头的 Git Tag（如 `v1.2.3`）时触发。**直接合并 PR 到 `release` 不会触发部署。**
- **前端 (`readio-prod`)**：Cloudflare Pages 连接到 `release` 分支，随分支更新自动发布。
- **后端 API**：GitHub Action `deploy-cloud-prod.yml` 监听 Tag 事件，构建镜像并部署至 VPS 的 `production` 容器。

### 触发场景对照表

| 触发方式 | Branch Guard | 构建镜像 | VERSION | 备注 |
| :--- | :--- | :--- | :--- | :--- |
| `git push tag v1.2.3`（在 `release` 分支）| ✅ 通过 | ✅ 构建 | `v1.2.3` | 正常发布 |
| `git push tag v1.2.3`（在 feature 分支）| ❌ 拦截报错 | — | — | |
| `workflow_dispatch` 从 `release`，填 `image_tag: v1.2.1` | ✅ 通过 | ⏭ 跳过 | `v1.2.1` | 回滚 |
| `workflow_dispatch` 从 `release`，`image_tag` 留空 | ✅ 通过 | ✅ 构建 | 短 SHA | 紧急热修复，输出 warning，事后须补打 tag |
| `workflow_dispatch` 从其他分支 | ❌ 拦截报错 | — | — | |
| 预发：PR 合并到 `main` | ✅ 通过 | ✅ 构建 | 短 SHA | |
| 预发：`workflow_dispatch` 从 `main` | ✅ 通过 | ✅ 构建 | 短 SHA | |
| 预发：`workflow_dispatch` 从 feature 分支 | ❌ 直接跳过 | — | — | |

### 生产发布 SOP (Standard Operating Procedure)

为了保证生产环境的绝对稳定，发布必须严格遵守以下步骤：

#### 1. 正常发布 (Regular Release)
1.  **验证**：确保功能已合并至 `main` 并在预发环境（Preprod）通过验证。
2.  **晋升**：提交并合并 **PR (main -> release)**。
3.  **打标**：进入 GitHub **Releases** 页面，点击 **Draft a new release**。
4.  **配置**：
    -   **Target**: 选择 `release` 分支。
    -   **Tag**: 输入新版本号（如 `v1.2.0`）。
5.  **发布**：点击 **Publish**。Action 将自动构建镜像并部署。

#### 2. 紧急回滚 (Emergency Rollback)
1.  **进入 Actions**：在 GitHub 仓库点击 **Actions** 标签页。
2.  **选择任务**：在左侧列表中点击 **Deploy Cloud (Production)**。
3.  **触发**：点击 **Run workflow**，选择 `release` 分支。
4.  **填写版本**：在 **`image_tag`** 字段输入上一个稳定版本号（如 `v1.2.1`）。构建步骤会被跳过，直接拉取已有镜像部署。
5.  **确认**：在 `confirm` 输入框填入 `deploy`，点击 **Run**。

#### 3. 紧急热修复部署 (Emergency Hotfix Deploy / Break-glass)

仅在修复必须立即上线、来不及走正常 tag 发布流程时使用。

1. **进入 Actions**：在 GitHub Actions 页面选择 **Deploy Cloud (Production)**。
2. **触发**：点击 **Run workflow**，选择 `release` 分支。
3. **`image_tag` 留空**：从当前 `release` HEAD 构建短 SHA 镜像，run log 会输出 warning 提示。
4. **确认**：在 `confirm` 字段填入 `deploy`，点击 **Run**。
5. **事后必须补打 tag**：修复验证通过后，立即创建正式版本号（如 `v1.2.1`），恢复正常发布线。

#### 5. 紧急修复 (Hotfix)
1.  **切分支**：基于 `release` 分支创建 `hotfix/*` 分支。
2.  **修复**：在 hotfix 分支上进行修复并提交。
3.  **合并**：将 hotfix 分支合并回 `release`。
4.  **发布**：重复“正常发布”中的打标步骤，发布补丁版本（如 `v1.2.1`）。
5.  **回流**：**必须**在本地切回 `main` 分支，使用 `git cherry-pick [Hotfix_SHA]` 将修复同步回主干。
为了保持代码库的整洁和可维护性，必须遵守以下流程：

1. **线性历史 (Linear History)**：
   - 禁止在 `main` 或 `release` 分支上使用 `git merge`（产生合并节点）。
   - 所有的功能开发应在 `feature/*` 分支进行，通过 **Rebase** 保持基线领先，并使用 GitHub 的 **Squash and merge** 或 **Rebase and merge** 合入主干。

2. **晋升流程 (Environment Promotion)**：
   - 代码在 `main` 环境验证无误后，提交 **PR (main -> release)**。
   - 合并该 PR，此时 `release` 代码与 `main` 同步。

3. **打标发布 (Tagging)**：
   - 在 GitHub UI 的 **Releases** 页面，基于 `release` 分支创建一个新的 Release。
   - 填写版本号（如 `v1.1.0`），点击 Publish。这会触发生产环境的正式部署。

4. **紧急修复 (Hotfix & Back-porting)**：
   - 若生产环境发现 Bug，应基于 `release` 分支创建 hotfix 分支。
   - 修复并合并回 `release` 后，立即通过 **`git cherry-pick`** 将该提交同步回 `main` 分支，确保主干不丢失修复。

### 部署细节
- **前端路由**：Cloudflare Pages 将 Cloud UI 作为 SPA 托管。只要构建产物没有根级 `404.html`，节目页、单集详情页这类深链会回退到 `index.html`，再由客户端路由解析。
- **仓库部署命令**：根目录 `Makefile` 负责 API 部署和运维的 Docker Context 命令入口。
- **远程执行**：部署使用 Docker Context over SSH。workflow 使用 `CLOUD_SSH_PRIVATE_KEY` 和固定的 `CLOUD_SSH_KNOWN_HOSTS` 非交互配置 SSH，然后执行 `make context-setup` 与对应部署目标。
- **运行时环境**：CI 不再在 VPS 上写入或修改明文 env 文件。`docker-compose.readio.yml` 显式列出必需环境变量，由 Compose 从 workflow 或本地 shell 环境中插值。
- **密钥边界**：容器环境变量仍处在 Docker 管理员边界内；它们不是 memory-only，Docker 管理员或 Docker daemon metadata 可能看到这些值。
- **镜像清理**：每次部署后，VPS 会保留当前运行中容器正在使用的镜像，以及最近 10 个后端镜像 tag；更早的本地镜像会自动删除。
- **镜像 tag 规则**：正常生产发布使用 Git 版本号作为 tag（如 `readio-cloud-api-prod:v1.2.3`）；预发镜像使用短 commit SHA（如 `readio-cloud-api-pre:a1b2c3d4`）。紧急热修复从 `release` 分支 `workflow_dispatch` 触发时可能构建短 SHA 镜像，事后必须立即打版本号 tag 恢复正常发布线。
- **自动化冒烟测试**：每个部署 job 都会执行自动化的冒烟测试。它通过探测 `GET /api/v1/config` 并验证其返回状态码和响应时间来确保新版本已成功启动并可对外服务。如果探测失败，部署将被标记为失败，并跳过后续的镜像清理步骤，以保留回滚现场。
- **GHCR 清理**：测试和生产分别使用独立的 GHCR package：`readio-cloud-api-pre` 与 `readio-cloud-api-prod`。每个 package 都会独立保留最近 10 个后端镜像版本。
- **目录隔离**：
  - preproduction 使用 `/etc/readio/pre`，compose project 名为 `readio-pre`
  - production 使用 `/etc/readio/prod`，compose project 名为 `readio-prod`
  - 两套环境可以在同一台 VPS 上并存
  - `cloud-ui` 的生产发布与 VPS 解耦，不走 Docker Compose
- **可观测性**：
  - 此部署不安装 Grafana Alloy、node_exporter、本地 Prometheus 或 Loki agent。
  - 配置 `READIO_GRAFANA_OTLP_ENDPOINT` 和 OTLP 认证（`READIO_GRAFANA_OTLP_HEADERS` 或 `READIO_GRAFANA_OTLP_INSTANCE_ID` + `READIO_GRAFANA_OTLP_TOKEN`）后，`apps/cloud-api` 会通过 OTLP 向 Grafana Cloud 推送应用指标和主机关键指标。需要 `metrics:write` 权限。
  - 设置 `READIO_TRACING_ENABLED=true` 并配置 OTLP endpoint 和认证后，`apps/cloud-api` 可通过 OTLP 直接向 Grafana Cloud Tempo 推送分布式追踪数据。metrics 和 traces 共用同一个 OTLP endpoint 和认证解析器，推荐使用 `READIO_GRAFANA_OTLP_HEADERS`。需要 `traces:write` 权限。默认禁用。
  - 同时配置 `READIO_GRAFANA_LOKI_URL`、`READIO_GRAFANA_LOKI_USER` 和 `READIO_GRAFANA_LOKI_TOKEN` 后，`apps/cloud-api` 可以直接把脱敏后的应用日志推送到 Grafana Cloud Logs/Loki。这是应用侧 push 路径，不是 Alloy、Promtail、Docker log driver 或主机日志 agent。需要 `logs:write` 权限。
  - metrics、traces 和 Loki 日志是三条独立 pipeline：OTLP metrics 和 OTLP traces 共用同一 endpoint 和认证解析器，Loki 日志使用独立的 Loki URL/user/token。
  - Loki `env` label 优先来自后端 `READIO_DEPLOY_ENV`，未设置时回退到 `VITE_GRAFANA_FARO_ENV`。取值会规范化为 `production`、`preproduction` 或 `unknown`。
  - `/admin/logs` 仍是 same-origin、memory-only 的运维日志界面。启用 Loki 后，Loki 是持久化远端应用日志界面。
  - 主机日志不会投递到 Grafana；Docker logs 仍可通过 `make logs-pre`、`make logs-prod` 或直接 Docker 操作查看。
  - 主机指标是 `apps/cloud-api` 内部采集的 best-effort 信号，不是 node_exporter 的完整替代。
  - 主机指标采集默认 `READIO_HOST_METRICS_INTERVAL_SECONDS=60`。
  - 主机指标名称包括 `readio_cloud_process_goroutines`、`readio_cloud_process_heap_alloc_bytes`、`readio_cloud_process_memory_sys_bytes`、`readio_cloud_process_uptime_seconds`、`readio_cloud_sqlite_db_size_bytes`、`readio_cloud_sqlite_wal_size_bytes`、`readio_cloud_data_filesystem_free_bytes`、`readio_cloud_data_filesystem_total_bytes` 和 `readio_cloud_transcript_assets_size_bytes`。
  - Loki 投递使用有界队列，并且不会阻塞请求路径。队列满或 Loki 不可用时会本地丢弃日志，请求继续执行；服务关闭时会进行 best-effort flush。
  - Loki labels 只允许 `service`、`env` 和 `level`；高基数字段不得进入 labels。
  - Loki 日志行在请求上下文存在活跃 span 时会包含 `trace_id` 和 `span_id` payload 字段（不是 stream label），用于在 Grafana 中从 Loki 跳转到 Tempo。若 Grafana 未自动建立 `trace_id` → Tempo 链接，需在 Loki 数据源中手动配置 derived field：字段名 `trace_id`、正则 `"trace_id":"(\w+)"`、指向 Tempo 数据源的内部链接。
  - 前端 Grafana Faro RUM（`VITE_GRAFANA_FARO_*`）是独立 pipeline，与后端 Tempo 追踪互不关联。
  - `trace_id` 不会作为 Loki stream label，也不会出现在公开或面向用户的 API 响应中。

## 6. 运维常用命令

- **验证测试环境配置**：`make check-env-pre`

- **查看生产环境日志**：`make logs-prod`
- **清理旧测试环境 API 镜像**：`make prune-images-pre`
- **清理旧生产环境 API 镜像**：`make prune-images-prod`
- **更新配置**：更新 GitHub Secrets/Variables 或本地环境变量后，重新执行对应部署命令。
- **测试环境数据库**：`/etc/readio/pre/data/readio.db`
- **生产环境数据库**：`/etc/readio/prod/data/readio.db`

## 7. 回滚 (Rollback)

得益于 Tag 机制，回滚快速且安全——无需重新构建镜像。

1. **镜像 Tag 回滚（推荐）**：
   - 进入 GitHub Actions 页面，选择 **Deploy Cloud (Production)**。
   - 点击 **Run workflow**，选择 `release` 分支。
   - 在 **`image_tag`** 字段输入上一个稳定版本号（如 `v1.0.9`）。
   - 在 `confirm` 字段填入 `deploy`，点击 **Run**。
   - 构建步骤会被完全跳过，直接拉取已有镜像部署，速度更快。

2. **代码回滚 (Revert PR)**：
   - 如果需要永久回退代码，在 GitHub PR 页面点击 **Revert**，这会产生一个反向修改的 PR。
