# Cloud Deployment Guide

This guide describes the standard Docker Context based deployment process for Readio Cloud.

## 1. Runtime Config

The system uses a backend-driven asynchronous configuration mechanism:

- **Flow**: The frontend fetches environment parameters via `GET /api/v1/config` at startup.
- **Isolation**: Server-owned secrets stay in GitHub Actions secrets and the backend container environment. Browser-public runtime values are the only values exposed through `/env.js`.
- **Adaptation**:
    - **Same-origin**: Fetches directly from `/api/v1/config`.
    - **Decoupled**: Uses the `VITE_API_BASE_URL` environment variable to target the backend.

## 2. GitHub Configuration

Create three environments in GitHub **Settings -> Environments**: `cloud-production`, `cloud-preproduction`, and `cloud-ui-production`.

### 2.1 Secrets

| Name | Description |
| :--- | :--- |
| `CLOUD_SSH_HOST` | Server public IP or DNS name |
| `CLOUD_SSH_PORT` | SSH port, usually `22` |
| `CLOUD_SSH_USER` | Login username used by the deploy workflow |
| `CLOUD_SSH_PRIVATE_KEY` | SSH private key for that user |
| `CLOUD_SSH_KNOWN_HOSTS` | Pinned SSH host key entry used by Docker Context with strict host checking |
| `READIO_ADMIN_TOKEN` | Bearer Token for Admin UI |
| `PODCAST_INDEX_API_KEY` | PodcastIndex API Key |
| `PODCAST_INDEX_API_SECRET` | PodcastIndex API Secret |
| `READIO_GRAFANA_OTLP_TOKEN` | Grafana OTLP Push Token |
| `READIO_ASR_RELAY_PUBLIC_TOKEN` | Public relay token consumed by browser/runtime config |
| `READIO_ASR_WORKER_SHARED_SECRET` | Shared secret between `cloud-api` and the ASR Worker |
| `CLOUDFLARE_API_TOKEN` | API token used by the Cloudflare Pages deploy workflow |

## 3. VPS Prerequisites

1. **Dependencies**: Provision Docker Engine, Docker Compose V2, Nginx, the deploy user, and data directories with `infra/ansible/vps_optimize.yml`.
2. **Directories**: The VPS must already contain `/etc/readio/pre/data` and `/etc/readio/prod/data` with deploy-user compatible ownership.
3. **Network**: Configure Nginx reverse-proxy with HTTPS (refer to `scripts/nginx-readio-cloud.conf`).
4. **Registry Access**: The VPS must be able to pull `ghcr.io/${owner}/readio-cloud-api-pre` and `ghcr.io/${owner}/readio-cloud-api-prod`. Keep both packages public, or pre-configure `docker login ghcr.io` on the server.
5. **Cleanup**: Ensure ports 8080 (Prod) / 8079 (Pre) are not occupied.

## 4. CI/CD & Release

- **Frontend / Cloudflare Pages**: `deploy-cloud-ui-pages.yml` builds `apps/cloud-ui` and deploys `apps/cloud-ui/dist` to Cloudflare Pages on push to `main`, or by manual dispatch.
- **Frontend routing**: `apps/cloud-ui/public/_redirects` is committed with `/* /index.html 200`, so direct deep links such as podcast and episode detail routes resolve correctly on Pages.
- **Pre-production API**: `deploy-cloud-preprod.yml` is triggered automatically on push to `main`, and may also be run manually.
- **Production API**: `deploy-cloud-prod.yml` is triggered manually via the GitHub Actions `Run workflow` button, and the workflow only deploys when the selected ref is `main`.
- **Repo-owned deploy commands**: the root `Makefile` owns the Docker Context command surface for API deploys and operations.
- **Remote execution**: deployment uses Docker Context over SSH. The workflows configure SSH non-interactively with `CLOUD_SSH_PRIVATE_KEY` and pinned `CLOUD_SSH_KNOWN_HOSTS`, then run `make context-setup` and the relevant deploy target.
- **Runtime environment**: CI no longer writes or mutates plaintext env files on the VPS. `docker-compose.readio.yml` lists required environment variables explicitly and Compose interpolates them from the workflow or local shell environment.
- **Secret boundary**: container environment values remain inside the Docker administrator boundary; they are not memory-only and can be visible to Docker administrators or Docker daemon metadata.
- **Image Management**: After each deploy, the VPS keeps the images currently used by running containers plus the latest 5 backend image tags; older local images are removed automatically.
- **GHCR Retention**: Backend images are tagged only with the deployment `sha8`. Preproduction and production use separate GHCR packages: `readio-cloud-api-pre` and `readio-cloud-api-prod`. Each package keeps its latest 10 backend image versions independently.
- **Deployment isolation**:
  - preproduction runs under `/etc/readio/pre` with compose project `readio-pre`
  - production runs under `/etc/readio/prod` with compose project `readio-prod`
  - the two environments can coexist on the same VPS without replacing each other
  - `cloud-ui` production deploy is isolated from the VPS workflows and does not use Docker Compose
- **Observability**:
  - Grafana Alloy, node_exporter, local Prometheus, and Loki agents are not installed by this deployment.
  - `apps/cloud-api` pushes application and app-owned host-critical metrics to Grafana Cloud through OTLP when `READIO_GRAFANA_OTLP_ENDPOINT`, `READIO_GRAFANA_OTLP_INSTANCE_ID`, and `READIO_GRAFANA_OTLP_TOKEN` are configured.
  - Host logs are not shipped to Grafana. `/admin/logs` and Docker logs remain the log surfaces.
  - Host metrics are best-effort app-owned signals collected inside `apps/cloud-api`, not a replacement for node_exporter.
  - Host metric collection defaults to `READIO_HOST_METRICS_INTERVAL_SECONDS=60`.
  - Host metric names include `readio_cloud_process_goroutines`, `readio_cloud_process_heap_alloc_bytes`, `readio_cloud_process_memory_sys_bytes`, `readio_cloud_process_uptime_seconds`, `readio_cloud_sqlite_db_size_bytes`, `readio_cloud_sqlite_wal_size_bytes`, `readio_cloud_data_filesystem_free_bytes`, `readio_cloud_data_filesystem_total_bytes`, and `readio_cloud_transcript_assets_size_bytes`.
  - `/admin/logs` remains a same-origin, memory-only operator log surface.
  - Docker logs remain available through `make logs-pre`, `make logs-prod`, or direct Docker operations.

## 5. Operations

- **Validate preproduction config**: `make check-env-pre`
- **Validate production config**: `make check-env-prod`
- **Create/update Docker Context**: `make context-setup`
- **Dry-run Docker Context setup**: `make context-setup DRY_RUN=1`
- **Deploy preproduction**: `make deploy-pre`
- **Deploy production**: `make deploy-prod CONFIRM=deploy`
- **View preproduction logs**: `make logs-pre`
- **View production logs**: `make logs-prod`
- **Prune old preproduction API images**: `make prune-images-pre`
- **Prune old production API images**: `make prune-images-prod`
- **Update Config**: Update GitHub Secrets/Variables or local environment variables, then re-run the relevant deploy command.
- **Preproduction DB path**: `/etc/readio/pre/data/readio.db`
- **Production DB path**: `/etc/readio/prod/data/readio.db`
- **Cleanup**: Use `make prune-images-pre` and `make prune-images-prod` to remove old Readio API images without touching Docker volumes or unrelated workload images.

## 6. Rollback

1. Identify the previous GHCR image tag from GitHub Actions or GHCR package history.
2. Re-run the relevant Makefile deployment target with the previous `IMAGE_TAG`.
3. Never delete `/etc/readio/pre/data` or `/etc/readio/prod/data` during rollback.
4. Do not run destructive Docker volume cleanup as part of normal rollback.
