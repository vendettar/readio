# Cloud Deployment Guide

This guide describes the standard Docker Context based deployment process for Readio Cloud.

## 1. Runtime Config

The system uses a backend-driven asynchronous configuration mechanism:

- **Flow**: The frontend fetches environment parameters via `GET /api/v1/config` at startup.
- **Isolation**: Server-owned secrets stay in GitHub Actions secrets and the backend container environment. Browser-public runtime values are the only values exposed through `/env.js`.
- **Adaptation**:
    - **Same-origin**: Fetches directly from `/api/v1/config`.
    - **Decoupled**: Uses the `VITE_API_BASE_URL` environment variable to target the backend.

## 2. GitHub Configuration

Create two environments in GitHub **Settings -> Environments**: `cloud-production` and `cloud-preproduction`.

### 2.1 Secrets

| Name | Description |
| :--- | :--- |
| `CLOUD_SSH_HOST` | Server public IP or DNS name |
| `CLOUD_SSH_PORT` | SSH port, usually `22` |
| `CLOUD_SSH_USER` | Login username used by the deploy workflow |
| `CLOUD_SSH_PRIVATE_KEY` | SSH private key for that user |
| `CLOUD_SSH_KNOWN_HOSTS` | Pinned SSH host key entry used by Docker Context with strict host checking |
| `READIO_ADMIN_TOKEN` | Bearer token for backend `/admin/*` endpoints. The production Cloudflare Pages `/ops` route does not use this token. |
| `PODCAST_INDEX_API_KEY` | PodcastIndex API Key |
| `PODCAST_INDEX_API_SECRET` | PodcastIndex API Secret |
| `READIO_GRAFANA_OTLP_TOKEN` | Grafana OTLP Push Token (instance/token fallback auth; takes lower precedence than `READIO_GRAFANA_OTLP_HEADERS`) |
| `READIO_GRAFANA_OTLP_HEADERS` | Optional. Preferred header-based OTLP auth value copied verbatim from the Grafana Cloud OpenTelemetry tile (e.g. `Authorization=Basic <base64>`). When set, takes precedence over `READIO_GRAFANA_OTLP_INSTANCE_ID` + `READIO_GRAFANA_OTLP_TOKEN` for both metrics and traces. Must never appear in `/env.js`, browser allowlists, logs, or documentation examples with real values. |
| `READIO_GRAFANA_LOKI_TOKEN` | Optional Grafana Cloud Logs token. Loki log shipping stays disabled unless this token, `READIO_GRAFANA_LOKI_URL`, and `READIO_GRAFANA_LOKI_USER` are all configured. |
| `READIO_ASR_RELAY_PUBLIC_TOKEN` | Public relay token consumed by browser/runtime config |
| `READIO_ASR_WORKER_SHARED_SECRET` | Shared secret between `cloud-api` and the ASR Worker |

## 3. Cloudflare Pages Configuration

The frontend UI (`cloud-ui`) is deployed natively by Cloudflare Pages. Frontend environment variables must be configured directly in the Cloudflare Pages dashboard under **Settings -> Environment variables** for each respective project (`readio-pre` and `readio-prod`).

| Name | Description |
| :--- | :--- |
| `VITE_API_BASE_URL` | Public API base URL baked into the `cloud-ui` production build. Set to `https://api-pre.readio.top` for pre-production and `https://api.readio.top` for production. |
| `VITE_GRAFANA_FARO_URL` | Browser-public Grafana Faro collector URL. |
| `VITE_GRAFANA_FARO_APP_NAME` | Browser-public Faro application name. |
| `VITE_GRAFANA_FARO_ENV` | Browser-public Faro environment name (e.g., `preproduction` or `production`). |
| `VITE_GRAFANA_FARO_SAMPLE_RATE` | Browser-public Faro sample rate. |

## 4. VPS Prerequisites

1. **Dependencies**: Provision Docker Engine, Docker Compose V2, Nginx, the deploy user, and data directories with `infra/ansible/vps_optimize.yml`.
2. **Directories**: The VPS must already contain `/etc/readio/pre/data` and `/etc/readio/prod/data` with deploy-user compatible ownership.
3. **Network**: Configure Nginx reverse-proxy with HTTPS (refer to `scripts/nginx-readio-cloud.conf`).
4. **Registry Access**: The VPS must be able to pull `ghcr.io/${owner}/readio-cloud-api-pre` and `ghcr.io/${owner}/readio-cloud-api-prod`. Keep both packages public, or pre-configure `docker login ghcr.io` on the server.
5. **Cleanup**: Ensure ports 8080 (Prod) / 8079 (Pre) are not occupied.

## 5. CI/CD & Release Architecture

The project follows a strict branch-to-environment deployment model. Frontend deployments are automatically triggered by branch updates; backend deployments are automatically triggered when backend/deploy paths change on the matching branch.

### Pre-production (`main` branch)
The `main` branch is the integration branch and deploys to the **Pre-production** environment.
- **Trigger**: Automatically triggered by any push or PR merge into `main`.
- **Frontend (`readio-pre`)**: Cloudflare Pages monitors the `main` branch for automatic builds.
- **Backend API**: GitHub Action `deploy-cloud-preprod.yml` runs automatically, deploying to the VPS `preproduction` container.
- **Purpose**: Used for developer self-testing and full-stack observability validation (Tracing/Logging).

### Production (`release` branch & Tags)
Production environment follows a strict **"Tag-triggered"** model to ensure every release is versioned and immutable.
- **Trigger**: Only triggered when a Git Tag starting with `v` (e.g., `v1.2.3`) is pushed. **Merging a PR into `release` does NOT trigger a deployment.**
- **Frontend (`readio-prod`)**: Cloudflare Pages is connected to the `release` branch and deploys automatically upon branch updates.
- **Backend API**: GitHub Action `deploy-cloud-prod.yml` listens for Tag events to build the image and deploy to the VPS `production` container.

### Trigger Behavior Reference

| Trigger | Branch Guard | Build Image | `VERSION` | Notes |
| :--- | :--- | :--- | :--- | :--- |
| `git push tag v1.2.3` (on `release`) | ✅ Pass | ✅ Build | `v1.2.3` | Normal release |
| `git push tag v1.2.3` (on feature branch) | ❌ Blocked | — | — | |
| `workflow_dispatch` from `release`, `image_tag: v1.2.1` | ✅ Pass | ⏭ Skipped | `v1.2.1` | Rollback |
| `workflow_dispatch` from `release`, `image_tag` empty | ✅ Pass | ✅ Build | short SHA | Emergency hotfix — warning emitted; create a version tag after validation |
| `workflow_dispatch` from any other branch | ❌ Blocked | — | — | |
| Preprod: PR merged into `main` | ✅ Pass | ✅ Build | short SHA | |
| Preprod: `workflow_dispatch` from `main` | ✅ Pass | ✅ Build | short SHA | |
| Preprod: `workflow_dispatch` from feature branch | ❌ Blocked | — | — | |

### Production SOP (Standard Operating Procedure)

To maintain absolute production stability, all deployments must strictly follow these procedures:

#### 1. Regular Release
1.  **Validation**: Ensure features are merged into `main` and verified in the Pre-production environment.
2.  **Promotion**: Submit and merge a **PR (main -> release)**.
3.  **Tagging**: Go to the GitHub **Releases** page and click **Draft a new release**.
4.  **Configuration**:
    -   **Target**: Select the `release` branch.
    -   **Tag**: Enter the new version (e.g., `v1.2.0`).
5.  **Publish**: Click **Publish**. The Action will automatically build and deploy the image.

#### 2. Emergency Rollback
1.  **Actions**: Click the **Actions** tab in the GitHub repository.
2.  **Workflow**: Select **Deploy Cloud (Production)** from the left sidebar.
3.  **Trigger**: Click **Run workflow**, select the `release` branch.
4.  **Image Tag**: Enter the previous stable version (e.g. `v1.2.1`) in the `image_tag` field. The build step is skipped; the existing image is pulled directly.
5.  **Confirm**: Type `deploy` in the `confirm` input and click **Run**.

#### 3. Emergency Hotfix Deploy (Break-glass)

Use this path only when a critical fix must reach production immediately and there is no time to go through the normal tag release process.

1. **Actions**: Navigate to **Deploy Cloud (Production)** in the GitHub Actions tab.
2. **Trigger**: Click **Run workflow**, select the `release` branch.
3. **Leave `image_tag` empty**: builds from the current `release` HEAD as a short-SHA image. A warning is emitted in the run log.
4. **Confirm**: Type `deploy` in the `confirm` field and click **Run**.
5. **Follow up**: After the fix is validated, create a proper version tag (e.g. `v1.2.1`) immediately to restore the normal release line.

#### 4. Hotfix
1.  **Branching**: Create a `hotfix/*` branch based on `release`.
2.  **Fixing**: Commit the necessary fixes to the hotfix branch.
3.  **Merging**: Merge the hotfix branch back into `release`.
4.  **Release**: Repeat the tagging steps from "Regular Release" to deploy a patch version (e.g., `v1.2.1`).
5.  **Back-porting**: **You MUST** switch to the `main` branch locally and use `git cherry-pick [Hotfix_SHA]` to sync the fix back to the integration trunk.

### Professional Release Workflow
To maintain a clean and maintainable codebase, the following workflow is mandatory:

1. **Linear History**:
   - Avoid `git merge` on `main` or `release` branches (to prevent merge commits).
   - All feature development happens on `feature/*` branches. Keep branches up-to-date via **Rebase** and merge into main using GitHub's **Squash and merge** or **Rebase and merge**.

2. **Environment Promotion**:
   - Once code is validated in `main`, submit a **PR (main -> release)**.
   - Merge the PR to sync the `release` branch with `main`.

3. **Tag-based Release**:
   - Go to the **Releases** page in the GitHub UI and draft a new release targeting the `release` branch.
   - Enter a version tag (e.g., `v1.1.0`) and click **Publish**. This initiates the official production deployment.

4. **Hotfixes & Back-porting**:
   - If a bug is found in production, create a hotfix branch based on `release`.
   - After fixing and merging back into `release`, immediately **`cherry-pick`** the fix into the `main` branch to ensure the fix is not lost in future updates.

### Deployment Details
- **Frontend routing**: `apps/cloud-ui/public/_redirects` is committed with `/* /index.html 200`, so direct deep links such as podcast and episode detail routes resolve correctly on Pages.
- **Repo-owned deploy commands**: the root `Makefile` owns the Docker Context command surface for API deploys and operations.
- **Remote execution**: deployment uses Docker Context over SSH. The workflows configure SSH non-interactively with `CLOUD_SSH_PRIVATE_KEY` and pinned `CLOUD_SSH_KNOWN_HOSTS`, then run `make context-setup` and the relevant deploy target.
- **Runtime environment**: CI no longer writes or mutates plaintext env files on the VPS. `docker-compose.readio.yml` lists required environment variables explicitly and Compose interpolates them from the workflow or local shell environment.
- **Secret boundary**: container environment values remain inside the Docker administrator boundary; they are not memory-only and can be visible to Docker administrators or Docker daemon metadata.
- **Image Management**: After each deploy, the VPS keeps the images currently used by running containers plus the latest 10 backend image tags; older local images are removed automatically.
- **Image Tagging**: Normal production releases use Git version tags (e.g. `readio-cloud-api-prod:v1.2.3`); preproduction images are tagged with the short commit SHA (e.g. `readio-cloud-api-pre:a1b2c3d4`). Emergency manual deploys from `release` may build a short-SHA image and must be followed by a version tag release to restore the normal release line.
- **Automated Smoke Testing**: Every deployment job executes an automated smoke test. It verifies that the new version has successfully started and is serving traffic by probing `GET /api/v1/config` and validating its return status and response time. If probing fails, the deployment is marked as failed, and the subsequent image cleanup step is skipped to preserve the rollback state.
- **GHCR Retention**: Preproduction and production use separate GHCR packages: `readio-cloud-api-pre` and `readio-cloud-api-prod`. Each package keeps its latest 10 backend image versions independently.
- **Deployment isolation**:
  - preproduction runs under `/etc/readio/pre` with compose project `readio-pre`
  - production runs under `/etc/readio/prod` with compose project `readio-prod`
  - the two environments can coexist on the same VPS without replacing each other
  - `cloud-ui` production deploy is isolated from the VPS workflows and does not use Docker Compose
- **Observability**:
  - Grafana Alloy, node_exporter, local Prometheus, and Loki agents are not installed by this deployment.
  - `apps/cloud-api` pushes application and app-owned host-critical metrics to Grafana Cloud through OTLP when `READIO_GRAFANA_OTLP_ENDPOINT` and OTLP auth (`READIO_GRAFANA_OTLP_HEADERS` or `READIO_GRAFANA_OTLP_INSTANCE_ID` + `READIO_GRAFANA_OTLP_TOKEN`) are configured. Requires `metrics:write` scope.
  - `apps/cloud-api` can push distributed traces directly to Grafana Cloud Tempo via OTLP when `READIO_TRACING_ENABLED=true` and OTLP endpoint/auth are configured. The same OTLP endpoint and auth resolver is shared with metrics; `READIO_GRAFANA_OTLP_HEADERS` is the preferred auth path. Requires `traces:write` scope. Tracing is disabled by default.
  - `apps/cloud-api` can push sanitized application logs directly to Grafana Cloud Logs/Loki when `READIO_GRAFANA_LOKI_URL`, `READIO_GRAFANA_LOKI_USER`, and `READIO_GRAFANA_LOKI_TOKEN` are all configured. This is an application-side push path, not Alloy, Promtail, a Docker log driver, or a host log agent. Requires `logs:write` scope.
  - Metrics, traces, and Loki logs are three distinct pipelines: OTLP metrics and OTLP traces share the same endpoint and auth resolver, while Loki logs use the separate Loki URL/user/token.
  - Loki `env` labels come from backend `READIO_DEPLOY_ENV` first, then fall back to `VITE_GRAFANA_FARO_ENV` if unset. Values normalize to `production`, `preproduction`, or `unknown`.
  - `/admin/*` remains a same-origin, memory-backed operator surface on the backend. Loki is the durable remote application log surface when enabled.
  - Host logs are not shipped to Grafana. Docker logs remain available through `make logs-pre`, `make logs-prod`, or direct Docker operations.
  - Host metrics are best-effort app-owned signals collected inside `apps/cloud-api`, not a replacement for node_exporter.
  - Host metric collection defaults to `READIO_HOST_METRICS_INTERVAL_SECONDS=60`.
  - Host metric names include `readio_cloud_process_goroutines`, `readio_cloud_process_heap_alloc_bytes`, `readio_cloud_process_memory_sys_bytes`, `readio_cloud_process_uptime_seconds`, `readio_cloud_sqlite_db_size_bytes`, `readio_cloud_sqlite_wal_size_bytes`, `readio_cloud_data_filesystem_free_bytes`, `readio_cloud_data_filesystem_total_bytes`, and `readio_cloud_transcript_assets_size_bytes`.
  - Loki shipping is bounded and non-blocking for request paths. If the queue is full or Loki is unavailable, logs are dropped locally and requests continue; shutdown performs a best-effort flush.
  - Loki labels are limited to `service`, `env`, and `level`; high-cardinality values stay out of labels.
  - Loki log lines include `trace_id` and `span_id` payload fields (not stream labels) when the request context carries an active trace span. This enables Loki-to-Tempo navigation in Grafana. If Grafana does not auto-link `trace_id` to Tempo, configure a derived field on the Loki data source: field name `trace_id`, regex `"trace_id":"(\w+)"`, internal link pointing to the Tempo data source with value `${__value.raw}`.
  - Frontend Grafana Faro RUM is a separate pipeline (`VITE_GRAFANA_FARO_*`) and is not connected to backend Tempo tracing.
  - `trace_id` is never a Loki stream label and is never returned in public or user-facing API responses.

## 6. Operations

- **Validate preproduction config**: `make check-env-pre`
- **Validate production config**: `make check-env-prod`
- **Create/update Docker Context**: `make context-setup`
- **Dry-run Docker Context setup**: `make context-setup DRY_RUN=1`
- **Deploy preproduction**: `make deploy-pre`
- **Deploy production**: `make deploy-prod CONFIRM=deploy`
- **View preproduction logs**: `make logs-pre`
- **View production logs**: `make logs-prod`
- **Prune old preproduction API images**: `make prune-images-pre`
- **Prune old production API images**: `make prune-images-prod`
- **Update Config**: Update GitHub Secrets/Variables or local environment variables, then re-run the relevant deploy command.
- **Preproduction DB path**: `/etc/readio/pre/data/readio.db`
- **Production DB path**: `/etc/readio/prod/data/readio.db`
- **Cleanup**: Use `make prune-images-pre` and `make prune-images-prod` to remove old Readio API images without touching Docker volumes or unrelated workload images.

## 7. Rollback

Thanks to the tagging mechanism, rollbacks are fast and safe — no rebuild required.

1. **Image Tag Rollback (Recommended)**:
   - Navigate to the **Deploy Cloud (Production)** workflow in the GitHub Actions tab.
   - Click **Run workflow**, select the `release` branch.
   - Enter the previous stable version (e.g. `v1.0.9`) in the **`image_tag`** field.
   - Type `deploy` in the `confirm` field and click **Run**.
   - The build step is skipped entirely; the existing image is pulled and deployed directly.

2. **Code Reversion (Revert PR)**:
   - If a permanent code rollback is required, use the **Revert** button on the GitHub PR page.
   - After merging the revert PR, follow the normal tagging process to deploy the corrected state.
