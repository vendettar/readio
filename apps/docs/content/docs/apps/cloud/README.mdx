The Cloud bootstrap currently splits into `apps/cloud-ui` and `apps/cloud-api`.
`apps/cloud-ui` is the Cloud UI build, and `apps/cloud-api` is the Go service that serves that build, boots SQLite for the current phase, and owns the backend same-origin networking used by migrated discovery, search, and feed flows.

## Runtime Config Ownership

Cloud production runtime config ownership is split into three buckets:

1. **Server-owned runtime config**: Values that stay in backend runtime environment (VPS env/systemd env files). These include service listen address, static asset location, database path, and backend abuse-control settings (`PORT`, `READIO_CLOUD_UI_DIST_DIR`, `READIO_CLOUD_DB_PATH`, `READIO_ASR_ALLOWED_ORIGINS`, `READIO_ASR_RATE_LIMIT_BURST`, `READIO_ASR_RATE_LIMIT_WINDOW_MS`).

2. **Browser-public runtime config**: Values emitted by `apps/cloud-api` via `/env.js` using an explicit allowlist. Cloud does **not** use `apps/cloud-ui/public/env.js` or `dist/env.js` as runtime truth. The backend serves `/env.js` dynamically with only browser-safe fields. Browser-public fields include app name/version, same-origin backend URLs, ASR relay token (abuse-control only, not a secret), ASR provider/model defaults, feature toggles, and UI defaults.

3. **Browser-local/user-local config**: Values that remain browser-owned and are not migrated to backend runtime (`READIO_ASR_API_KEY`, `READIO_OPENAI_API_KEY`, IndexedDB/Dexie persistence, playback/session state).

See `handoff/runtime-config-ownership-audit.md` for the full audit and `handoff/index.mdx` Section 8 for the runtime config ownership summary.

Cloud Settings no longer includes the `CORS Proxy` block for the migrated Cloud contract.
Cloud media stays browser-direct first when available, then falls back through `apps/cloud-api` only for approved failure classes; that fallback is pass-through and non-caching, and browser-local media artifacts stay browser-local.
Cloud ASR provider transcription now goes through a dedicated same-origin relay in `apps/cloud-api`; in this phase provider credentials remain user-owned and browser-supplied per request, and the backend must treat them as transient request data only.
Cloud ASR provider verification/readiness checks remain browser-direct in the current phase.
Cloud currently keeps Workbox/PWA disabled in this phase.

See the Cloud handoff docs under `apps/docs/content/docs/apps/cloud/handoff/` for the current backend contracts and task-level implementation boundaries.
