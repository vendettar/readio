# 架构决策记录

## D001 — 全表使用 UUID 主键
- **决策**: 所有表使用 UUID 作为主键；自然键（如 `feedUrl`, `key`）仅作为唯一索引。
- **理由**: 全局唯一性，同步就绪，稳定的身份标识。
- **影响**: 架构更改需要重置数据库（首发版策略）。

## D002 — 直接使用 I18n
- **决策**: 组件直接使用 `react-i18next` 的 `useTranslation()`；不使用 `useI18n` 适配器。
- **理由**: API 标准化，层级更少，上手更清晰。

## D003 — 离线策略 A
- **决策**: 离线通过时隐藏远程 Discovery 和远程搜索结果；保留本地内容并显示离线徽章。
- **理由**: 用户体验可预测，避免网络调用失败。

## D004 — ObjectURL 所有权
- **决策**: 音频 Object URL 由 Store 拥有；图片 Object URL 由组件拥有并在卸载时清理。
- **理由**: 防止音频内存泄漏，同时保留封面的 UI 灵活性。

## D005 — 部署目标 [已实施]
- **决策**: 主要部署在 Vercel；支持 Docker 自托管。
- **理由**: 快速 Web 部署，具备便携的后备方案。

## D006 — 必需的法律页面 [已实施]
- **决策**: 隐私政策和服务条款是生产环境必需的。
- **理由**: 商店合规性和用户信任。

## D007 — 冒烟端到端测试 [已实施]
- **决策**: 发布前维护针对核心用户旅程的 Playwright 冒烟测试。
- **理由**: 防止单元测试遗漏的端到端回归。

## D008 — 严格 CSP [已实施]
- **决策**: 对于所有 Web 部署，通过 **HTTP 响应头** (server/edge) 强制执行严格的内容安全策略。
- **理由**: 降低 XSS 和不受信任资源加载的风险。

## D009 — 数据保留策略 [已实施]
- **决策**: 将播放历史记录修剪到有限的窗口（时间 + 数量）。
- **理由**: 防止长期 IndexedDB 膨胀和性能下降。

## D010 — PWA 更新生命周期 [已实施]
- **决策**: 当有新 PWA 版本可用时通知用户并允许刷新。
- **理由**: 避免客户端过时，确保用户获得修复。

## D011 — CI 流水线 [已实施]
- **决策**: 在每次 PR 上通过 CI 强制执行 lint/typecheck/tests。
- **理由**: 防止回归绕过本地检查。

## D012 — 导入完整性检查 [已实施]
- **决策**: 在写入 DB 之前验证导入的 ID 冲突和损坏的引用（指令 049）。
- **理由**: 防止 Vault/OPML 导入期间的静默损坏。

## D013 — 包大小预算 [已实施]
- **决策**: 在 CI 中强制执行包大小预算（指令 050）。
- **理由**: 防止依赖膨胀导致性能回归。

## D014 — Iframe 沙箱策略
- **决策**: 需要时，对不受信任的富文本 HTML 使用严格的 iframe 沙箱。
- **理由**: 将不受信任的内容隔离在 DOMPurify 之外。

## D015 — 反馈循环
- **决策**: 提供应用内的错误报告和技术支持联系链接。
- **理由**: 闭合产品反馈循环。

## D016 — 离线内容可见性
- **决策**: 显示离线可用性徽章并提供缓存清理工具。
- **理由**: 使离线行为透明且可管理。

## D017 — 剧集排序与过滤
- **决策**: 为剧集列表提供排序/过滤控制。
- **理由**: 提高大规模数据的可发现性。

## D018 — 内容归属
- **决策**: 显示 RSS 内容的归属和版权投诉入口。
- **理由**: 减少法律风险并明确内容所有权。

## D019 — 多语言排版 [已实施]
- **决策**: 通过 `:root[lang]` 选择器使用特定语言的字体栈。
- **理由**: 确保跨脚本（拉丁语、中日韩）的渲染一致性。
- **实施**: 在 `index.css` 中配置特定语言的字体栈，并开启 optimizeLegibility 渲染。

## D020 — DB 访问防护 [已实施]
- **决策**: 通过 lint 规则防止 UI 组件直接访问 DB。
- **理由**: 强制执行架构边界并维护 Store 优先的持久化。
- **实施**: `pnpm lint:db-guard` 脚本检查组件中是否有 DB 导入。

## D021 — 网络测试使用 MSW [已实施]
- **决策**: 在测试中使用 MSW 模拟外部 API。
- **理由**: 稳定且确定性的测试运行。
- **实施**: `src/__tests__/setup.ts` 和 `handlers.ts` 拦截请求。

## D022 — 贡献与宣言
- **决策**: 记录指令驱动的工作流和架构宣言。
- **理由**: 减少入职摩擦并保持规则一致。

## D023 — PWA 音频范围缓存 [已实施]
- **决策**: 在 SW 媒体缓存中支持范围请求 (206) 以进行离线搜索。
- **理由**: 启用大型音频文件的离线拖拽。

## D024 — 内存压力测试 [已实施]
- **决策**: 维护长会话内存压力测试以检测泄漏。
- **理由**: 防止长时间播放后移动设备崩溃。
- **实施**: `apps/lite/tests/stress/memory.test.ts` (Playwright)。

## D025 — 代理故障转移
- **决策**: 支持代理列表，并在 429/5xx 时自动故障转移。
- **理由**: 保持发现功能在代理失败时的弹性。

## D026 — I18n 覆盖率审核 [已实施]
- **决策**: 在构建时强制执行所有语言的翻译键对齐。
- **理由**: 防止非英语区域设置中出现缺失键的回归。
- **实施**: `apps/lite/scripts/audit-i18n.ts` 将所有语言与 `en` 进行比较，并集成到 CI 中（指令 063）。

## D027 — IndexedDB 迁移
- **决策**: 从仅重置策略迁移到版本化迁移。
- **理由**: 在生产环境中保留用户数据。

## D028 — 浏览器支持矩阵
- **决策**: 定义 browserslist 和遗留 polyfill 策略。
- **理由**: 避免在旧设备/WebView 上出现白屏。

## D029 — Turborepo 缓存加固 [已实施]
- **决策**: 为 Turbo 管道明确输入/输出。
- **理由**: 确保缓存正确性和快速 CI。
- **实施**: `turbo.json` 定义明确的 `inputs`（源文件、配置）和 `outputs`（`dist/**`、`stats.html`）。跟踪全局依赖项（`.npmrc`、`pnpm-workspace.yaml`）。测试文件不包括在构建输入中（指令 066）。

## D030 — 原子写入与自愈 [已实施]
- **决策**: 复杂的写入（导入、删除）必须是事务性的；启动过程检查完整性。
- **理由**: 如果浏览器在文件导入期间崩溃，防止出现“半写入”状态。
- **实施**: 在 `db.transaction` 中包装 `ingestFiles` 和 `deleteFolder`。在 `useAppInitialization` 中添加 `runIntegrityCheck` 以标记孤立的（无 blob）轨道（指令 067）。

## D031 — 异步竞争条件防护 [已实施]
- **决策**: 使用 requestId/AbortController 确保最后动作优先。
- **理由**: 避免快速交互下的播放/UI 不同步。
- **实施**: 向 `playerStore` 和 `exploreStore` 添加 `loadRequestId`；在 `fetchUtils.ts` 中针对同一资源请求实施自动中止。
- **规则文档**: `apps/docs/content/docs/apps/lite/coding-standards/logic-flow.mdx`（部分：异步交互安全）。

## D032 — 健壮的元数据解析器（阶段 1）
- **决策**: 使用 `music-metadata` 进行 ID3/M4A 解析。
- **理由**: 用二进制流解析替换基于文件名的启发式方法，以提取高质量的元数据（封面、标题、专辑、艺术家、时长）。
- **封面策略**: 提取嵌入的封面，作为 Blob 存储在 Dexie 中，并在 UI 中通过临时 `ObjectURL` 提供服务。
- **实施**: 在专用 Web Worker (`src/workers/metadata.worker.ts`) 中运行，以防止在大型文件导入期间阻塞主线程。在 `main.tsx` 中应用了 Buffer polyfill。

## D033 — 骨架屏 UI 系统 [已实施]
- **决策**: 为关键界面（PodcastCard、EpisodeRow、ExploreHero）标准化基于微光的骨架屏。
- **理由**: 减少布局偏移并提高内容加载期间的感知性能。
- **实施**: `ui/skeleton.tsx` 中的基础 `Skeleton` 组件使用 `animate-shimmer`。特定的骨架屏镜像内容组件的布局尺寸。

## D034 — 浮层层级标准化
- **决策**: 标准化 Portal 使用和 z-index token，无需全局模态框 Store。
- **理由**: 防止堆叠错误，同时保持本地状态完整。

## D035 — 媒体缓存驱逐
- **决策**: 强制执行基于 LRU 的音频 blob 用于配置限制的修剪。
- **理由**: 防止不受控制的存储增长。

## D036 — 网络回退与熔断 [已实施]
- **决策**: 添加指数回退和每个代理的熔断器。
- **理由**: 避免重试风暴并提高代理弹性。
- **实施**: 
    - `fetchUtils.ts` 中的 `withRetry` 助手具有指数回退（1s、2s、4s、8s）和抖动。
    - `CircuitBreaker` 状态管理在连续 5 次失败后阻止失败的代理 30 秒。
    - 5xx 或网络级错误自动重试；4xx 跳过。

## D037 — 存储配额监控
- **决策**: 在设置中显示配额使用情况和警告。
- **理由**: 在存储耗尽之前让用户可见。

## D038 — Intl API 升级
- **决策**: 使用 Intl API 处理日期、数字和相对时间。
- **理由**: 提高本地化正确性并减少手动格式错误。

## D039 — 细粒度错误边界 [已实施]
- **决策**: 为关键 UI 节点添加组件级错误边界。
- **理由**: 防止单个组件崩溃导致整个页面瘫痪。
- **实施**:
    - `ComponentErrorBoundary` 类组件处理 `getDerivedStateFromError` 和 `componentDidCatch`。
    - 集成 `logError` 进行诊断。
    - 包装 `TranscriptView`、`EpisodeRow` 和 `Sidebar` 以进行局部故障隔离。
    - 添加 i18n 支持的本地化回退 UI 和重试机制。（指令 076）

## D040 — Lighthouse CI 预算
- **决策**: 在 CI 中强制执行 Lighthouse 分数预算。
- **理由**: 防止性能和可访问性回归。

## D041 — 依赖审计与许可策略
- **决策**: 在 CI 中强制执行漏洞和许可检查。
- **理由**: 降低供应链和合规风险。

## D042 — 浮动 Promise 审计
- **决策**: 要求对异步 Store 操作进行显式错误处理。
- **理由**: 防止静默失败和状态不一致。

## D043 — 事件监听器泄漏审计
- **决策**: 通过共享 Hook 标准化全局监听器管理。
- **理由**: 避免长时间会话下的内存泄漏。

## D044 — Zustand 选择器优化
- **决策**: 强制执行原子选择器和 `useShallow` 进行多选。
- **理由**: 减少不必要的重新渲染。

## D045 — RSS 内容 CSS 隔离
- **决策**: 通过作用域重置（Shadow DOM 回退）隔离节目说明 CSS。
- **理由**: 防止不受信任的 HTML 污染 UI。

## D046 — 文件验证策略
- **决策**: 分层方法以兼顾性能和安全性。
  - **Web (Lite)**: 仅 MIME 类型和扩展名验证（注重用户体验）。
  - **后端**: 深度内容验证（Magic Numbers、完全解析）。
  - **原生**: 基于解码器的验证或轻量级头部嗅探。
- **理由**: 保持客户端逻辑快速轻量，同时将密集检查推迟到安全环境。

## D047 — UI 模式变更门禁
- **决策**: 凡是对 `ui-patterns/*.mdx` 进行修改并引入新的标准化交互（例如标准表单、文件拖放区），必须记录在决策日志中。
- **理由**: 确保架构一致性并在整个项目中维护 UI 标准化的单一来源（指令 012-015）。

## D048 — 标准化文件拖放区
- **决策**: 所有文件上传界面必须使用基于 `react-dropzone` 的 `FileDropZone` 组件，并进行基于 Zod 的验证。
- **理由**: 提供一致的视觉反馈（玻璃拟态）、集中的错误处理和整个应用程序中的类型安全验证（指令 015）。

## D049 — 高级玻璃视觉效果
- **决策**: 核心结构表面（侧边栏、播放器页脚、浮层）必须使用高级玻璃材质（`backdrop-blur-xl`、`backdrop-saturate-150` 和 `bg-background/80` 或 `bg-popover/95`）。
- **理由**: 获得高质量、类似原生的美感，适应内容和主题，同时保持可读性（指令 016）。

## D050 — 列表行交互
- **决策**: 所有列表项和卡片必须使用 `bg-accent/50` 作为悬停状态，并使用 `border-border/50` 进行分隔，采用“智能分隔线”逻辑（指令 016）。
- **理由**: 保持不同列表类型的视觉一致性，并为交互式行创建集中、悬浮的效果。

## D051 — 现代 Toast 通知 (Sonner)
- **决策**: 用 `sonner` 替换 Radix Toast 用于所有通知反馈。
- **理由**: Sonner 提供卓越的开发体验（命令式 API）、更好的视觉堆叠、内置 Promise 支持以及原生的动画手感（指令 017）。

## D052 — 命令面板搜索 (cmdk)
- **决策**: 用基于 `cmdk` 的内联侧边栏 `CommandPalette` 替换自定义浮动搜索浮层。
- **理由**: 提供一致的键盘导航、改进的可访问性以及现代的“内联侧边栏搜索体验”，全局通用（指令 018）。
- **实现细节更新**: 保持侧边栏内联输入体验不变，但将结果面板壳层标准化为 `Popover` Portal（`PopoverAnchor` + `PopoverContent`），并统一通过 store 驱动的 `onOpenChange(false)` / `closeOverlay()` 关闭路径。

## D053 — 标准化徽章系统
- **决策**: 列表和命令面板中的所有状态指示器（订阅、收藏、历史、文件、稍后收听）必须使用统一的视觉语言：右对齐、垂直居中且颜色为 `text-primary`。
- **理由**: 确保视觉一致性，并通过在整个应用程序中为“状态”元数据建立可预测的位置和颜色来减少认知负荷。

## D055 — 标准化 Bits 动画
- **决策**: 在 `src/components/bits/` 中引入一套标准化的、高质量的动画组件 (Bits)，从 `AnimatedList` 和 `CountUp` 开始。
- **理由**: 用流畅、高级的动效模式 (Framer Motion) 提升应用程序的“氛围”和感知质量，同时确保可重用性和架构一致性（指令 021）。
- **实施**:
    - `AnimatedList`: 列表和网格的交错进入动画。
    - `CountUp`: 用于统计数据和用量值的平滑弹簧式数字过渡。

## D056 — 有条件的“最后播放”时间戳更新
- **决策**: `PlaybackSession` 中的 `lastPlayedAt` 时间戳仅在 `isPlaying` 为真时更新。
- **理由**: 防止会话恢复（例如重新打开标签页）或在暂停时手动搜索错误地将项目顶到“最近”列表的顶部。“最后播放”应语义上代表内容的主动消费。
- **实施**:
    - 删除了 `DB.updatePlaybackSession` 中强制的 `Date.now()`。
    - `playerStore` 动作（`updateProgress`, `saveProgressNow`）根据当前 `isPlaying` 状态有条件地包含 `lastPlayedAt`。

## D057 — 共享元素过渡
- **决策**: 使用 `layoutId` (Framer Motion) 在全局 UI 状态（MiniPlayer ↔ FullPlayer）之间同步持久 UI 元素（特别是封面）。
- **理由**: 保持空间连续性，减少视觉突变，并获得高端“原生”手感。
- **约束**: `layoutId` 在同一个可见树中必须唯一。如果不同布局（移动/桌面）的多个节点共享跟踪同一对象的语义 ID，则必须使用条件渲染（例如 `isDesktop`）。

## D058 — 应用启动序列
- **决策**: 显示全屏 `BootLoader`（呼吸 Logo），直到所有核心 IndexedDB 存储水合且播放会话恢复。
- **理由**: 防止“水合闪烁”（空状态短暂闪烁），并在冷启动期间提供高级的原生应用手感。
- **约束**: 
    - Loader 必须使用 `fixed inset-0 z-50 bg-background` 完美覆盖内容。
    - 过渡必须使用 `AnimatePresence` 进行平滑淡出。

## D059 — 共享元素过渡作用域
- **决策**: 当同一逻辑对象在单页中出现多次时，`layoutId` 必须通过唯一前缀（如 `sectionId`）进行作用域隔离，避免冲突。
- **状态更新（传输层已被 D080 覆盖）**: `fromLayoutPrefix` 仍是转场元数据键，但仅允许通过类型化 `location.state` 传递，不再进入 URL `search`。
- **理由**: 既要避免 Framer Motion 在重复内容场景中的冲突，也要保证转场元数据不污染可分享的规范化 URL。

## D060 — 逻辑属性迁移 (RTL 支持)
- **决策**: 将所有物理 CSS 属性 (`left`, `right`, `ml`, `mr`, `pl`, `pr`, `text-left`, `text-right`, `border-l`, `border-r`) 迁移到逻辑等效项 (`start`, `end`, `ms`, `me`, `ps`, `pe`, `text-start`, `text-end`, `border-s`, `border-e`)。
- **理由**: 无需复制 CSS 规则或使用复杂的覆盖即可无缝支持从右到左 (RTL) 语言（例如阿拉伯语、希伯来语）。符合现代 Web 标准并改善未来的可访问性。
- **约束**: 在适当的情况下，必须使用 `className="rtl:rotate-180"` 显式翻转方向图标。

## D061 — Store 持久化 Repository 边界（Phase 1）[已实施]
- **决策**: 为 `exploreStore`、`historyStore`、`playerStore`、`filesStore` 引入 `LibraryRepository`、`PlaybackRepository`、`FilesRepository` 作为持久层边界。
- **理由**: 在不改变 store API/行为的前提下，降低 store 对 Dexie 直连耦合。
- **实施**:
  - 目标 store 已迁移为 repository 调用。
  - 新增 repository 契约测试与 store repository-boundary 测试。
  - Dexie schema/version 与 `DB` 导出保持不变。

## D062 — Fetch/Runtime Config 内部模块拆分（Phase 1）[已实施]
- **决策**: 保持 `fetchUtils.ts`、`runtimeConfig.ts` 对外入口不变，将内部策略逻辑拆分到专用模块。
- **理由**: 降低关键文件复杂度，在不破坏外部契约时提升可维护性与可测试性。
- **实施**:
  - 新增 `lib/networking/proxyUrl.ts`、`lib/networking/circuitBreaker.ts`、`lib/networking/timeouts.ts`。

## D070 — CORS Proxy 架构与故障排除 [2.0 重构]
- **决策**: 
    1. **接口标准**: 强制要求所有 CORS 代理使用 Raw/Passthrough 模式（返回原始 Content-Type 和 Body），废弃特定于 AllOrigins 的 JSON 包裹模式。
    2. **配置**: 此标准允许通过简单的 URL 模版（`{url}`）无缝支持 Cloudflare Worker、Nginx 反代等任何标准代理。
    3. **可靠性**: 在客户端 Fetch 逻辑中**禁用**熔断器（Circuit Breaker）和全局请求去重（`abortPrevious`）。
- **理由**: 
    - **通用性**: 消除对特定公共代理（`api.allorigins.win`）的硬编码依赖。
    - **客户端弹性**: 在单用户浏览器环境中，网络状态瞬息万变。熔断器的“预判失败”逻辑和全局去重的“后浪杀前浪”逻辑会导致部分请求（如 Top Episodes 在 Top Shows 失败后）被错误地跳过或取消。必须强制对每个配置的代理进行**穷尽式重试**，以确保数据加载的绝对可靠性。

## D073 — 播客 Transcript 自动导入缓存

- **理由**: 在不影响音频控制响应的前提下保持 read-along 连续性；当网络失败时可继续使用 stale 缓存，避免重验证失败导致可用数据丢失。
- **守卫**: 本地手动字幕优先；远程 transcript 应用必须通过播放上下文校验（`loadRequestId` + `audioUrl`），防止旧异步响应覆盖当前曲目。

## D116 — Cloud 脚手架使用 Go `net/http`、SQLite 启动，并复用 Lite Dist [已实施]
- **决策**：
  - 当前 Cloud 脚手架阶段在 `apps/cloud` 中使用纯 Go 的 `net/http` 后端。
  - 在这一阶段，`apps/cloud` 直接提供构建后的 `apps/lite/dist` 前端产物。
  - SQLite 是 Cloud 当前的数据库启动目标，用于自托管和本地可移植性。
  - 早先的 Spring Boot / Hono 方向不属于当前脚手架契约，不能再作为本阶段已实现的 Cloud 后端来描述。
- **理由**：
  - 这个脚手架故意保持最小化，必须继续保持可移植、可审查、易于自托管。
  - 复用 `apps/lite/dist` 可以避免重复前端工作，同时让 Cloud 后端只提供当前契约。
  - 纯 Go 服务加 SQLite 可以保持启动路径无 CGO，并降低部署摩擦。
- **风险说明**：
  - 文档不能暗示当前已经存在单独的 Cloud Web 前端。
  - 文档不能声称脚手架阶段尚未具备的同步、认证或 schema 层。
  - Cloud 的长期演进可以改变技术栈，但当前阶段契约必须保持明确且收敛。

## D117 — Cloud ASR Provider 转写使用专用同源 Relay [已实施]
- **决策**：
  - Cloud ASR provider 转写必须使用专用同源后端 relay（`POST /api/v1/asr/transcriptions`），而不是浏览器直连 provider 调用。
  - `/api/proxy` 保持为有限的 media/RSS 代理入口，不能变成通用的 provider POST 隧道。
  - 在这一阶段，provider credential 仍由用户拥有并由浏览器按请求提供；后端 relay 只能把它们当作瞬时请求数据，不能持久化、缓存或记录。
- **理由**：
  - 浏览器直连 provider 提交在媒体 fallback 工作完成后仍会因为 provider CORS 而失败。
  - 专用 relay 在不提前改变产品用户提供的密钥模型的情况下修复了传输所有权。
  - 保持 `/api/proxy` 收敛可避免在便利抽象下扩大 SSRF/body-tunneling 风险。
- **风险说明**：
  - Provider 验证 / readiness 保持 browser-direct，在本阶段有意不在范围内。
  - 浏览器本地 blob/已下载 blob 保持为 browser-local 输入；relay 不会把音频存储移到服务端。
  - 请求体必须保持有限，错误映射必须保留现有 ASR 重试/冷却语义。

## D118 — Cloud 生产环境运行时配置所有权迁移至后端生成的 /env.js [已实施]
- **决策**：
  - Cloud 生产环境运行时配置所有权迁移至由 `apps/cloud-api` 动态生成的 `/env.js`。
  - `apps/cloud-ui/public/env.js` 面向开发/默认模板，不是生产环境 Cloud 基准。
  - 服务端所有的运行时配置（监听地址、路径、滥用控制设置）放在 VPS/systemd 环境文件中。
  - 浏览器公开运行时配置由后端通过显式 allowlist 发出，不是从进程环境变量转储。
- **备选方案**：
  - 保持静态前端文件（`dist/env.js`）作为生产基准：被拒绝，因为 Cloud 是后端所有的产品表面，生产配置不应要求手动编辑静态前端资产。
  - 共享配置包：被拒绝，因为过于提前泛化；Cloud 的所有权模型不同于 Lite 的浏览器直连模型。
- **理由**：
  - Cloud 现在在后端拥有网络，必须拥有运行时配置契约。
  - 生产配置不应要求在 VPS 上手动编辑静态前端资产。
  - 浏览器公开字段不是密钥；通过后端 allowlist 发出使契约显式且可审计。
- **风险说明**：
  - 浏览器公开字段（例如 `READIO_ASR_RELAY_TOKEN`）是轻量级滥用控制 token，不是强密钥边界。
  - 如果字段被发出到 `/env.js`，假设每个浏览器用户都能读取它；设计时需考虑这一点。
  - allowlist 在 Go handler（`apps/cloud-api/main.go`）的 map 字面量构造中是隐式的，只发出浏览器安全的字段。
