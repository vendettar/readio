# RSS Case Samples

This page stores representative RSS samples captured on 2026-04-13 (Asia/Shanghai) as reference cases for parser and data-model analysis.

Each case keeps the `channel` header and exactly one representative `item`. Full feeds are intentionally omitted because item volume does not add structural value for this document.

## Type Overview

This case library currently covers 4 feed types:

1. Traditional RSS 2.0 podcast feeds
2. RSS 2.0 feeds with richer Podcasting 2.0 / vendor extensions
3. RSS 2.0 feeds with stronger internationalization / multi-namespace variation
4. True Atom feeds

## Type 1: Traditional RSS 2.0 Podcast Feeds

These are the most common podcast feeds in the current Readio pipeline.

### What to watch

- Root structure is standard RSS: `<rss><channel><item>`
- Audio usually comes from `<enclosure>`
- Common fields may still vary between `description`, `itunes:summary`, `content:encoded`, `itunes:image`, and `<image><url>`
- This is the primary compatibility target for the current parser

## Type 2: RSS 2.0 With Rich Podcasting 2.0 / Vendor Extensions

These feeds are still RSS, but they carry much richer extension data than traditional podcast feeds.

### What to watch

- Still RSS structurally; do not confuse them with Atom
- May include multiple `podcast:transcript` nodes, `podcast:chapters`, `podcast:value`, `alternateEnclosure`, vendor-specific tags, or extra media nodes
- Current parser only maps a subset of these extensions, so these cases are useful both for supported behavior and explicit non-support boundaries
- These feeds are often where "it is valid XML, but our DTO intentionally ignores most of it" becomes important

## Type 3: RSS 2.0 With Internationalization / Multi-Namespace Variation

These feeds are still RSS, but the main compatibility pressure comes from language, encoding, CDATA, and namespace diversity rather than from richer podcast feature semantics.

### What to watch

- Non-English text and localized punctuation
- Heavy CDATA usage
- More exotic or less product-relevant namespaces
- Useful for checking that sanitization, text extraction, and HTML handling do not assume English-only content

## Type 4: True Atom Feed Boundary References

These are not RSS feeds. They are structurally different enough that they should be treated as a separate compatibility boundary.

### What to watch

- Root is `<feed>`, not `<rss>`
- Entries are `<entry>`, not `<item>`
- Time fields are typically `published` / `updated`, not `pubDate`
- Current Readio parser is RSS-oriented, so this type is mainly useful for making the unsupported boundary explicit
- This section is a format-boundary reference, not a confirmed Readio-relevant podcast input category
