# Apple Discovery API (iTunes Implementation)

This document provides a reference for the Apple/iTunes APIs used in the Readio project as the implementation for the Discovery Provider. Each section includes a description, the API URL, and a `curl` command for testing.

## PWA & iOS Integration (App Metadata + Sharing)

- **Manifest**: Generated by `vite-plugin-pwa` as `manifest.json`; source of truth is `apps/lite/vite.config.ts` (`VitePWA.manifest`). Do not maintain `apps/lite/public/manifest.json`.
- **iOS Meta Tags**: `apps/lite/index.html` includes:
  - `apple-mobile-web-app-capable`
  - `apple-mobile-web-app-status-bar-style`
  - `apple-touch-icon`
- **Native Share**: `apps/lite/src/components/Player/ShareButton.tsx` uses `navigator.share` with title + URL, with clipboard fallback.

## 1. Top Charts (Apple Podcasts RSS API)

These APIs are used to fetch the most popular content across different regions.

- **URL**: `https://rss.marketingtools.apple.com/api/v2/{country}/podcasts/top/{limit}/podcasts.json`

> **Client-Side Filtering**: 
> Apple's Search API sometimes returns irrelevant "popular" results when it cannot find a good match (especially for queries with typos like `the kevin spacy trail`).
> 
> To improve user experience, Readio implements a **strict client-side filter**:
> - We tokenize the user's query and remove stop words (`the`, `podcast`, `it`, etc.).
> - For episode search, we currently check if the result's `collectionName` or `trackName` contains **at least one** matching token.
> - If no tokens match, the result is discarded. This prevents showing completely unrelated podcasts.

## 2.5 Transcript Metadata Gap

`transcriptUrl` is not currently symmetric across the Apple discovery surfaces used by Readio.

- RSS/feed parsing can expose transcript metadata through the Podcasting 2.0 `podcast:transcript` tag.
- Apple Search episode results do **not** currently expose an equivalent transcript field in the payload we receive and map.

This matters because:

- feed/detail/favorite/history playback paths can become transcript-bearing
- search-direct playback cannot become transcript-bearing unless transcript metadata is recovered through a later lookup/feed step

### Feed/RSS Example With Transcript Metadata

The feed parser can extract `transcriptUrl` from a real feed response like this:

```json
{
  "title": "Stuff You Should Know",
  "episodes": [
    {
      "id": "62e06b83-685e-47fb-ab42-b42001040745",
      "title": "Selects: The Skinny on Lyme Disease",
      "audioUrl": "https://podtrac.com/pts/redirect.mp3/pscrb.fm/rss/p/traffic.omny.fm/d/clips/...",
      "transcriptUrl": "https://api.omny.fm/orgs/e73c998e-6e60-432f-8610-ae210140c5b1/clips/62e06b83-685e-47fb-ab42-b42001040745/transcript?format=TextWithTimestamps&t=1775145077"
    },
    {
      "id": "4925df4b-12a7-4e30-972d-b41d0115cbad",
      "title": "The Colorado River Compact",
      "audioUrl": "https://podtrac.com/pts/redirect.mp3/pscrb.fm/rss/p/traffic.omny.fm/d/clips/...",
      "transcriptUrl": "https://api.omny.fm/orgs/e73c998e-6e60-432f-8610-ae210140c5b1/clips/4925df4b-12a7-4e30-972d-b41d0115cbad/transcript?format=TextWithTimestamps&t=1774889704"
    }
  ]
}
```

In Readio's feed parser, this comes from the RSS item-level `podcast:transcript` tag.

### Search Result Limitation

By contrast, the current Apple Search episode result payload includes fields such as:

- `trackName`
- `episodeUrl`
- `feedUrl`
- `description`
- `releaseDate`

But it does **not** include transcript metadata in the mapped `SearchEpisode` payload.

As a result:

- playing an episode from feed/detail data may have `transcriptUrl`
- playing the same episode directly from Search may not

If Readio wants transcript-first playback parity for search-direct playback, it must either:

1. receive transcript metadata directly from the search provider, or
2. enrich the search result through a follow-up feed/lookup step before playback

### 🎯 Lookup Episode (The "Podcast + Episode" Approach)
**Note**: The iTunes Lookup API often fails to retrieve a specific episode by its single `trackId` directly. The most reliable method is to lookup the **Podcast ID** and include episodes.

> ⚠️ **Critical Limitation (Tested 2026-01-09)**: The `limit` parameter is **ineffective** for most podcasts. Even with `limit=300`, the Provider API only returns ~25 episodes (matching what the podcast provider puts in their RSS feed). For example, "The Daily" has `trackCount: 2475` but the API only returns 23 episodes. **You cannot retrieve a podcast's full episode archive via this API.**

- **URL Prefix**: `https://itunes.apple.com/lookup`
- **Params**: `id={PODCAST_ID}`, `entity=podcastEpisode`, `limit=1`

# Example: Lookup 'The Daily' (1200361736) and its most recent episode

## Parameters Summary

| Parameter | Description | Common Values |
| :--- | :--- | :--- |
| `country` | Two-letter ISO country code | `us`, `cn`, `jp`, `kr`, `gb` |
| `limit` | Number of results to return | `10`, `30`, `50` |
| `term` | URL-encoded search string | `search+query` |
| `media` | Type of media to search | `podcast` |
| `entity` | Specific object type | `podcast`, `podcastEpisode` |
| `id` | iTunes ID (Numeric) | e.g. `1200361736` |

## 4. Additional APIs (Not Currently Used)

### 🎤 Artist/Publisher Lookup
Retrieve all podcasts from a specific publisher/artist.
- **URL Prefix**: `https://itunes.apple.com/lookup`
- **Params**: `id={ARTIST_ID}`, `entity=podcast`
- **Use Case**: "More from The New York Times" - show all podcasts from the same publisher
```bash
# Example: All podcasts from The New York Times (artistId: 121664449)
curl "https://itunes.apple.com/lookup?id=121664449&country=us&entity=podcast"
```
**Notes**:
- `artistId` can be found in podcast metadata response (e.g., `artistId: 121664449`)
- Returns all podcasts published by that artist/company
- Useful for "Related Shows" or "From the Same Publisher" sections

## 5. Known Issues & Gotchas
 
### ⚠️ Inconsistent Episode Sorting & Limit
**Issue**: For frequently updated podcasts (e.g., *NPR Up First*), the iTunes `lookup` API may sort episodes inconsistently. The "latest" episode might NOT appear in the first 50 results, even if it was released today.
**Impact**: If you query with `limit=50`, you might miss the newest episode, causing "Episode Not Found" errors in your app when users try to access it via deep links or top charts.
**Solution**: Always use a higher limit (e.g., `limit=300`) when using the `lookup` API for critical episode resolution, especially for fallback logic.

## 6. Unavailable / Internal Apple APIs

The following capabilities are **NOT available** through public APIs:

| Feature | Status | Notes |
|---------|--------|-------|
| **Full Episode Archive** | ❌ Unavailable | Even with `limit=300`, API returns ~25 episodes. Cannot access complete podcast history. |
| **Play Count / Statistics** | ❌ Unavailable | Apple does not expose listening statistics |
| **User Reviews** | ❌ Unavailable | Podcast reviews are not accessible via API |
| **Subscriber Count** | ❌ Unavailable | Only "Top Subscriber" ranking is public, not actual numbers |
| **Apple Podcasts Connect** | 🔐 Requires Auth | For podcast publishers only, requires Apple Developer account |
| **Private RSS Feeds** | 🔐 Requires Auth | Paid/private podcasts require authentication |

**Workarounds**:
- For full episode history: Some podcasts provide complete archives in their RSS feed, or via their own website
- For reviews: Third-party services like Chartable or Podtrac may provide some analytics
