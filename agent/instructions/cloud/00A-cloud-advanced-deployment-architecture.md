# Instruction 00A — Lean Bare Metal Deployment Architecture [COMPLETED]

## Objective

Move Readio Cloud deployment from YAML-heavy SSH scripts and server-side plaintext env file mutation toward a reproducible, low-memory bare-metal deployment model.

This is an architecture master instruction. It must be split before implementation. Do not implement all phases in one Worker pass.

## Decision

Adopt **Lean Bare Metal + App-Owned Observability**:

- **OS bootstrap**: Ansible owns repeatable VPS provisioning.
- **Runtime**: Docker Engine + Docker Compose remain the container runtime.
- **Reverse proxy**: Nginx remains the only reverse proxy.
- **Workflow**: Makefile owns local and CI deployment commands.
- **Remote execution**: Docker Context over SSH replaces ad hoc remote shell deployment blocks.
- **Observability**: `apps/cloud-api` owns application metrics and pushes them to Grafana Cloud through OTLP when configured.

## Non-Goals

Do not add or rely on:

- Grafana Alloy
- Prometheus/node_exporter on the VPS
- Loki/systemd journal shipping agent
- Traefik
- Coolify
- Docker Swarm
- Kubernetes
- any persistent deployment platform agent

Allowed persistent services:

- Docker Engine
- Nginx
- systemd/OS services required for the VPS

## Current Observability Boundary

The Go backend already owns application-level observability:

- OTLP metric push is controlled by `READIO_GRAFANA_OTLP_ENDPOINT`, `READIO_GRAFANA_OTLP_INSTANCE_ID`, and `READIO_GRAFANA_OTLP_TOKEN`.
- Admin logs are available through the in-memory `/admin/logs` ring buffer.
- Basic process health is available through `/admin/health`.

Current code does **not** implement:

- remote log shipping to Grafana Loki
- systemd journal shipping
- host CPU/load metrics
- host memory metrics
- filesystem free/used metrics
- SQLite WAL growth metrics
- transcript storage pressure metrics

Do not document or imply that these host-level signals exist until a separate Go implementation instruction adds them.

## Secret Model

Do not claim "fileless secrets" or "secrets only exist in container memory."

Docker Compose `environment:` interpolation removes repo-managed `.env` file mutation, but the resulting container environment is still visible to Docker administrators and may be persisted in Docker daemon metadata. The accurate target is:

- no plaintext env artifact generated by CI and copied to the VPS
- no secrets committed to the repo
- no secrets printed in logs
- no browser exposure for server-owned secrets
- secrets stay within the GitHub Actions secret store, SSH transport, Docker daemon admin boundary, and container environment

If a stricter "no Docker metadata secret visibility" model is required later, create a separate architecture decision for Docker secrets, tmpfs secret files, or an external secret manager. That is out of scope for 00A.

## 00A1 — VPS Ansible Bootstrap

### Goal

Create idempotent Ansible provisioning for a raw Ubuntu Noble VPS.

### Scope

- `infra/ansible/`
- Ansible inventory/template files if needed
- docs updates only if needed to explain provisioning

### Required Content

- install Docker Engine and Docker Compose plugin
- ensure deployment user exists
- configure authorized SSH keys through variables/templates, not hard-coded keys
- configure passwordless sudo only for the deployment operations that are required
- create preproduction and production directories:
  - `/etc/readio/pre/data`
  - `/etc/readio/prod/data`
- preserve ownership compatible with the deploy user and Docker runtime
- install or validate Nginx
- link or render `scripts/nginx-readio-cloud.conf` into the expected Nginx site path
- configure journald size limits
- apply conservative sysctl tuning only when documented

### Guardrails

- Do not disable `unattended-upgrades` by default. If RAM pressure requires changing it, document the security tradeoff and require an explicit variable.
- Do not install Grafana Alloy or any host observability agent.
- Do not write production secrets to files.
- Do not hard-code hostnames, SSH keys, usernames, or domains in the playbook.

### Verification

- `ansible-playbook --syntax-check infra/ansible/vps_optimize.yml`
- idempotency notes for running the playbook twice
- docs list required variables

## 00A2 — Makefile Docker Context Workflow

### Goal

Move deployment workflow logic into repo-owned Makefile targets that work locally and in GitHub Actions.

### Scope

- root `Makefile`
- optional helper scripts under `scripts/`
- docs updates for command usage

### Required Targets

- `check-env-pre`
- `check-env-prod`
- `context-setup`
- `deploy-pre`
- `deploy-prod`
- `logs-pre`
- `logs-prod`
- `prune-images-pre`
- `prune-images-prod`

### Safety Requirements

- `deploy-prod` must require an explicit confirmation, for example `CONFIRM=deploy`.
- `check-env-*` must fail before Docker context access when required variables are missing.
- `check-env-*` must print missing variable names only, never values.
- production and preproduction must use distinct:
  - Compose project names
  - container names
  - host ports
  - data directories
  - image package names
- commands must be idempotent.

### Verification

- `make check-env-pre` fails cleanly when required variables are missing
- `make check-env-prod` fails cleanly when required variables are missing
- `make context-setup DRY_RUN=1` or documented equivalent
- shellcheck for helper scripts if scripts are added

## 00A3 — Compose Environment Model

### Goal

Remove CI-generated plaintext env files from deployment flow and make Compose runtime inputs explicit.

### Scope

- `docker-compose.readio.yml`
- Makefile targets from 00A2 if needed
- docs updates

### Required Changes

- remove `env_file:` from `docker-compose.readio.yml`
- explicitly list required environment variables under `environment:`
- keep browser-public config separate from server-owned secrets
- preserve SQLite data volume mapping
- preserve healthcheck behavior
- preserve production/preproduction isolation

### Required Environment Classification

Classify each variable as one of:

- server secret
- server non-secret runtime variable
- browser-public runtime variable
- deployment-only variable

Server secrets must not appear in `/env.js`, browser allowlists, frontend `.env`, or static assets.

### Verification

- `docker compose -f docker-compose.readio.yml config` succeeds with a complete test environment
- config validation fails before deployment when required variables are missing
- generated Compose config must not contain empty required secrets

## 00A4 — GitHub Actions Cutover and Docs Sync

### Goal

Reduce deployment workflows to build/push plus Makefile deployment calls.

### Scope

- `.github/workflows/deploy-cloud-prod.yml`
- `.github/workflows/deploy-cloud-preprod.yml`
- deployment docs in English and Chinese
- Cloud handoff docs if runtime ownership changes

### Required Changes

- remove `appleboy/ssh-action` and `appleboy/scp-action` deployment blocks
- keep Docker build/push steps
- configure SSH for Docker Context without interactive prompts
- pass GitHub secrets and variables through the step `env:` block
- call:
  - `make context-setup`
  - `make deploy-pre` or `make deploy-prod CONFIRM=deploy`
- keep GHCR cleanup behavior, either through existing action or a Makefile/script target

### Documentation Sync

Update docs to state:

- deployment uses Docker Context over SSH
- CI no longer writes a plaintext env file on the VPS
- Docker container environment remains inside the Docker admin boundary, not "memory only"
- Grafana Alloy is not used
- application metrics are pushed by `apps/cloud-api` through OTLP when configured
- host logs and host metrics are not available unless separately implemented

Update both English and Chinese docs when a `.zh.mdx` counterpart exists.

### Verification

- workflow syntax inspection
- `actionlint` if available
- docs mention no Grafana Alloy requirement
- docs do not claim host logs or host metrics are shipped to Grafana
