## Objective

Implement the code and test portion of the Cloud runtime config ownership cutover:

- classify the active Cloud runtime config surface
- move production Cloud runtime ownership to `apps/cloud-api`
- serve browser-public runtime config from backend-generated `/env.js`
- keep `apps/cloud-ui` consuming `window.__READIO_ENV__`
- preserve front-end-only dev behavior with static `public/env.js`

## Required 8-Scope Scan

Before editing code, report focused findings across:

1. config and env parsing
2. persistence and data integrity
3. routing and param validation
4. logging and error handling
5. network and caching
6. storage and serialization
7. UI state and hooks
8. tests and mocks

Also explicitly scan:

- Cloud runtime config keys currently read by `apps/cloud-ui`
- any production path that still depends on `dist/env.js`
- any browser-visible field that is actually backend-only
- any tests that assume static `public/env.js` is the production source of truth

## Required Work

### 1. Produce a real runtime-config ownership audit artifact

Create a written audit artifact under the Cloud handoff suite that classifies the current Cloud runtime config surface into:

- server-owned
- browser-public runtime
- browser-local only

The artifact must reference the actual current keys and runtime readers, not a generic list.

### 2. Add backend-generated `/env.js`

Implement a dedicated `apps/cloud-api` handler for `/env.js` that:

- responds with JavaScript before SPA/static fallback
- emits `window.__READIO_ENV__ = { ... }`
- uses an explicit browser-safe allowlist
- reads backend env directly
- uses explicit defaults
- does not parse `apps/cloud-ui/public/env.js`

### 3. Keep Cloud front-end runtime contract stable

Keep `apps/cloud-ui/src/lib/runtimeConfig.ts` reading `window.__READIO_ENV__`.

Do not force broad front-end rewrites if the existing runtime consumption contract can remain stable.

### 4. Preserve dev/static behavior

Do not break:

- `apps/cloud-ui/public/env.js`
- `apps/cloud-ui/public/env.local.js`
- front-end-only dev server usage

Make the code/tests clear that these files are dev/default-template inputs, not the Cloud production source of truth.

### 5. Move production-owned fields off static front-end truth

At minimum, production Cloud must no longer treat these as front-end-authored runtime truth:

- `READIO_ASR_RELAY_PUBLIC_TOKEN`
- same-origin backend-owned URL roots
- backend abuse-control and transport-boundary knobs

### 6. Add focused verification coverage

Add or update tests proving at minimum:

1. `apps/cloud-api` serves `/env.js` before SPA fallback
2. production `/env.js` contains expected browser-safe fields
3. production `/env.js` does not expose backend-only fields
4. Cloud front-end still reads `window.__READIO_ENV__` correctly
5. Cloud local dev still works with static `public/env.js`
6. the ASR relay token path still works under backend-generated runtime env
